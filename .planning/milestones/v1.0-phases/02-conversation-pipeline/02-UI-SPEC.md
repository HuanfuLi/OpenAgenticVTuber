# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2 (Conversation Pipeline). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope of this contract:** the **behavioral additions** the existing Phase 1 `ChatView` (root `src/shell.jsx:245-359`) needs to host real LLM streaming. Phase 2 ships **zero new components and zero new tokens** — every spacing, color, typography, and copy decision is inherited verbatim from `01-UI-SPEC.md`. The phase is a behavior change, not a chrome change.
>
> **What changes from Phase 1:** the assistant-bubble lifecycle (one growing bubble per turn instead of one bubble per send), the turn-start affordance ("Thinking…" inside the bubble instead of `…`), one new log-line shape (`[INTENT] kind=… name=…`) flowing through the existing Logs drawer, and a streaming sticky-scroll rule that did not need to exist when Phase 1 echo was synchronous.
>
> **What does NOT change from Phase 1:** every visual token. The 7 named theme classes, the spacing scale, the typography sizes/weights, the bubble radius (`rounded-2xl`), the `--card` / `--muted` bubble surface assignments, all copywriting verbatim except for the four new strings declared in this contract, the Logs drawer chrome, the empty-state copy, the error-banner shapes. Phase 2 EXTENDS, NEVER REPLACES.

## Spacing Scale

**Inherited verbatim from `01-UI-SPEC.md`.** Phase 2 introduces no new spacing tokens. The growing-bubble within-turn append uses zero vertical gap (sentences merge into the *same* bubble's body, no inter-sentence padding); between turns uses the existing `md` (16px) `gap-4` between bubble blocks declared in Phase 1.

| Token | Value | Tailwind | Phase 2 usage |
|-------|-------|----------|---------------|
| xs | 4px | `gap-1` | Inline tag-stripped-text / log-line `kind=`/`name=` separator gap |
| sm | 8px | `gap-2`, `p-2` | Bubble inner padding (unchanged), Logs drawer line padding (unchanged) |
| md | 16px | `gap-4` | Vertical rhythm between turn-pairs (unchanged) |
| lg | 24px | `gap-6` | (unused by Phase 2 surfaces) |
| xl | 32px | `gap-8` | (unused by Phase 2 surfaces) |
| 2xl | 48px | `gap-12` | (unused by Phase 2 surfaces) |

**Critical layout rule (new for Phase 2):** within a single growing assistant bubble, sentences append into the same `<div class="body">` as additional text nodes — **no `<br>`, no `<p>` boundary, no inter-sentence margin**. The bubble body is a single text block whose content grows. This matches OLVT's `appendAIMessage` (`chat-history-context.tsx:101-109`) which concatenates `content` into the last AI message rather than appending a new line element. Justification: the user perceives one continuous "stream of thought" within a turn; pysbd-segmented sentence boundaries are an implementation detail, not a UX feature.

**Sentence-arrival visible separation** is achieved by the natural sentence-end punctuation (`.`, `!`, `?`, JP `。` / `？` / `！`) that pysbd preserves — no extra margin needed.

## Color

**Inherited verbatim from `01-UI-SPEC.md`** — all 7 theme classes, all 60/30/10 assignments, all status-color tokens, the saturation cap (chroma ≤ 0.13 for accents, ≤ 0.04 for surfaces). Phase 2 adds **zero new color tokens**.

### Bubble surface assignments (Phase 1 inheritance, unchanged)

| Surface | Token | Phase 2 behavior |
|---------|-------|------------------|
| User bubble background | `--muted` (1 step elevated from `--background`) | Renders user-typed message immediately on Enter (unchanged) |
| Assistant bubble background | `--card` | Renders "Thinking…" then accumulates streamed sentences (D-01, D-03) |
| Bubble foreground (text) | `--foreground` | Inherits theme — confirmed AA 4.5:1 in all 7 variants per Phase 1 |
| Bubble timestamp / role label | `--muted-foreground` | (unchanged) |
| "Thinking…" placeholder text | `--muted-foreground` italic | New rule for Phase 2 — see Typography above |

### ActionIntent log-line color (locked decision — only net new color rule)

The Logs drawer in Phase 1 renders all log lines in `--foreground` (mono 12px). Phase 2 surfaces a new structured event (`[INTENT] kind=… name=… …`) that the user must be able to distinguish from plain `[INFO]` / `[READY]` / `[VTS]` / `[LLM]` lines at a glance during SC #2 verification.

**Decision: prefix-token `[INTENT]` renders in `--success` resolved color; the rest of the line renders in `--foreground`.**

| Light theme | `oklch(0.55 0.13 150)` ≈ `#3F8C58` (deep green) |
| Dark themes | `oklch(0.66 0.13 150)` ≈ `#5CA876` (mid green) |

Why `--success` and not a fresh hue:
- Reuses an already-declared status token (no token-set growth)
- Visually distinct from `[ERROR]` (`--destructive` red) and plain log lines (`--foreground`)
- Status-green chroma (0.13) is at the saturation cap — distinct enough from any user-pickable accent (Blush 0.10 / Sunrise 0.12 / Ember 0.13 / Sky 0.08 / Pewter 0.005) without breaking the "single accent at a time" rule (status colors live in their own slot — `01-UI-SPEC.md` Status Colors)
- Semantic fit: an extracted ActionIntent IS a successful pipeline event (the orchestrator parsed an intent the avatar can act on)

The remainder of the line renders in `--foreground` in mono 12px. The `kind=`, `name=`, `strength=`, `avatar=` key-value pairs are space-separated, no syntax highlighting — same flat aesthetic as every other Logs drawer line.

**Forbidden alternatives:**
- Inline chip / badge with rounded background (would break the mono Logs drawer aesthetic and add a new component to the inventory)
- Color the entire line (loses the "scan for `[INTENT]`" affordance — only the prefix carries the signal)
- A fresh non-status color (would grow the token set without need)
- Icon prefix (lucide icons inside mono text would misalign vertically)

### Color usage for sentence-arrival animation

**No color change during streaming.** Sentences appear in `--foreground` from the first character forward. There is **no fade-in**, **no opacity ramp**, **no color shimmer**. The sentence simply *exists* once it arrives. This honors the Aesthetic Guardrails ban on shadow-glow and the implicit ban on attention-pulling motion in chrome-secondary surfaces.

The only motion permitted during streaming: the chat-scroll auto-pin (described in Interaction Patterns below).

- `Loading…` (generic; doesn't match the avatar-companion voice — "Thinking…" is warmer)

## Interaction Patterns

### IP-1: Growing assistant bubble (D-01, locked)

**State machine for the assistant bubble within a single turn:**

```
[turn-start envelope received from sidecar]
  └─→ create new assistant bubble
      └─→ render CHAT.THINKING ("Thinking…") in italic --muted-foreground

[first audio-payload.display_text arrives]
  └─→ replace CHAT.THINKING with display_text content
      └─→ subsequent display_text arrivals: append to SAME bubble's body string
          (NOT new bubbles, NOT new <p> elements — pure string concat)

[force-new-message envelope received]
  └─→ set internal flag forceNewMessage = true
      (no visual change yet — bubble stays as-is)

[next assistant turn's first display_text arrives]
  └─→ flag === true → create NEW assistant bubble
      └─→ flag is reset to false

[chain-end envelope received]
  └─→ input field re-enabled (it was disabled during the turn)
      └─→ Send button cursor returns to enabled state
      (no visual change to the bubble itself)
```

**Why no visual seal:** the OLVT pattern uses the *next* AI sentence's bubble-creation as the seal, not a divider. A user typing a new message implicitly seals the prior turn (the new user bubble appears immediately after the sealed assistant bubble with the existing 16px `md` gap). Adding a separator line would break the existing chat density and contradict the Aesthetic Guardrails ban on `border-border` (full-opacity borders) outside input rest states.

**Sentence-arrival cadence (locked decision):** sentences render **immediately on receive — no artificial stagger.** The `audio-payload` envelopes arrive as fast as pysbd can segment the LLM's token stream. On a typical ~30 tok/s LM Studio rig, the natural cadence is ~200-800ms between sentences, which is well above the visual jitter threshold. Adding a `setTimeout(30)` stagger would introduce unnecessary IPC latency for zero perceptible benefit. **If profiling reveals visual jitter under exotic conditions** (e.g., a 200 tok/s GPU rig finishing all three sentences in <100ms), the planner may add a CSS-only `transition: opacity 80ms ease` on the bubble's text container as a remediation — but the default ships without any animation. Implementation: just call `setChatMessages` synchronously in the WS dispatcher.

**Forbidden:**
- Per-character typewriter animation (this is a sentence-streaming UX, not a token-streaming UX — the OLVT pattern explicitly buffers tokens until a sentence completes; per-token rendering would defeat the buffer-then-extract that protects against `[joy]` BPE leaks per SC #3)
- Cursor / caret blink at the end of the growing text
- "more coming…" indicator after the latest arrived sentence (the absence of a chain-end envelope IS the indicator; chrome should not duplicate it)

### IP-2: Sticky-scroll during streaming (locked decision)

**Rule:** while a turn is in progress (between turn-start and chain-end envelopes), the chat scroll container auto-scrolls to bottom on every sentence arrival **UNLESS the user has scrolled up by more than 40px from the bottom**.

```
Phase 2 scroll algorithm (per sentence arrival):
  let isAtBottom = (scrollTop + clientHeight) >= (scrollHeight - 40);
  // append the sentence to the bubble
  if (isAtBottom) {
    scrollContainer.scrollTop = scrollContainer.scrollHeight;
  }
  // else: user has scrolled up to read history; respect that — don't yank them down
```

When a turn starts (turn-start envelope), reset to "follow tail" by force (a fresh user message implies the user wants to see the response). The 40px threshold is wide enough to absorb sub-pixel layout shift but tight enough that a deliberate user scroll-up disengages.

**No "scroll to latest" floating button** in skeleton. If a user scrolls up during streaming, they can scroll back down themselves. (Adding such a button is a v2 UX polish item.)

### IP-3: Input lifecycle during a turn (locked decision)

| Phase | Input state | Send button state |
|-------|-------------|-------------------|
| Idle (no turn in flight) | enabled, placeholder = `Type a message...` | enabled iff input has content |
| User just pressed Enter | clear input, fire user-message envelope | disabled |
| Turn-start envelope received | input remains **disabled** (the user can't type while the avatar is replying) | disabled |
| Sentences streaming | input remains disabled | disabled |
| chain-end envelope received | input re-enabled, focus restored to input | re-enabled iff input has content |
| Stream-error / context-overflow banner | input disabled (banner has its own retry path) | disabled |

**Why disable input during a turn:** matches OLVT behavior (`websocket-handler.tsx`'s `chain-start` blocks subsequent sends until `chain-end`) and prevents user interleaving that would corrupt the orchestrator's `_memory` ordering (the user message is already in `_memory` at the moment Enter was pressed; receiving another user message before chain-end would create two consecutive user roles which most LLM APIs reject). Phase 2 ships with input disabled during turns — a queue-and-send-after-chain-end UX is a v2 enhancement.

**Visual cue for disabled input:** existing Phase 1 input disabled state (CSS `disabled` selector — `--muted-foreground` placeholder, reduced opacity, cursor: not-allowed). No new style needed.

### IP-4: ActionIntent log line surfacing (D-14, locked)

When `actions_extractor` emits an `ActionIntent`, loguru writes to sidecar stdout:

```
[INTENT] kind=expression name=joy strength=1.0 avatar=teto
```

The sidecar→renderer log channel (`{type:"log", level:"info", message:"[INTENT] kind=…"}`) is the existing Phase 1 envelope. The Logs drawer's render loop (`src/shell.jsx:237-239`) gains one new visual rule:

```
// inside the .line render
const line = log.message;
if (line.startsWith("[INTENT]")) {
  return (
    <div className="line">
      <span style={{ color: "var(--success)" }}>[INTENT]</span>
      <span>{line.slice(8)}</span>
    </div>
  );
}
// else render as before
```

**SC #2 verification path (executor reference):** type `hello [joy] world` in the chat input → sidecar's `actions_extractor` strips `[joy]`, emits ActionIntent → Logs drawer (if enabled in Settings §15) renders the line with green `[INTENT]` prefix → chat bubble shows `hello world` (no bracket, no `joy`). All within ~1s of LLM stream-end.

**If the Logs drawer is disabled in Settings:** the loguru line still flows to the sidecar's stdout and the on-disk log file (Phase 1's `[Open log folder]` button surfaces the file). The drawer is the *primary* visualization but not the only one.

### IP-5: Stub-TTS log line (D-23, supporting)

Stub TTS emits a sibling log line per sentence:

```
[STUB-TTS] sentence_id=42 text="On a quiet afternoon, the cat noticed a glint."
```

This line flows through the same `[type:"log"]` envelope. **No special coloring** — renders in `--foreground` mono. The reason: stub-TTS is a development-trace breadcrumb; Phase 3 replaces stub-TTS with real synth and these lines transform into `[TTS] sentence 2/3 synth 312ms` (which is visible in Phase 1's USERFLOW.md K mockup). Highlighting stub-TTS lines now would create churn when Phase 3 swaps in the real format.

### IP-6: "Thinking…" → first sentence transition (D-03)

The `full-text:"Thinking…"` envelope arrives after `chain-start` and before any `audio-payload`. Renderer behavior:

```
on receive {type:"full-text", text:"Thinking…"}:
  if no assistant bubble exists for the current turn:
    create assistant bubble, body = THINKING_PLACEHOLDER (italic, muted)
  else:
    bubble already exists (rare race); skip

on receive {type:"audio-payload", display_text:{text}, ...} (first one in turn):
  bubble.body = text  // replace THINKING_PLACEHOLDER with first sentence
  (subsequent audio-payload arrivals: bubble.body += text per IP-1)
```

The transition is a wholesale text replacement, no animation. The "Thinking…" placeholder remaining visible for the duration between turn-start and first-sentence-arrival is the entire visual signal that the avatar is processing. Typical duration on LM Studio with cold-start: 1-30s; with warm model: 200-800ms.

**Edge case — full-text arrives but no audio-payload ever does (LLM error mid-thought):**
The bubble stays in "Thinking…" state until the orchestrator emits `chain-end` (which it must, even on error — single-conversation.py:174 in OLVT guarantees this). When chain-end arrives without any sentence having streamed, replace "Thinking…" with the `STREAM_ERROR` banner copy in `--destructive` color and `⚠` prefix. The bubble persists in error state in the chat history.

### IP-7: Force-new-message + chain-end pair (D-04)

Already described in IP-1 state machine. Restated for Aesthetic Guardrails clarity: **neither envelope produces a visible UI change at the moment of receipt.** Their effect is:

| Envelope | UI side effect | When observable |
|----------|----------------|-----------------|
| `force-new-message` | sets a renderer-internal flag | observable when the *next* assistant sentence arrives (it lands in a fresh bubble) |
| `chain-end` | re-enables input + focus | observable as the input cursor blinking and accepting keystrokes |

This is intentional: a "turn complete" badge or divider would add chrome that contradicts the OLVT pattern and the Phase 1 calm-aesthetic guardrails.

## Accessibility

**Inherited from `01-UI-SPEC.md`** — chat bubbles use `role="log"` `aria-live="polite"`, focus-rings on input/buttons, AA contrast verified for all 7 theme variants.

### Phase 2 additions

| Surface | a11y requirement | Implementation |
|---------|------------------|----------------|
| "Thinking…" affordance | screen reader announces "Thinking" (the ellipsis is decorative) | The text node renders as plain `Thinking…`; `aria-live="polite"` on the parent log container handles announcement automatically. **Do NOT add `aria-busy="true"`** — that's reserved for components in a busy/loading state where the entire surface should be skipped; here the bubble itself is the announcement vehicle. |
| Growing bubble during streaming | screen reader does not re-announce the entire bubble on every sentence arrival | `role="log"` on the chat scroll container with `aria-live="polite"` ensures only newly-added content is announced. The bubble's *body* string is being mutated (concat), not re-mounted; ARIA live regions handle text-content mutations as additive announcements. **Verify in plan-time** that the React reconciler uses a stable key on the bubble (e.g., turn ID) so it's not unmounted/remounted between sentences. |
| `[INTENT]` log lines | colorblind users can identify them as intent events without seeing green | The prefix `[INTENT]` is itself a unique 8-char string that screen readers and search-in-logs can find. Color is *additional* signal, not the sole signal. (Same pattern as Phase 1's status icon: color + glyph, never color alone.) |
| Disabled input during turn | screen reader announces disabled state | `<input disabled>` is the standard pattern; Phase 1's CSS already styles this. No `aria-disabled` needed when the native `disabled` attribute is present. |
| `STREAM_ERROR` / `CONTEXT_OVERFLOW` banners | announced as alerts | Reuse Phase 1's banner pattern: the `<div class="banner">` is rendered as a new DOM node (not a mutation of an existing one) → screen readers announce it. **Add `role="alert"`** to these two new banners (Phase 1's existing banners are status-level and don't carry alert role; Phase 2's two new banners are destructive errors warranting the alert role for assertive announcement). |
