## Overview

v3.0 refactors audio I/O into sidecar-owned provider systems so the existing Piper ordered playback, renderer audio payloads, RMS/lipsync, VTS compositor, and conversation-history semantics remain intact while users gain GPT-SoVITS voice output and Chinese/English voice input. Phases 16 through 20.2 are closed: provider systems, Settings, local STT live acceptance, final-only PTT/VAD, stop, and edit/regenerate are implemented. The remaining roadmap is evidence work: harden final-transcript code-switch quality, then empirically decide no-headphones/AEC behavior.

## Phase Details

### Phase 16: Audio Contracts + TTS Provider Shell
**Goal**: The app has a stable audio-provider foundation that preserves existing Piper playback and lipsync while making TTS/STT configuration safe to evolve.
**Depends on**: Phase 15
**Requirements**: AUDIO-02, AUDIO-03, AUDIO-04, TTS-05, PERF-03
**Success Criteria** (what must be TRUE):
  1. Existing Piper chat responses still play in sentence order and drive renderer audio payloads plus VTS lipsync/RMS exactly through the current playback path.
  2. Audio settings persist through a versioned config migration without breaking existing LLM, VTS, conversation history, or avatar catalog settings.
  3. Provider health checks return clear unavailable, missing credential, external-service failure, and timeout states instead of empty turns or silent fallback.
  4. Slow or failed provider work cannot block the chat WebSocket, compositor, HUD traffic, or ordered TTS queue.
**Plans**:
- **Wave 1**:
  - `16-01`: Audio Contracts and Config Migration - shared provider/config/health contracts, schemaVersion 2 migration, and sidecar audio-config env handoff.
- **Wave 2** *(blocked on Wave 1 completion)*:
  - `16-02`: TTS Provider Shell and Piper Adapter - provider-neutral synthesis surface while preserving ordered Piper playback/RMS payload behavior.
- **Wave 3** *(blocked on Wave 2 completion)*:
  - `16-03`: Provider Health, Failure Semantics, and Non-Blocking Regression - typed health endpoint, queue-safe failure behavior, compact diagnostics, and final evidence.
- **Wave 4** *(gap closure after UAT)*:
  - `16-04`: Restore Audible Voice Playback - renderer playback for non-empty sidecar WAV payloads while preserving chat streaming and VTS lipsync state.
**Cross-cutting constraints:**
- Existing Piper ordered playback, renderer `AudioPayloadMessage`, and `SpeechEnvelopePayload`/RMS lipsync path must remain the regression baseline.
- Config migration must preserve existing LLM, plugin, VTS, conversation history, and avatar catalog behavior.
- Provider failure states must be explicit and typed; no silent mid-turn provider fallback.
- Provider synthesis and stream writes must not block chat WebSocket, compositor, HUD, or ordered queue traffic.

### Phase 20.2: CPU Local STT Enhancement
**Goal**: Voice input feels responsive on local CPU STT without degrading model quality, and users can recover from STT typos through explicit stop and edit/regenerate flows.
**Depends on**: Phase 20.1
**Requirements**: STT-03, STT-06, VIN-02, VIN-04, VIN-05, PERF-01
**Success Criteria** (what must be TRUE):
  1. Chat voice input no longer runs live preview transcription; only final captured audio is transcribed and submitted.
  2. faster-whisper CPU local STT reuses loaded models, warms safely when appropriate, and applies bounded CPU thread tuning without changing the selected model quality.
  3. During assistant streaming or TTS speaking, the input row replaces the paper-plane Send button with a red square Stop button.
  4. Stop cancels the active turn, stops queued/playing audio, discards partial assistant output from history, and unlocks editing without removing the user message.
  5. When no turn is active, users can edit a sent user message; saving the edit truncates later conversation messages and dispatches the edited message as a normal fresh request.
  6. Voice transcript typo recovery works through the same edit/regenerate path after final STT submission, with no grace-window undo and no delayed dispatch.
**Plans**: 5
**Plan list**:

- [x] 20.2-02-PLAN.md - Stop-current-turn protocol and send/stop UX

- [x] 20.2-04-PLAN.md - Cross-feature regression and UAT evidence
- [x] 20.2-05-PLAN.md - Gap closure: stopped-message edit/regenerate anchoring
**Wave dependencies**:
  - Wave 1: 20.2-01 removes live preview transcription, keeps final-only voice input, and tunes faster-whisper CPU runtime/warmup.
  - Wave 2: 20.2-02 depends on 20.2-01 and adds stop-current-turn protocol, sidecar cancellation, renderer audio stop, and Send-to-Stop UI.
  - Wave 3: 20.2-03 depends on 20.2-02 and adds sent-message edit/regenerate through conversation truncation and normal text-input dispatch.
  - Wave 4: 20.2-04 depends on 20.2-01 through 20.2-03 and verifies the combined final-only STT, stop, and edit/regenerate workflow.
