# Phase 20.1 Research: Settings UX Refactor + Enablement Simplification

## Scope Confirmation

Phase 20.1 is not a voice-only polish pass. The Settings screen needs a whole-screen information architecture refactor and a behavior cleanup that removes stale mandatory test-before-enable gates from all configurations.

Locked decisions from discussion:

- Replace the current single long scroll plus anchor pills with a sidebar category model.
- Use user-facing category names while keeping underlying system names visible inside panels, for example `Voice output (TTS)` and `Voice input (STT)`.
- Default Settings to `Connection`.
- Hide deferred disabled placeholders until those areas are real.
- Enable/save blockers must be limited to required inputs, explicit consent, local assets/cache, valid references/paths, and real unavailable dependencies.
- Tests remain available as optional standalone diagnostics only.
- Remove "not tested yet", "test required", checklist, and gate-adjacent explanations.
- Failed optional tests must not mutate saved/enabled configuration.
- Runtime provider failures must remain visible and must not silently fall back between providers.

## Current Code Shape

### Settings Screen

Primary file: `apps/renderer/src/screens/Settings/Settings.tsx`

The current Settings screen is still organized as a single long scrolling page:

- A header comment documents the old model as a "single long scroll" with numbered sections and anchor pills.
- The `anchors` array exposes `Connection`, `Avatars`, `Plugins`, `VTube Studio`, `TTS`, `Voice in`, `Conversation`, `Memory`, `Appearance`, `Diagnostics`, and `About`.
- The renderer emits an `anchor-pills` block, then renders each section sequentially inside `settings-scroll`.
- `C.PLACEHOLDERS.filter((p) => p.num > 8)` still renders deferred sections for Skills, Agent, Scheduler, Form factor, and Hotkeys.

This makes the screen hard to reason about because related dependencies are scattered, voice settings are split by historical implementation layers, and deferred product areas are visible even though they are not usable.

### Copy

Primary file: `apps/renderer/src/lib/copy.ts`

There is still explicit test-gate copy:

- GPT-SoVITS help says activation requires health check and audible test synthesis.
- GPT-SoVITS not-ready copy tells the user to run health check and test synthesis before activation.
- Voice input copy says the microphone turns on only after the selected STT provider passes a test.
- Save/enable copy says voice input is off because the selected STT provider has not passed a test.
- STT test copy tells the user to run a test before enabling Chat voice input.

These strings conflict with the Phase 20.1 decision that tests are optional standalone diagnostics.

### GPT-SoVITS Test Gate

Primary files:

- `packages/contracts/py/contracts/audio_provider.py`
- `packages/contracts/ts/audio-provider.ts`
- `sidecar/src/sidecar/tts/tts_gateway.py`
- `apps/renderer/src/screens/Settings/Settings.tsx`

Current behavior:

- `GptSoVitsActivationGate` includes `health_check_passed`, `test_synthesis_passed`, timestamps, and `active_allowed`.
- Settings computes `activationReady` from health pass, test pass, a candidate preset, and reference metadata.
- The activate button is disabled until the test gate passes.
- The sidecar TTS gateway rejects GPT-SoVITS unless health and test gates are present and active.

Required Phase 20.1 change:

- GPT-SoVITS activation must be blocked by missing required configuration only: URL/service availability when activation is attempted, preset, reference audio metadata, and required path/reference fields.
- Health/test results may remain visible as diagnostics, but must not be activation permission.
- Runtime synthesis must fail visibly if GPT-SoVITS is unhealthy or unavailable; it must not silently switch to another provider mid-turn.

### STT Test Gate

Primary files:

- `packages/contracts/py/contracts/audio_provider.py`
- `sidecar/src/sidecar/stt/readiness.py`
- `sidecar/src/sidecar/admin/audio.py`
- `apps/electron-main/src/ipc.ts`
- `apps/electron-main/src/safe-storage.ts`
- `apps/renderer/src/screens/Settings/Settings.tsx`
- `apps/renderer/src/screens/Settings/VoiceInputSection.tsx`

Current behavior:

- `STTProviderReadiness` includes health/test pass flags, fingerprint invalidation, and `active_allowed`.
- `readiness_from_test_result` grants activation only when a test passes.
- `validate_stt_readiness` invalidates activation if health/test flags are missing or stale.
- Sidecar voice input readiness returns blocked states such as `readiness_not_active`.
- `/stt/enable` and `/voice-input` depend on readiness.
- Renderer helpers block save/enable when `active_allowed`, `health_check_passed`, or `test_transcription_passed` are false.
- `runSttTest` can save `enabled: false` after a failed optional test.

Required Phase 20.1 change:

- STT enablement must be based on required setup only: selected provider, enabled flag, local model/cache presence for local providers, explicit consent/API key for cloud providers, and available microphone/runtime dependencies where applicable.
- Test result state must be diagnostic and may be invalidated/cleared when related config changes, but must not control enablement.
- A failed STT test must never disable a saved provider or mutate enabled state.
- Runtime transcription must fail visibly if the configured provider cannot run.

## Implementation Risks

- Contract fields such as `active_allowed` may be referenced across generated TypeScript, Electron defaults, sidecar validation, and renderer state. Either remove them with a migration/codegen pass or keep them as diagnostic compatibility fields with no enablement authority.
- `Settings.tsx` is large and will be touched by both layout and behavior work. Execution should sequence renderer work rather than parallelize edits to this file.
- Tests currently encode test-gate behavior. Expect renderer, sidecar, Electron, and contract tests to need updates.
- Removing gates must not weaken cloud privacy boundaries. Consent and credentials remain hard blockers.
- Removing gates must not introduce silent fallback. Provider runtime failure should still surface as a typed, visible failure state.

## Recommended Plan Decomposition

1. Clean up cross-tier enablement contracts and runtime gate behavior outside the Settings layout.
2. Refactor Settings into a sidebar category model and hide deferred placeholders.
3. Update renderer Settings behavior and copy so optional tests are diagnostics only.
4. Add final regression coverage and verification evidence that no test-gate language or behavior remains.
