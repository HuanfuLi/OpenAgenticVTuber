# 20.1-06 Plan: Voice Output Activation And Action Hierarchy

<objective>
Make Voice output logically transparent by normalizing Piper and GPT-SoVITS enablement and cleaning up confusing GPT-SoVITS button priority.
</objective>

<execution_context>
This is a UX and behavior gap closure. Preserve Phase 20.1's rule that diagnostics are optional standalone tests. Do not restore mandatory health/test gates for activation.
</execution_context>

<tasks>

## 1. Normalize Provider Activation

Refactor the Voice output panel so provider selection and activation are consistent:

- Piper has an explicit, understandable activation/save action or the shared activation action clearly applies to it.
- GPT-SoVITS activation uses the same conceptual path: choose provider/preset, satisfy required fields, activate.
- Switching the active provider persists the selected provider and triggers the same sidecar/runtime refresh behavior expected for active TTS changes.
- The UI makes the currently active provider and pending unsaved provider obvious.

## 2. Rebuild GPT-SoVITS Action Priority

Rearrange GPT-SoVITS controls:

- `Activate voice preset` is placed near the selected preset/provider state and styled as the primary workflow action.
- `Run health check` and `Test synthesis` are grouped as optional diagnostics with secondary styling.
- `New preset` and `Save preset` use priority that matches their risk and frequency; saving/updating an existing preset should not be visually weaker than creating a separate one when editing.
- Destructive actions remain visually distinct and guarded.

## 3. Update Copy And Tests

Update user-facing copy so it explains required activation inputs without saying tests are required.

Add renderer tests for:

- Piper can be activated through the visible workflow;
- GPT-SoVITS can be activated without prior health/test success when required fields are present;
- provider switch invokes expected save/restart/refresh path;
- diagnostics do not outrank activation in rendered button variants/order.

</tasks>
