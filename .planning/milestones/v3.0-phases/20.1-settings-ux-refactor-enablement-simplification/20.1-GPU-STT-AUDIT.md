# STT GPU Compatibility Audit

## User-Reported Failure

- Settings `Test selected STT` with faster-whisper CUDA stayed stuck at `Testing...`.
- Chat PTT with faster-whisper CUDA stayed stuck at `Finalizing`.

## Local Environment Findings

- `nvidia-smi` detects two NVIDIA GPUs:
  - device 0: NVIDIA GeForce RTX 5060 Ti, compute capability 12.0, driver 595.71
  - device 1: NVIDIA GeForce RTX 4060, compute capability 8.9, driver 595.71
- Python sidecar package versions:
  - `faster-whisper 1.2.1`
  - `ctranslate2 4.7.1`
  - `torch 2.11.0`
  - `torchaudio 2.11.0`
  - `numpy 2.4.4`
- CTranslate2 reports CUDA devices and CUDA compute types:
  - device count: 2
  - compute types include `float16` and `int8_float16`
- PyTorch reports:
  - `torch.cuda.is_available() == False`
  - `torch.version.cuda == None`
- Windows PATH does not contain the CUDA 12 CTranslate2 inference libraries:
  - `cublas64_12.dll` missing
  - `cublasLt64_12.dll` missing
  - `cudnn64_9.dll` missing
- Clean subprocess probe against the downloaded faster-whisper `small` model:
  - model construction succeeds on CUDA device 0 and device 1
  - first inference fails with `RuntimeError: Library cublas64_12.dll is not found or cannot be loaded`
  - CPU inference against the same model succeeds

## Compatibility Interpretation

- faster-whisper CUDA uses CTranslate2, not PyTorch. CTranslate2 can enumerate CUDA devices without proving the full inference stack is loadable.
- CTranslate2/faster-whisper load CUDA runtime libraries lazily. A model can construct successfully while inference later fails when cuBLAS is first needed.
- FunASR currently has no Settings GPU runtime selector. The installed PyTorch is CPU-only, so FunASR GPU acceleration is not available through this sidecar environment as configured.
- The app must not assume that "NVIDIA driver present" means "STT GPU path is safe." Runtime/model initialization can still fail or hang.
- Official faster-whisper guidance requires cuBLAS for CUDA 12 and cuDNN 9 for CUDA 12 when using current CTranslate2. Official CTranslate2 guidance also says Windows/Linux wheels support GPU execution only when the CUDA runtime libraries are installed.
- NVIDIA Blackwell compatibility guidance matters for the RTX 5060 Ti: CUDA binaries must include compatible cubins or forward-compatible PTX. This app cannot infer compatibility from `nvidia-smi` alone.

## Root Cause In App Behavior

Phase 20.1 added faster-whisper runtime validation, but that validation only constructed the model. It did not prove CUDA inference libraries could be loaded. On this machine, CTranslate2 reports CUDA devices and the model constructor returns, but transcription fails when cuBLAS is loaded lazily.

The first timeout fix used `asyncio.to_thread`, which returned typed timeout responses but could not kill the underlying stuck native worker thread. If a CTranslate2 call wedged while holding the local provider lock, later Settings or Chat requests could keep timing out behind the stale lock. Renderer state then stayed in or appeared stuck in:

- Settings: `Testing...`
- Chat PTT: `Finalizing`

## Fix Implemented

- Moved STT runtime validation and transcription work behind timeout-protected worker execution in the sidecar.
- Added typed timeout failures for runtime validation and transcription.
- Added Electron IPC abort timeouts for Settings STT diagnostics and Chat voice-input transcription requests.
- Preserved the no-silent-provider-fallback invariant: CUDA failure does not switch STT providers automatically.
- Existing Settings CPU recovery remains explicit: user can switch faster-whisper runtime back to CPU and save.
- Added Windows CUDA library preflight for faster-whisper CUDA before model construction:
  - checks `cublas64_12.dll` / `cublasLt64_12.dll` / `cudnn64_9.dll` and reports missing required libraries as a typed runtime failure
  - avoids entering the CTranslate2 transcription path when required CUDA 12 libraries are absent
- Evicts cached local STT provider instances and locks after timeout, so a wedged CUDA worker does not poison all later requests for that runtime key.

## Residual Live Check

Retry faster-whisper CUDA in the app. Expected behavior after this fix:

- With the current machine PATH, the diagnostic should fail fast with a missing CUDA 12 library error.
- If CUDA libraries are installed later, CUDA should be retested from Settings. If it still hangs, the timeout path should return an error and evict the poisoned provider cache.
- Until CUDA dependencies are installed, switch faster-whisper runtime to CPU and save settings.

## Follow-Up Provider Note

- Consider a `whisper.cpp` local STT provider with Vulkan acceleration as the easier Windows GPU path.
- Rationale: Vulkan avoids CUDA/cuDNN/cuBLAS provisioning and is driven by the graphics driver stack.
- Keep faster-whisper CUDA as an advanced high-performance path after dependency probing passes.
