# 20.1-04 Plan: Regression Evidence And Phase Verification

<objective>
Close Phase 20.1 with focused regression coverage and verification evidence that Settings is understandable, required-input-only, privacy-safe, and free of mandatory test-before-enable gates.
</objective>

<execution_context>
This is a verification and gap-closure plan. It should not introduce broad new UI architecture unless prior plans missed a Phase 20.1 success criterion. Prefer tests, search evidence, and small fixes tied directly to failed verification.
</execution_context>

<tasks>

## 1. Run Goal-Backward Verification

Verify each Phase 20.1 roadmap success criterion directly:

1. Settings is organized around user goals and provider state.
2. Required/optional/blocked/recommended states are understandable without scattered dependency copy.
3. No configuration requires a passing test before save/enable.
4. Tests remain available as optional diagnostics only.
5. Cloud consent, redaction, visible failures, and no silent fallback remain intact.

Document evidence in `20.1-VERIFICATION.md`.

## 2. Add Missing Regression Tests

Fill any coverage gaps found during verification:

- Settings sidebar/category default and panel selection.
- Hidden deferred placeholders.
- Required-input blockers for cloud consent/API key, local model/cache, and GPT-SoVITS reference audio.
- Optional diagnostic tests not mutating enabled state.
- Runtime provider failures visible without silent fallback.
- Redaction of secrets, reference paths, and transcript-like diagnostics.

## 3. Search For Stale Gate Language

Run repository-wide searches:

```powershell
rg "not tested yet|test .*required|required .*test|Run .*test.*before|has not passed a test|before activating|before enabling|readiness_not_active|active_allowed" apps packages sidecar
```

For any remaining match:

- remove it if user-facing or behavior-gating;
- justify it in verification if it is generated compatibility, historical migration handling, or diagnostic-only internal state;
- add a test if the match could regress into behavior.

## 4. Run Final Checks

Run the relevant automated checks:

```powershell
npm run check:contracts
npm test -- --run apps/renderer/tests/Settings.test.tsx apps/renderer/tests/VoiceInputSection.test.tsx apps/electron-main/tests/ipc-voice-input.test.ts
pytest sidecar/tests/admin/test_audio_voice_input_endpoint.py sidecar/tests/admin/test_audio_status_endpoint.py sidecar/tests/stt/test_stt_readiness.py
```

If a live GPT-SoVITS server, STT provider, microphone, or cloud credential is unavailable, record that as environment-blocked UAT rather than faking success.

</tasks>

<verification>

The plan is complete only when `20.1-VERIFICATION.md` contains:

- PASS/FAIL for each roadmap success criterion;
- commands run and outcomes;
- stale-gate search results;
- any environment-blocked UAT with exact blocker;
- residual risks, if any.
</verification>
