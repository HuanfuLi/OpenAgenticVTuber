# Phase 20.1: Settings UX Refactor + Enablement Simplification - Context

**Gathered:** 2026-05-12T03:21:59-04:00
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 20.1 audits and refactors the entire Settings screen so configuration is understandable, logically organized, and no longer blocked by mandatory test-before-enable gates. This phase is not voice-only: it covers the full Settings experience, including Connection, Avatars, Body motion plugin, VTube Studio, TTS / Voice out, Voice in, Conversation, Memory, Appearance, Diagnostics, and About.

The phase must remove stale readiness/test-gate logic that obstructs configuration while preserving real required inputs, explicit cloud consent, required local assets, redaction, visible runtime failures, and the no-silent-provider-fallback invariant.

</domain>

<decisions>
## Implementation Decisions

### Settings Navigation And Whole-Screen Organization
- **D-01:** Replace the current single long-scroll Settings page and anchor pills with a sidebar category model: category list on the left, selected settings panel on the right.
- **D-02:** Use a hybrid information architecture: user-facing category names, with underlying systems visible inside each panel. Example: a user-facing `Voice` category may expose `Voice output (TTS)` and `Voice input (STT)` sections.
- **D-03:** Phase 20.1 must audit and rearrange the entire Settings screen, not only voice settings.
- **D-04:** Hide deferred placeholders until real. Do not show Skills, Agent, Scheduler, Form factor, or Hotkeys as disabled placeholder settings unless those features have usable settings surfaces.
- **D-05:** The first/default Settings panel should be Connection, because LLM/provider setup is the app's first required dependency.

### Enablement Rules
- **D-06:** Enablement and saving should be blocked only by required inputs. Examples: missing cloud consent/API key for cloud STT, missing local model selection/cache where a local provider cannot operate without it, missing GPT-SoVITS URL/preset/reference fields, or missing required reference-audio metadata.
- **D-07:** Tests and health checks must not block enabling or saving solely because they have not passed or have not been run.
- **D-08:** Full cleanup is required: remove or replace stale readiness/test-gate concepts across contracts, sidecar, Electron, renderer, and tests wherever they obstruct configuration.
- **D-09:** Test results may remain as standalone diagnostics/history while the current screen is open, but not as activation permission.
- **D-10:** Runtime use of an untested or unhealthy selected provider should fail visibly with no silent fallback. Attempt the selected provider, show a clear failure if it does not work, and do not silently switch providers.

### Optional Test Behavior
- **D-11:** Tests stay inside each relevant panel. LLM test remains in Connection, GPT-SoVITS health/test remains in Voice output, and STT test remains in Voice input.
- **D-12:** Keep simple test labels such as `Test connection`, `Test synthesis`, and `Test selected STT`. Do not rename tests into certification-like or checklist-like language.
- **D-13:** Clear previous test results when related configuration changes so stale passed or failed state does not hang around.
- **D-14:** Failed optional tests must never mutate saved/enabled configuration. Failed tests show errors only; they do not disable providers, clear enabled state, or change saved settings.
- **D-15:** Tests are purely standalone and optional. Do not add "not tested yet" warnings, explanatory copy, checklists, or gate-adjacent language around tests.

### Transparency Language
- **D-16:** Explain required inputs inline only where the dependency is unclear, such as cloud consent, local model cache, reference audio, or provider-specific paths.
- **D-17:** Do not explain, warn, checklist, or gate around optional tests. The UI should not make tests feel like part of enablement.
- **D-18:** For complex panels with multiple missing required inputs, use both inline validation next to the field and a concise panel-level message listing missing required inputs.
- **D-19:** Required-field copy should be direct and minimal. Avoid heavy explanatory copy on every required field.

### the agent's Discretion
Downstream agents may choose exact category names, sidebar styling, component extraction boundaries, panel routing/state approach, and validation-message placement consistent with the decisions above and existing app design patterns. Do not re-open the whole-screen scope, sidebar model, required-input-only enablement policy, optional-test behavior, or no-silent-fallback invariant unless the roadmap changes.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Settings.tsx` already centralizes all current Settings sections and is the primary refactor target.
- `RadioRow`, `kv-row`, `settings-form`, `preset-card`, and existing button/switch styles can seed the new sidebar/panel layout without inventing a new design language from scratch.
- `VoiceInputPreferencesFields` already isolates PTT/VAD preference controls and can move into the new Voice panel.
- `ProviderSelect` and `TestLog` from LLM setup are reused inside Connection and should remain available for optional connection testing.

### Established Patterns
- Renderer copy is centralized in `apps/renderer/src/lib/copy.ts`; gate language and new IA labels should be updated there rather than hardcoded in components.
- Config persistence flows through Electron/preload APIs and safe storage; Settings should keep this boundary rather than reaching directly into sidecar or filesystem state.
- Pydantic contracts are the source of truth for cross-tier audio provider models; contract cleanup must regenerate TypeScript and JSON schema mirrors.
- Current test coverage expects STT and GPT-SoVITS gate behavior. Phase 20.1 must update tests to verify required-input validation and optional standalone tests instead.

### Integration Points
- `Settings` root should become a sidebar/category shell instead of rendering all sections in a fixed long-scroll sequence.
- Connection remains the default panel.
- Voice panel should expose voice output (TTS/GPT-SoVITS/presets/reference audio) and voice input (STT/PTT/VAD/cache/cloud consent) while keeping TTS/STT system names visible.
- Required-input validation should live close to each panel/control and should not reuse test-passed fields as activation permission.
- Runtime provider failures should remain visible through existing chat/settings/status surfaces without silently changing provider selection.

</code_context>
