# 20.1-05 Plan: Truthful Hosted-Provider State

<objective>
Close the UAT issue where OpenAI/Gemini/Claude-style API support still appears as stale v2 mock state after v2 has already shipped.
</objective>

<execution_context>
Start by auditing the actual LLM setup support already present in the renderer/store/Electron layer. Do not invent backend support that is not wired. The fix can either expose implemented hosted providers or remove stale disabled placeholders in favor of a truthful current setup path.
</execution_context>

<tasks>

## 1. Audit Current Hosted LLM Support

Inspect:

- `apps/renderer/src/lib/copy.ts`
- `apps/renderer/src/screens/LLMSetup/ProviderSelect.tsx`
- `apps/renderer/src/screens/LLMSetup/LLMSetup.tsx`
- `apps/renderer/src/state/setup-store.ts`
- related renderer tests

Determine whether OpenAI, Anthropic/Claude, and Gemini can be configured directly today or whether only the Custom OpenAI-compatible path is functional.

## 2. Remove Stale V2 Mock State

Replace stale provider statuses/tooltips/comments:

- no `Coming v2`;
- no `Comming`;
- no disabled cards that imply an already-shipped milestone is future work;
- provider copy must name the actual supported path.

If direct hosted providers are not implemented, show a current-state alternative such as Custom OpenAI-compatible for compatible endpoints and clearly separate unsupported native integrations from configured providers.

## 3. Add Regression Coverage

Add or update tests that assert:

- Connection/LLM setup does not render `Coming v2` or `Comming`;
- available provider choices match implemented behavior;
- unsupported providers do not appear as mock selectable cards.

</tasks>
