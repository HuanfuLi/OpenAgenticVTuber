# 20.1-07 Plan: Faster-Whisper CUDA Runtime Regression

<objective>
Fix the faster-whisper CUDA transcription regression exposed during UAT while preserving explicit provider failures and no silent provider fallback.
</objective>

<execution_context>
The user reported the exact failure `faster-whisper CUDA transcription failed. Check NVIDIA driver, CUDA availability, and ctranslate2 GPU support.` Treat this as a real runtime regression, not as acceptable environment noise. The fix must not hide failures by switching STT providers mid-turn.
</execution_context>

<tasks>

## 1. Reproduce And Isolate CUDA Runtime Handling

Inspect the current faster-whisper load/transcribe path:

- `sidecar/src/sidecar/stt/providers/faster_whisper_provider.py`
- `sidecar/src/sidecar/admin/audio.py`
- readiness fingerprint/runtime config paths
- renderer runtime-device controls

Identify whether the regression comes from forcing CUDA, using an unsupported compute type, missing preflight, stale readiness after runtime changes, or a real local driver/library mismatch.

## 2. Add Explicit Runtime Validation

Add a validation path for faster-whisper runtime settings:

- CPU remains default.
- CUDA can be selected only as an explicit runtime choice.
- CUDA load/preflight failures are reported before normal transcription is treated as ready.
- Failure metadata includes runtime device and compute type but remains redacted.
- Do not silently fall back to a different STT provider. If CPU fallback within faster-whisper is offered, it must be an explicit user action or clearly visible recovery choice, not hidden mid-turn behavior.

## 3. Repair Settings Feedback

Update Voice input Settings so CUDA runtime state is understandable:

- selecting CUDA communicates that it requires a working NVIDIA/CUDA/CTranslate2 stack;
- failed CUDA validation points to the runtime setting and offers a clear path back to CPU;
- failed diagnostics do not disable or rewrite enabled voice-input config unless the user explicitly saves a different runtime.

## 4. Add Regression Tests

Cover:

- CUDA load failure produces typed/actionable failure before transcription readiness is assumed;
- CPU path remains default and unaffected;
- Settings renders CUDA failure feedback and keeps provider selection explicit;
- no silent provider fallback happens on CUDA failure.

</tasks>
