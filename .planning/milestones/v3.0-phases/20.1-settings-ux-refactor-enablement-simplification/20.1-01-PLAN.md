# 20.1-01 Plan: Required-Input Enablement Contracts And Runtime Cleanup

<objective>
Remove cross-tier mandatory test-before-enable behavior while preserving required-input validation, explicit cloud consent, local model/cache requirements, redacted diagnostics, and visible provider runtime failures.
</objective>

<execution_context>
Start from the existing Phase 19/20 STT readiness and Phase 17 GPT-SoVITS activation contracts. These contracts may be removed, renamed, or retained for compatibility, but after this plan they must no longer decide whether a user may save or enable a configuration.

Do not refactor the Settings layout in this plan. Touch renderer files only if generated types/defaults require compile fixes. The renderer behavior and copy cleanup happen in later plans.
</execution_context>

<tasks>

## 1. Map All Test-Gate Call Sites

Search for the stale gate vocabulary before editing:

- `active_allowed`
- `health_check_passed`
- `test_synthesis_passed`
- `test_transcription_passed`
- `readiness_not_active`
- `activationReady`
- `requires health check`
- `test required`

Classify each use as one of:

- diagnostic display state;
- required-input validation;
- runtime provider failure handling;
- stale activation permission that must be removed.

## 2. Update Contracts And Defaults

Revise audio-provider contracts so health/test result fields are no longer activation authority.

Acceptable implementation paths:

- remove activation/readiness gate fields and replace them with explicit diagnostic result fields; or
- keep compatibility fields temporarily but document and enforce that they are diagnostic only.

Required contract behavior:

- GPT-SoVITS required setup is represented by provider selection, base URL/server mode, active preset, reference audio metadata, and provider-specific required values.
- STT required setup is represented by selected provider, enabled flag, local model/cache presence for local providers, explicit cloud consent/API key for cloud providers, and language/provider settings.
- Optional health/test outputs can store last status, timestamp, provider id, latency, and redacted failure metadata.
- No contract should imply that a passing test is required to save, enable, or use the provider.

Regenerate and check contracts:

```powershell
npm run codegen:contracts
npm run check:contracts
```

## 3. Remove GPT-SoVITS Runtime Test Gate

In `sidecar/src/sidecar/tts/tts_gateway.py` and related server/config code:

- stop rejecting GPT-SoVITS solely because health/test diagnostics have not passed;
- validate required runtime inputs before synthesis: configured provider, active preset, base URL/service mode, reference audio metadata, and required generation fields;
- when the configured provider fails during runtime synthesis, return the existing visible failed-audio/error path;
- do not switch to Piper or another provider from this failure path.

Update tests to prove:

- untested but fully configured GPT-SoVITS is allowed to attempt synthesis;
- missing preset/reference/base URL still blocks with a clear required-input error;
- provider runtime failure remains visible and does not silently fallback.

## 4. Remove STT Runtime Test Gate

In `sidecar/src/sidecar/stt/readiness.py` and `sidecar/src/sidecar/admin/audio.py`:

- stop deriving runtime readiness from prior Settings test pass/fail;
- keep optional test results as diagnostic data only;
- make `/stt/enable`, `/stt/readiness`, and `/voice-input` report blockers only for required setup and runtime availability;
- preserve local model/cache blockers for local providers;
- preserve explicit cloud consent/API key blockers for cloud providers;
- preserve redaction of credential and transcript-like data.

Update tests to prove:

- voice input can be enabled with required setup even when no STT test has run;
- a failed STT test does not set `enabled: false`;
- missing local model/cache still blocks local STT;
- missing cloud consent/API key still blocks cloud STT;
- runtime transcription failure is visible and typed.

## 5. Update Electron Defaults And Bridges

Update Electron config defaults, migrations, IPC mapping, and tests for the changed contract shape.

Rules:

- Stored config should not reintroduce activation gates through defaults.
- Existing persisted configs with old readiness/activation fields should migrate safely.
- IPC responses should keep optional diagnostic status available to Settings.
- Redacted safe-storage behavior must remain intact.

</tasks>
