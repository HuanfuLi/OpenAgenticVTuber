# 22-02 Plan: Active-TTS Safety Verification

<objective>
Verify and harden the shipped active-turn safety behavior before adding any new self-speech guard: VAD must pause, PTT/final transcripts must queue safely, Stop must remain explicit, and edit/regenerate must remain available after recovery.
</objective>

<tasks>

## 1. Audit Current Active-Turn Signals

Read:

- `apps/renderer/src/ws/audio-player.ts`
- `apps/renderer/src/ws/store.ts`
- `apps/renderer/src/screens/Chat/useStreamingMessages.ts`
- `apps/renderer/src/screens/Chat/Chat.tsx`
- `apps/renderer/src/screens/Chat/VoiceInputControl.tsx`
- current tests for Chat voice input, streaming, and ws audio.

Write `.planning/phases/22-aec-spike-no-headphones-decision/22-SAFETY-AUDIT.md` with:

- current definition of active turn;
- when `isSpeaking` becomes true/false;
- whether queued browser audio can outlive `isSpeaking`;
- how VAD ignores speech during active turns;
- how final PTT/VAD candidates queue and promote;
- Stop and edit/regenerate recovery paths;
- any gap that requires 22-03 guard work.

## 2. Add Playback Lifecycle Diagnostics If Needed

If current `isSpeaking` is not precise enough to prove queued audio playback windows, extend `audio-player.ts` with a small subscription or diagnostic API that emits:

- queue non-empty / drained;
- current audio started;
- current audio ended or errored;
- playback stopped by user.

Rules:

- do not store audio bytes or object URLs in diagnostics;
- do not change sentence ordering or renderer audio playback semantics;
- preserve `stopAudioPlayback()`.

If current state is sufficient, add tests documenting that conclusion instead of adding new lifecycle code.

## 3. Lock VAD Pause Behavior

Add or update tests proving:

- VAD monitoring can continue while a turn is active;
- speech above threshold sets ignored reason `turn_in_progress`;
- VAD does not start recording during active TTS/turn-in-progress;
- UI labels this as `VAD paused`;
- disabling active turn lets VAD resume normal detection.

## 4. Lock Queued Final Transcript Behavior

Add or update Chat voice tests proving:

- final transcript during active TTS/input-disabled state is queued, not submitted immediately;
- queued transcript promotes only after speaking/input/settlement clears;
- queued transcript order remains first question, first answer, second question, second answer;
- cancellation of queued transcript does not stop active playback.

## 5. Lock Stop And Edit/Regenerate Recovery

Add or update tests proving:

- Stop during Thinking/streaming/TTS stops active playback and unlocks input;
- stopped user message remains anchored in chat order;
- save edit on stopped message dispatches a new `text-input`;
- normal sent-message edit/regenerate still works;
- voice final transcripts after Stop follow the same submission rules as typed text.

## 6. Record Guard Need

At the end of the audit, write a clear `Guard decision input` section in `22-SAFETY-AUDIT.md`:

- `existing_mitigations_sufficient: yes|no|unknown`;
- evidence rows supporting the value;
- exact failure mode if `no` or `unknown`;
- whether 22-03 should implement a guard or only document no-guard rationale.

</tasks>
