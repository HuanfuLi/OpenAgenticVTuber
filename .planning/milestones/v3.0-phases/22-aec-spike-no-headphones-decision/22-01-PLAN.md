# 22-01 Plan: AEC Diagnostics On Current Capture Path

<objective>
Instrument the shipped renderer microphone path so Phase 22 can prove what WebRTC echo-cancellation constraints were requested and applied without changing voice input semantics or storing audio.
</objective>

<tasks>

## 1. Audit Current Voice Capture Path

Read the current implementations before editing:

- `apps/renderer/src/state/audio-settings.ts`
- `apps/renderer/src/audio/voice-capture.ts`
- `apps/renderer/src/audio/vad-controller.ts`
- `apps/renderer/src/state/voice-input-store.ts`
- `apps/renderer/src/screens/Chat/VoiceInputControl.tsx`
- `apps/renderer/tests/voice-capture.test.ts`
- `apps/renderer/tests/voice-input-store.test.ts`

Write `.planning/phases/22-aec-spike-no-headphones-decision/22-AEC-DIAGNOSTICS.md` with:

- current selected-microphone flow;
- current `voiceInputAudioConstraints()` values;
- where PTT and VAD obtain streams;
- what metadata can be recorded safely;
- confirmation that no raw audio/transcript evidence is retained.

## 2. Add AEC Diagnostics Helper

Create `apps/renderer/src/audio/aec-diagnostics.ts`.

Implement deterministic helpers to:

- inspect `navigator.mediaDevices.getSupportedConstraints()` when available;
- build AEC-aware audio constraints from the selected microphone;
- request `echoCancellation` capability-aware, preferring supported advanced values only when safe and falling back to boolean/ideal `true`;
- inspect `track.getCapabilities()`, `track.getConstraints()`, and `track.getSettings()` when available;
- return a redacted diagnostic object with selected input metadata, requested constraint, supported flags, applied setting, sample rate, channel count, timestamp, and source path;
- tolerate missing browser APIs without throwing.

Do not retain raw audio, waveform samples, or transcript text.

## 3. Wire Diagnostics Into Existing PTT Capture

Update `VoiceCapture` so successful stream acquisition records a diagnostic snapshot in `voice-input-store`.

Rules:

- use the same MediaRecorder and final STT request flow;
- preserve the selected physical microphone behavior from Phase 21;
- preserve the existing fallback when no exact device constraint is requested and `OverconstrainedError` occurs;
- do not mark no-headphones as Ready because diagnostics look favorable.

## 4. Wire Diagnostics Into Existing VAD Monitor

Update `VadController` so the monitor stream also records AEC diagnostics.

Rules:

- do not alter RMS threshold semantics;
- do not alter `shouldIgnoreSpeech()` active-turn behavior;
- do not start capture while Teto is speaking;
- keep VAD disabled by default.

## 5. Extend Voice Input Store State

Add a metadata-only `aecDiagnostics` field to `voice-input-store` or the closest existing renderer voice state.

The field should:

- reset cleanly in `resetVoiceInputStoreForTests()`;
- be available to Settings in later plans;
- avoid provider secrets, raw audio, transcript text, and unredacted device IDs when display labels are enough;
- preserve existing `vadDiagnostics`, readiness, final/queued candidate, and settings behavior.

## 6. Add Tests

Add focused tests for:

- constraint construction when `echoCancellation` is supported;
- fallback when advanced values are unsupported/missing;
- partial browser APIs;
- PTT diagnostic snapshot after stream acquisition;
- VAD diagnostic snapshot after monitor stream acquisition;
- no raw audio/transcript fields in diagnostics;
- conservative VAD defaults remain unchanged.

</tasks>
