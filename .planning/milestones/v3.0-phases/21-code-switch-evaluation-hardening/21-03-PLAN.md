<implementation_notes>
- Start by inspecting current `Settings.tsx`; do not assume old component names or pre-20.1 layout.
- Keep badges short: Local, Cloud, Chinese/English, Code-switch tested, Limited code-switch, Requires API key, Local model, CUDA optional.
- Do not show scorecard tables, ranking charts, cloud quality comparison, or benchmark-like marketing in Settings.
- If contracts need a new capability string or recommendation/quality field, update Python source contracts first, then regenerate TypeScript/schema using the repo's established codegen path.
- If scorecard evidence does not justify a default change, leave FunASR as current local recommendation and label code-switch support accurately.
</implementation_notes>

<tasks>
1. Translate scorecard evidence into provider metadata.
   - Update `STTProviderDefinition`/catalog metadata with code-switch tested or limited-code-switch capabilities where evidence supports it.
   - Keep cloud providers opt-in and not recommended by default.
   - Update registry tests to assert capability/recommendation output for local and cloud providers.

2. Apply targeted provider hardening only if the scorecard identifies a concrete issue.
   - For FunASR/SenseVoice, consider safe cleanup of known non-speech/language tags or option settings only when tests prove final text semantics are preserved.
   - For faster-whisper, preserve transcription task behavior, `condition_on_previous_text=False`, selected model quality, and CUDA diagnostics. Do not downgrade from `small`.
   - Add focused provider tests for each hardening change. If no hardening is justified, document `No targeted provider hardening needed` in the summary and skip source changes.

3. Update Settings badges and copy.
   - Refactor `providerCapabilityText` or nearby helpers in `Settings.tsx` so provider rows render concise evidence-backed badges in the existing `Voice input` panel.
   - Centralize new labels in `apps/renderer/src/lib/copy.ts`.
   - Ensure labels distinguish required setup blockers from optional diagnostics and do not reintroduce test-before-enable language.
   - Do not promote cloud quality over local providers in user-facing copy.

4. Update generated contracts if needed.
   - If provider capability literals/schema change, regenerate TypeScript and JSON schema artifacts.
   - Keep generated files synchronized with `packages/contracts/tests/test_codegen.py`.

5. Add renderer regression tests.
   - Cover that Settings shows concise badges for FunASR, faster-whisper, OpenAI, and Groq.
   - Cover cloud providers show `Cloud`/`Requires API key` style setup claims without default/recommended copy.
   - Cover the detailed scorecard is not displayed in Settings.
</tasks>
