# 18-03 Plan: Audio Diagnostics, Provider Tests, And Redaction Regression

<objective>
Expose privacy-safe audio diagnostics in Settings: provider catalog status, selected TTS/STT health, last latency/timeout/failure information, and explicit test actions that never leak credentials, transcripts, reference paths, or raw provider errors.
</objective>

<execution_context>
This plan should extend Phase 16/17 admin audio endpoints if they already exist. If those phases shipped a different endpoint layout, preserve their route names and add only missing Phase 18 diagnostics fields.

Do not add real STT provider implementations. Phase 19 owns local/cloud STT adapters. Phase 18 can display catalog metadata and block cloud tests without consent.
</execution_context>

<tasks>

## 1. Add Or Extend Audio Diagnostics Models

Add sidecar diagnostic structures, or extend upstream equivalents, for:
- provider id and provider kind (`tts` or `stt`);
- capability labels used by Settings;
- health state;
- last health check timestamp;
- last latency in milliseconds;
- timeout state;
- normalized failure category;
- redacted safe error summary;
- redacted reference audio display name or asset id;
- consent/credential blocked status for cloud STT.

Every diagnostics payload returned to Electron must already be redacted.

## 2. Add Admin Audio Endpoints

Extend `sidecar/src/sidecar/admin/audio.py` or upstream equivalent:
- `GET /admin/audio/providers`
- `GET /admin/audio/status`
- `POST /admin/audio/tts/test`
- `POST /admin/audio/stt/test`

Endpoint rules:
- provider list works even when selected providers are unhealthy;
- status distinguishes sidecar availability from selected provider health;
- TTS test can call the upstream test-synthesis path if Phase 17 exposes it;
- STT cloud test returns blocked without consent/credential and never sends audio in that state;
- raw exceptions are normalized and redacted.

Wire the router in `sidecar/src/sidecar/ws/server.py` if not already included.

## 3. Proxy Diagnostics Through Electron

Extend Electron IPC/preload methods from 18-01 if needed:
- `listAudioProviders`
- `getAudioStatus`
- `testTts`
- `testStt`

Electron should avoid logging raw request bodies or response bodies if they may contain secrets. Renderer receives only redacted payloads.

## 4. Add Settings Audio Diagnostics Panel

Add `AudioDiagnosticsSection` or an equivalent sub-section within Voice Output/Input.

Show:
- selected TTS and STT provider health;
- latest latency and timeout values;
- normalized failure reason;
- last checked timestamp;
- whether a test action was blocked by missing consent/credential;
- links/buttons to run TTS/STT tests when supported.

Do not show:
- raw API keys;
- raw transcript text;
- full reference audio paths;
- raw provider exception text.

## 5. Add Regression Tests

Add sidecar tests for:
- provider status responses include latency/failure fields;
- cloud STT blocked state returns no outbound provider call when consent/key missing;
- diagnostics redaction covers keys, paths, transcript-like fields, and exception text.

Add renderer Settings tests for:
- diagnostics render latency/timeout/failure labels;
- blocked cloud STT test shows consent/credential copy;
- redacted diagnostics do not expose secret fixtures;
- provider health is not inferred from sidecar status alone.

</tasks>

</success_criteria>
