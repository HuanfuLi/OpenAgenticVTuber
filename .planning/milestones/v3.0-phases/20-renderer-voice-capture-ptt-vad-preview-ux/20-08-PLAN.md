# 20-08 Plan: VAD Detection And Observability Gap Closure

## Objective

Close the Phase 20 live UAT blocker where VAD did not recognize the user's speech and gave no useful indicator or log to debug what was happening.

## Context

UAT Test 2 failed with: "VAD did not recognize my speaking. Also, I have no idea who is going on with it, so it's hard to debug. No logs available, no any indicators."

This is a live UX and reliability gap in the renderer VAD layer. Phase 20.2 intentionally removed preview transcription, so the VAD UI now needs its own local observability rather than relying on STT preview text as feedback.

## Diagnosis

- `VadController` samples microphone RMS and compares it to fixed thresholds: `low = 0.12`, `medium = 0.08`, `high = 0.045`.
- Chat only shows coarse labels such as `VAD listening`; it does not show current mic level, threshold, speech-detected state, or why recording did not start.
- The controller does not expose level diagnostics to renderer state, tests, or logs.
- Settings has both `sttConfig.input_mode` and renderer-local `voiceSettings.vad.enabled`; Chat consumes only `voiceSettings.vad.enabled`. A user can reasonably believe VAD is selected without the setting that actually starts Chat VAD being active.

## Tasks

1. Add VAD diagnostics to the controller.
   - Add a throttled `onLevel` or equivalent callback from `VadController`.
   - Include RMS level, threshold, sensitivity, speechDetected, monitoring, recordingOwned, and ignored reason when speech is blocked by active playback.
   - Keep diagnostics local to renderer; do not send microphone audio or transcripts to logs.

2. Store and reset VAD diagnostics in renderer state.
   - Extend `voice-input-store.ts` with a compact VAD diagnostics object.
   - Reset diagnostics when VAD stops, readiness changes to not ready, session changes, or VAD is disabled.
   - Keep existing PTT state behavior unchanged.

3. Make Chat VAD observable.
   - When VAD is enabled, show a compact meter/status in `VoiceInputControl`.
   - The UI must distinguish at least: monitoring but silence below threshold, speech detected, recording, finalizing, and paused while Teto is speaking.
   - Show enough numeric or visual signal to tune sensitivity without opening dev tools.
   - Keep copy terse and stateful; do not add tutorial text.

4. Recalibrate VAD thresholds conservatively.
   - Lower the default opt-in threshold enough for normal near-field speech to trigger reliably in browser RMS sampling.
   - Preserve ordered sensitivity semantics: high sensitivity triggers at lower RMS than medium, medium lower than low.
   - Update tests to assert representative quiet/normal speech behavior.

5. Synchronize Settings VAD enablement.
   - Remove or bridge the confusing split between `sttConfig.input_mode = vad` and `voiceSettings.vad.enabled`.
   - If the user selects VAD input mode, ensure Chat VAD is actually enabled on save.
   - If the renderer VAD switch changes, keep the saved STT input mode coherent.
   - Preserve PTT as the default and keep VAD explicit opt-in.

6. Add regression coverage.
   - `vad-controller.test.ts`: below-threshold silence reports diagnostics and does not start recording; speech above threshold reports detection and starts recording; active-turn blocking reports the ignored reason.
   - `ChatVoiceInput.test.tsx`: VAD enabled shows meter/status; speech detected and blocked states are visible; no old preview transcript behavior returns.
   - `Settings.test.tsx`: selecting VAD/input-mode or toggling VAD saves coherent settings consumed by Chat.

## Verification

- `npm --workspace apps/renderer run test -- --run vad-controller ChatVoiceInput Settings`
- `npm --workspace apps/renderer run typecheck`

## Success Criteria

- Retesting UAT Test 2 shows VAD recognizes normal speech or clearly shows when level is below threshold.
- User can tell from Chat whether VAD is monitoring, hearing speech, recording, finalizing, or blocked.
- Settings no longer leaves the user in a contradictory "VAD selected but Chat VAD inactive" state.
- Phase 20 boundaries remain intact: no wake word, no barge-in, no no-headphones/AEC success claim.
