# 19-03 Plan: Cloud STT Providers, Consent, And Redacted Diagnostics

<objective>
Implement OpenAI and Groq STT providers as explicit opt-in cloud adapters behind the shared STT interface, with separate STT credentials, persistent cloud-audio consent, in-memory upload handling, readiness gates, and redacted diagnostics.
</objective>

<execution_context>
This plan depends on the 19-01 contracts, readiness helper, registry, and admin endpoint skeletons. It can run in parallel with 19-02 because it owns cloud provider adapters and redaction behavior, while 19-02 owns local provider adapters and model cache implementation.

Cloud STT must not inherit LLM provider credentials. It must use only the STT provider-specific fields in the persisted audio config established by Phase 18/19 contracts.
</execution_context>

<tasks>

## 1. Add Cloud Provider Dependencies With Lazy Imports

Update `sidecar/pyproject.toml` to include OpenAI and Groq SDK dependencies if they are not already present.

Rules:

- no top-level import of `openai` or `groq` from sidecar startup paths;
- cloud SDK import happens only inside provider health/test/transcribe flow;
- missing package returns unavailable/misconfigured health instead of sidecar boot failure.

## 2. Implement OpenAI STT Adapter

Create `sidecar/src/sidecar/stt/providers/openai_provider.py`.

The adapter should:

- require OpenAI STT API key from audio STT config, not LLM config;
- require provider consent before any network request;
- support configurable model id with a safe default from the provider catalog;
- send test audio as in-memory WAV or file-like payload without writing permanent temp files;
- call the transcription endpoint, not translation, to preserve source language/code-switching;
- normalize success into `STTResult`;
- normalize failures into safe categories such as missing credential, auth failed, rate limited, timeout, unsupported audio, no speech, and internal error;
- redact API keys, request headers, raw provider errors, and transcript-like content from diagnostics.

Do not implement Realtime API or streaming preview in Phase 19.

## 3. Implement Groq STT Adapter

Create `sidecar/src/sidecar/stt/providers/groq_provider.py`.

The adapter should:

- require Groq STT API key and consent;
- default to a provider-catalog model such as `whisper-large-v3-turbo`, with `whisper-large-v3` available as catalog metadata/configurable model id;
- call the OpenAI-compatible audio transcriptions endpoint through the Groq SDK or a minimal HTTP client;
- avoid translation endpoints for final transcript;
- handle provider-specific metadata like duration, latency, and status safely;
- map provider errors to the shared health taxonomy with redacted diagnostics.

Do not add automatic fallback from Groq to OpenAI or local providers.

## 4. Enforce Consent And Credential Gates In Admin Endpoints

Extend STT admin endpoints so cloud providers return blocked state before any network call when:

- provider consent is false;
- provider API key is absent;
- cloud provider is not explicitly selected for the test/enable action.

Add tests that spy on provider/client calls and prove no outbound call is attempted in blocked states.

## 5. Extend Redaction Coverage

Add or extend `sidecar/src/sidecar/audio/redaction.py` if it exists from Phase 18. If no shared helper exists yet, create one following the Phase 18 plan shape.

Redact:

- bearer tokens and API keys;
- request URLs with sensitive query strings;
- multipart filenames or local paths;
- transcript text in diagnostics/history-like fields;
- raw cloud provider exception bodies.

The immediate Settings test response may include the returned transcript for the user's current test. Persisted diagnostics and logs must not retain transcript text.

## 6. Integrate Cloud Providers With Readiness

Wire cloud provider health/test results into the 19-01 readiness helper.

Rules:

- passing health alone is not enough to enable;
- test success requires a non-empty transcript and no provider error;
- changing credential-present state, endpoint, model id, provider id, or language mode invalidates readiness;
- runtime provider failure marks provider unhealthy/unready and disabled until re-tested.

## 7. Add Cloud Provider Tests

Add tests with fake SDK/client classes for:

- OpenAI and Groq SDKs are lazy-imported only during explicit provider actions;
- missing credential blocks before network;
- missing consent blocks before network;
- transcription endpoint is used instead of translation;
- non-empty transcript marks readiness allowed after health;
- empty transcript does not mark readiness allowed;
- raw API keys, transcript text, local paths, and provider exception bodies do not appear in persisted diagnostics/log payloads;
- runtime cloud failure does not fallback and marks provider unready.

</tasks>
