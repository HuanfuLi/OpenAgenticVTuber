# 17-12 Plan: Lipsync Velocity And Sentence Latency

## Goal

GPT-SoVITS chat speech should reach the renderer with low sentence-to-sentence latency, without overlapping browser playback, and should drive calmer VTS mouth motion from the existing RMS/lipsync path.

## Root Cause

Phase 17 already routed GPT-SoVITS through the provider and ordered playback path, but the sidecar websocket delivery was still coupled to local `sounddevice` playback. That meant sentence N+1 could wait for sentence N local playback to finish before reaching the renderer. GPT-SoVITS RMS envelopes were also normalized per sentence, making the existing speech-driver gain/attack too aggressive for this provider.

OLVT's TTS path sends ordered sentence payloads as soon as each payload is ready and treats local playback as a separate stage. Phase 17 needed the same separation while preserving deterministic VTS speech envelope timing.

## Implementation Tasks

1. Split sidecar TTS delivery into separate ordered stages.
   - Keep synthesis ordered.
   - Send websocket audio payloads as soon as their sequence is ready.
   - Keep local sidecar playback serialized separately so local audio writes do not block renderer delivery.

2. Preserve lipsync timing.
   - Emit speech envelopes when local playback begins.
   - Keep the existing RMS payload path intact.
   - Avoid changing provider fallback or failure semantics.

3. Prevent renderer overlap.
   - Queue renderer audio playback so faster sidecar delivery does not start multiple `Audio` instances at once.
   - Keep the chat sentence stream and failed-audio behavior unchanged.

4. Dampen GPT-SoVITS mouth motion.
   - Map mouth movement from smoothed RMS.
   - Lower gain, max-open, attack, and release so normalized GPT-SoVITS RMS does not create overly fast mouth movement.

5. Add regression coverage.
   - Cover websocket delivery no longer waiting on local playback.
   - Cover renderer audio queue ordering.
   - Cover speech-driver damping.
