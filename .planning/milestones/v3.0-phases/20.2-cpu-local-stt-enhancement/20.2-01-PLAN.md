# 20.2-01 Plan: Final-Only Voice Capture And CPU STT Tuning

<objective>
Remove live STT preview transcription from Chat voice input and improve faster-whisper CPU latency through provider reuse, explicit-action warmup, and bounded CPU thread tuning without changing selected model quality.
</objective>

<execution_context>
Phase 20 introduced transient preview transcription, but live CPU STT showed that preview work makes local speech input feel stuck or slow. The user explicitly chose to discard preview completely and keep typo recovery through edit/regenerate instead.

This plan owns only voice capture/STT latency. Stop-current-turn and edit/regenerate are separate plans.
</execution_context>

<tasks>

## 1. Remove Preview From Shared Voice Input Contracts

Update the source Python audio-provider contracts and regenerate generated artifacts.

Required changes:

- remove `previewing` from `VoiceInputCaptureStatus`;
- change `VoiceInputTranscriptionMode` from `preview | final` to `final`;
- update generated TypeScript and JSON schema through contract codegen;
- update contract tests if they assert the old preview variants;
- keep backward compatibility only where old persisted state may exist in renderer-local settings, not in runtime contracts.

Run:

```powershell
npm run codegen:contracts
npm run check:contracts
```

## 2. Simplify Renderer Voice Capture To Final-Only

Update `apps/renderer/src/audio/voice-capture.ts`.

Required behavior:

- remove `DEFAULT_PREVIEW_TIMESLICE_MS` and `previewTimesliceMs`;
- start `MediaRecorder` with `recorder.start()` rather than a preview timeslice;
- keep `ondataavailable` only for accumulating chunks;
- remove `transcribePreview`, `setTransientPreview`, and `applyPreviewResult` imports/usages;
- `stop()` moves capture to `finalizing`;
- `finalizeRecording` remains the only transcription request and sends `mode: "final"`;
- preview failures no longer exist because preview requests no longer run.

Tests:

- update `apps/renderer/tests/voice-capture.test.ts` so recording data events do not trigger transcription until stop/finalization;
- keep final transcription success, final transcription failure, sidecar reconnect cancellation, and queued-final behavior covered.

## 3. Remove Preview State And UI

Update renderer state and Chat controls.

Required changes:

- remove `previewTranscript`, `previewSequenceId`, `setTransientPreview`, `clearTransientPreview`, and `applyPreviewResult` from `voice-input-store`;
- remove preview rendering from `VoiceInputControl`;
- remove `previewing` status copy and status-label branches;
- mic `aria-pressed` should reflect `recording` only;
- Chat should no longer call `clearTransientPreview()` after final candidate submission;
- tests should no longer expect preview text, preview status, or preview cleanup.

Keep:

- `finalizing` status and copy;
- queued final candidate behavior while a turn or TTS speaking is active;
- final transcript submission through the existing `submitFinalText` path.

## 4. Add Bounded faster-whisper CPU Thread Tuning

Update `sidecar/src/sidecar/stt/providers/faster_whisper_provider.py`.

Required behavior:

- keep `model_path`, `device`, and `compute_type` selection semantics;
- when `device == "cpu"`, pass a bounded `cpu_threads` argument to `WhisperModel`;
- choose a conservative default based on available logical CPU count, capped at 8 and never below 1;
- do not pass `cpu_threads` on CUDA runtime unless faster-whisper documentation requires it for that path;
- include `cpu_threads` in redacted diagnostics for CPU health/load/transcription results;
- do not change the selected model id or model path.

Tests:

- add or update provider construction tests/mocks to assert CPU passes the bounded thread count;
- assert CUDA construction does not fail because of CPU-thread-specific logic;
- preserve existing missing CUDA DLL diagnostics.

## 5. Warm Local STT After Explicit Enable/Readiness Actions

Update `sidecar/src/sidecar/admin/audio.py`.

Required behavior:

- keep app startup free of heavy local STT model loads;
- when a local STT provider is explicitly enabled or readiness is explicitly checked with complete required setup, schedule a best-effort background warmup using the cached provider path;
- warmup must not block the event loop or freeze Settings;
- warmup must reuse `_build_or_get_stt_provider`;
- warmup failures should be logged or returned as diagnostics only when the triggering endpoint already reports them; do not silently switch provider;
- do not evict a successfully warmed provider before the next voice-input request.

Implementation hint:

- add a small helper such as `_schedule_local_stt_warmup(config)` that starts a daemon thread or background task guarded by provider cache key;
- reuse `_validate_stt_runtime` under the existing provider transcribe lock so concurrent final transcription is serialized safely.

Tests:

- local enable/readiness action schedules or performs warmup with a downloaded model;
- missing model does not attempt warmup;
- warmup failure does not flip enabled config or fall back to a different STT provider.

</tasks>
