# 20.2-05 Plan: Stopped Message Edit/Regenerate Gap Closure

## Objective

Fix the Phase 20.2 UAT blocker where editing a stopped streaming user message can get stuck if later chat activity has happened, and where the stopped message drifts to the bottom of chat history. Normal persisted-message editing already passes and must not regress.

## Context

UAT Test 6 found that Save does nothing for a stopped message after later chat activity: edit mode stays open and no fresh request is dispatched. The same stopped message is also not fixed in its original chat position and can be reset to the bottom/newest position after later conversation.

Test 7 passed, which narrows the defect to stopped/interrupted streaming messages rather than the general persisted-message edit/regenerate path.

## Diagnosis

- `useStreamingMessages.ts` keeps stopped-turn restart metadata on the singleton `pendingTurn`.
- `restartStoppedTurn()` only succeeds when `state.pendingTurn?.stopped` matches the edited message.
- A later normal turn overwrites `pendingTurn`, orphaning the earlier stopped message's restart metadata.
- `Chat.tsx` merges persisted session messages first and transient streaming messages after them, so a transient stopped message visually drifts after later persisted history.
- Existing tests covered immediate stopped-message edit, but not delayed edit after later history changes.

## Tasks

1. Add stable stopped/interrupted user-message state.
   - Track stopped user messages by message id, independent of the current active `pendingTurn`.
   - Add or adapt restart logic so Save can restart a stopped message by id.
   - Keep `pendingTurn` scoped to the current active turn; later turns must not erase editability of earlier stopped messages.
   - Do not commit stopped interrupted turns as completed history until regenerated normally.

2. Keep stopped messages anchored.
   - Give transient stopped messages stable ordering metadata such as `createdAt` or an equivalent sequence.
   - Change Chat merge/order logic so transient stopped messages are ordered with persisted messages instead of appended after all persisted history.
   - Ensure regenerate does not create duplicate user bubbles.

3. Update the Save edit path.
   - Pass the edited message id into stopped-message restart logic.
   - On successful restart, dispatch the edited text through the normal `text-input` request path and exit edit mode.
   - Keep the UI recoverable if stopped metadata is missing; Cancel must still work and the failure should be visible in logs or state.

4. Add regression coverage.
   - Preserve the existing immediate stopped-message edit test.
   - Add a test where a stopped message is followed by a later normal turn/history update, then edited and saved.
   - Assert the stopped message stays in its original relative position.
   - Assert Save dispatches `text-input`, exits edit mode, and does not duplicate the user bubble.
   - Confirm normal persisted-message edit/regenerate still passes.

## Verification

- `npm --workspace apps/renderer run typecheck`

## Success Criteria

- UAT Test 6 passes: stopped voice transcript edits save and dispatch even after later chat activity.
- The stopped message remains visually anchored in chat history.
- UAT Test 7 remains passing for normal persisted-message edit/regenerate.
- No grace-window undo, delayed dispatch, or live STT preview behavior is introduced.
