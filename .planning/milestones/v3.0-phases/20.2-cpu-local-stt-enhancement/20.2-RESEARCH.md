# Phase 20.2 Research: Final-Only STT, Stop, Edit/Regenerate

## Summary

The current latency issue is not primarily a missing model cache problem. Local STT provider instances are cached in `sidecar/src/sidecar/admin/audio.py`, and `FasterWhisperSTTProvider.ensure_loaded()` is a no-op after the first successful load. The bigger Phase 20.2 wins are:

1. Remove live preview transcription so local CPU STT handles one final request per utterance.
2. Add bounded CPU thread tuning to faster-whisper without changing selected model quality.
3. Warm local STT after an explicit user enable/readiness action, not at app boot.
4. Add stop-current-turn across sidecar stream, sidecar TTS queues, renderer audio playback, and streaming chat state.
5. Add edit/regenerate by truncating persisted conversation history before the edited user message and dispatching the edited text through the normal `text-input` path.

## STT Latency Findings

- Current local STT cache:
  - `_STT_PROVIDER_CACHE` stores built local providers keyed by provider/model/path/runtime/compute type.
  - `_STT_PROVIDER_TRANSCRIBE_LOCKS` serializes transcription per provider key.
  - `_validate_stt_runtime()` calls provider `validate_runtime()`, which calls `ensure_loaded()`.
  - `_transcribe_with_provider()` reuses the cached provider for actual transcription.
- faster-whisper provider:
  - CPU uses `compute_type = "int8"`.
  - CUDA uses configured `cuda_compute_type`.
  - The provider validates missing CUDA runtime DLLs before constructing the model.
  - CPU does not currently pass `cpu_threads`.
- User-observed and previous benchmark notes:
  - Fresh model loading was previously suspected, but current code has provider reuse.
  - Removing preview requests should reduce local STT work immediately.
  - `cpu_threads=8` improved a sample transcription from roughly 3.4-3.6s to roughly 2.86-2.94s in prior diagnosis.
  - `cpu_threads=2` was worse; `cpu_threads=4` was near default.
  - Do not downgrade model size to improve latency.

## Preview Removal Impact

Current `VoiceCapture` starts MediaRecorder with a 1200 ms timeslice and transcribes previews from accumulated chunks while recording. Even though the current code uses accumulated chunks rather than isolated chunks, the effect is still expensive: for a long utterance, the app may transcribe overlapping prefixes multiple times before final transcription.

Final-only capture can simplify the state machine:

- `recorder.start()` with no timeslice.
- `ondataavailable` only accumulates final chunks.
- `stop()` moves directly to `finalizing`.
- no `previewing` capture status in active UI.
- no preview transcript display.
- no preview result application path.

Keeping dead preview contracts would leave stale state and tests. Since this app owns both sides of the contract, Phase 20.2 should remove preview mode/status from source contracts and regenerate TypeScript/schema artifacts.

## Stop-Current-Turn Design

Stop needs to interrupt both visible output and background work:

- Renderer:
  - send `type: "stop-turn"` or equivalent explicit control envelope;
  - stop renderer audio playback by pausing current audio, clearing queued base64 payloads, and revoking object URLs;
  - remove partial assistant streaming bubbles;
  - keep the triggering user message visible and editable;
  - re-enable input after the stop state settles.
- Sidecar:
  - cancel the active LLM/TTS turn task;
  - call `TTSTaskManager.clear()` to cancel synthesis/sender/playback queues;
  - clear queued pending inputs so no stale queued text starts after the stop;
  - roll back uncommitted sidecar memory for the canceled user/assistant turn or force exact resync from next renderer history.
- Protocol:
  - add a typed stop message to contracts instead of overloading the generic `control` string namespace.
  - renderer should treat chain-end after a local stop as idempotent.

The current `_turn_loop` awaits `turn(...)` directly, so it has no active-turn task to cancel from another handler. The least invasive path is to wrap each turn in `self._active_turn_task`, expose `cancel_active_turn()`, and have the stop handler call it.

## Edit/Regenerate Design

Completed turns are persisted only when `conversation-chain-end` arrives and the dispatcher commits a completed user+assistant pair. This means edit/regenerate should not mutate a completed user message and then call the existing `commitTurn`, because that would duplicate the user message.

Preferred flow:

1. User edits a user bubble when no turn is active.
2. If the bubble is persisted, Electron truncates the active session before the selected user message, removing that user and all later messages.
3. Renderer dispatches the edited text as a normal new `text-input` using the truncated session messages as history.
4. Streaming reducer appends the edited user bubble and assistant response.
5. Existing `commitConversationTurn` persists the edited user+assistant pair at completion.

For an interrupted in-flight user message, no persisted truncation is needed. The streaming reducer can replace that pending user text and dispatch normally with current persisted history.

This preserves the existing `commitTurn` model and avoids a new "assistant-only commit" mode.

## Test Strategy

- Renderer unit tests:
  - voice capture performs no preview transcription requests;
  - final transcription still submits or queues through existing path;
  - Stop button replaces Send during active streaming/speaking;
  - stop clears audio queue and partial assistant output but keeps the user message;
  - editing a persisted user message calls truncation before dispatch;
  - editing an interrupted streaming user message dispatches without persisted truncation.
- Electron tests:
  - conversation truncation removes selected user and all later messages;
  - truncation preserves session title semantics and summary counts;
  - invalid session/message ids fail clearly.
- Sidecar tests:
  - stop message cancels active turn task and clears pending inputs/TTS manager;
  - cancellation does not append partial assistant output to memory;
  - next edited request resyncs exact renderer history after truncation.
- Contract tests:
  - generated TypeScript and JSON schema stay in sync with Python source contracts.
