# 20.2-03 Plan: Edit Sent Message And Regenerate From There

<objective>
Let users edit a sent user message when no turn is active, truncate later conversation messages, and dispatch the edited text as a normal fresh request so STT typos can be manually repaired.
</objective>

<execution_context>
The user explicitly rejected grace-window undo. Recovery should happen by editing the sent message and regenerating from that point, like LM Studio. This must work for both completed persisted messages and the interrupted in-flight user message left behind by Stop.
</execution_context>

<tasks>

## 1. Add Conversation Truncation In Electron Store

Update `apps/electron-main/src/conversation-store.ts`.

Preferred API:

- add a function such as `truncateConversationBeforeMessage(sessionId, messageId)`;
- require the target message to exist and be a user message;
- remove the selected user message and every later message from that session;
- preserve earlier messages in order;
- update `updatedAt`, `lastMessageAt`, summary counts, and preview consistently;
- preserve a manually renamed title;
- if truncating the first message of an auto-titled session, keep the title as-is unless existing title rules require reset.

Why truncate before the selected user:

- the edited text will be appended as the next streaming user message;
- existing `commitConversationTurn` can persist the edited user+assistant pair without duplicating the user.

Tests:

- truncating a middle user removes that user, its assistant, and all later messages;
- truncating the latest user removes that user and its assistant;
- truncating a non-user message fails clearly;
- invalid ids fail clearly.

## 2. Expose Truncation Through IPC, Preload, And Renderer State

Update Electron IPC/preload and `apps/renderer/src/state/conversation-history.tsx`.

Required behavior:

- add a typed preload method such as `truncateConversationBeforeMessage(sessionId, messageId)`;
- expose a provider method that calls IPC and refreshes summaries/stats;
- keep fallback behavior safe in test environments where `window.api` is mocked;
- update preload `.d.ts`.

Tests:

- IPC invokes the store method;
- renderer provider refreshes active session and summary state after truncation.

## 3. Add Editable User Bubble UI

Update `apps/renderer/src/screens/Chat/Chat.tsx`.

Required behavior:

- show an edit affordance on user bubbles only when no turn is active and WebSocket is available;
- support persisted user messages from `activeSession.messages`;
- support the interrupted in-flight user message kept by plan 20.2-02;
- edit mode should use a compact textarea/input inside the user bubble;
- Save dispatches; Cancel restores without side effects;
- empty edits cannot be saved;
- editing is disabled while `turnInFlight || isSpeaking || banners.llm || !wsOpen`;
- no in-app explanatory text describing the feature beyond accessible labels/tooltips.

Implementation hint:

- while mapping `merged` messages, tag each message source as persisted or streaming;
- for persisted messages, Save first calls conversation truncation, then submits the edited text with the returned truncated history;
- for interrupted streaming messages, Save updates/reuses the pending user message and dispatches with current persisted history.

## 4. Dispatch Edited Text Through Normal Path

Refactor `submitFinalText` in `Chat.tsx` so it can accept an optional history override and optional existing/pending user handling.

Required behavior:

- typed input Send still appends user and sends with current `activeSession.messages`;
- final STT candidate still uses the same path;
- persisted edit/regenerate sends with truncated persisted history;
- interrupted-user edit sends with current persisted history and replaces the pending user text instead of adding a duplicate;
- all dispatches still use `type: "text-input"` and the selected `activeSession.id`;
- no delayed send or grace-window behavior.

Tests:

- editing a persisted user sends history before that user only;
- editing a stopped streaming user does not duplicate user bubbles;
- final STT transcript can be edited after Stop and dispatched normally;
- saving an edit while a turn is active is blocked.

## 5. Resync Sidecar Memory Exactly From Edited History

Update `sidecar/src/sidecar/orchestrator/orchestrator.py`.

Problem:

- `_sync_memory_from_history` currently replaces memory only when session id changes or incoming history grows.
- Edit/regenerate truncates history, so incoming history can shrink and still be authoritative.

Required behavior:

- when a text-input message includes `session_id` and `history`, treat the renderer history as authoritative at turn boundary;
- replace sidecar `_memory` when the normalized renderer history differs from the current memory suffix/prefix, including shrink/truncation cases;
- reset `_head_idx` safely on exact history replacement;
- keep forward-only pruning during normal ongoing turns after resync;
- avoid retaining canceled/edited prompt text in sidecar memory.

Tests:

- after a completed multi-turn memory, a truncated edit request replaces memory with shorter history before appending edited text;
- canceled prompt text is not retained after edited regeneration.

</tasks>
