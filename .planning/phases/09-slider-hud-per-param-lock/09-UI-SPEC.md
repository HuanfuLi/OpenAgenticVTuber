# Phase 9 — UI Design Contract

> Visual and interaction contract for Phase 9 (Slider HUD + Per-Param Lock). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope of this contract:** the **two new surfaces** Phase 9 adds — (1) a single "Open HUD" button inside the existing `Settings` screen's "Body motion plugin" section, and (2) a **separate Electron BrowserWindow** loading the same renderer bundle on a `/hud` route hash, hosting a scrollable param-row list with sliders and per-param lock toggles. Plus one new toast string (locks-cleared notice on avatar re-import) and one new lock/unlock icon pair added to the project's hand-rolled SVG icon set.
>
> **Net new design decisions in this contract:** the HUD route's overall layout (header + filter chips + param-row list + footer status), one new compound row component (`HudParamRow`), one new locked-state visual treatment (slider-fill color shift + lock-toggle icon flip), one new diagnostic-window framing convention (this is the project's first true "side-by-side with VTS" surface; visual conventions established here will guide any future floating diagnostic windows), and 8 new copy strings.
>
> **What does NOT change from Phase 1:** every visual token. The 7 named theme classes (blush / sunrise / ember / midnight-sky / midnight-pewter / onyx-sky / onyx-pewter), the spacing scale, the typography sizes/weights, the radii (16px cards / 12px buttons / pills), color tokens, error/empty-state shape conventions, the `ToastStack` component, the entire `index.css` token surface. Phase 9 EXTENDS, NEVER REPLACES.

### HUD Window Sizing (NEW)

- **Default size: 420 × 640 px** (slightly wider than the main shell's 400×700 to give param IDs and sliders horizontal room; shorter because no chrome rail consumes vertical space)
- **Minimum size: 360 × 480 px** (slider remains usable at this width; below this, hide filter chips into a single dropdown — but in v1 just enforce the floor)
- **Position: OS-default placement** (do not call `BrowserWindow.center()` — let the OS pick; user moves it where they want)
- **Persistence: none** — every open creates a fresh window at default size and OS-default placement (per D-A4)
- **Topmost: NO** — focused-only top-most is Electron's BrowserWindow default; do NOT call `setAlwaysOnTop`. When the user clicks back to VTS, the HUD recedes naturally; when the user wants to read it, they click on it (per D-A3)
- **Resizable: yes** — user may resize freely within the min-size floor; the resize is not persisted but works during the session
- **Frame: standard OS chrome** — the HUD uses the OS-supplied title bar and close button. The window title is `"AgenticLLMVTuber — HUD"` so it's distinguishable from the main window in OS task switchers

### Initialization Note

Phase 9 has no shadcn install step. The new lock/unlock icons are added to `apps/renderer/src/lib/icons.tsx` following the existing `makeIcon(paths)` pattern. New CSS rules append to `apps/renderer/src/index.css` under a new `/* HUD route */` section header. The `/hud` route is added to the `view` discriminated-union in `apps/renderer/src/state/app-store.ts` (or its `setView` equivalent) — but unlike the chat/settings/avatar-import views which all live under `AppShell`, the `/hud` route mounts WITHOUT `AppShell` (no TopBar, no BottomRail, no LogsDrawer) because the HUD window is a separate BrowserWindow with its own purpose. Plan-phase researcher decides whether to detect "is HUD window" via `window.location.hash === '#/hud'` at app boot and short-circuit `AppShell` mounting, or via a new top-level router branch in `App.tsx`. Either path is acceptable; the visual contract is the same: HUD window renders **only** the HUD root component.

## Spacing Scale

**Inherited verbatim from `01-UI-SPEC.md`.** Phase 9 introduces no new spacing tokens.

| Token | Value | Phase 9 usage |
|-------|-------|---------------|
| `xs` | 4px | Lock-toggle icon-to-edge gap; filter-chip inner horizontal padding (10px = sm + xs blend not used — chip uses `0 10px` per existing `.anchor-pill`) |
| `sm` | 8px | Param row vertical padding; row gap between value-display and slider; filter-chip row gap |
| `md` | 16px | HUD outer padding (left/right); vertical gap between HUD header and the param-row list; gap between filter-chip strip and list start |
| `lg` | 24px | (unused by Phase 9 — HUD is denser than Settings) |
| `xl` | 32px | Empty-state vertical breathing room (when filtered list returns zero rows) |
| `2xl` | 48px | (unused by Phase 9) |

**HUD-specific layout sizes (justified structural exceptions, not new scale tokens):**

- **HUD header height: 48px** — title (`"Slider HUD"` heading) + connection-status dot + filter-chip strip (right-side) all fit on one row at 48px tall (16px text + 16px chip + 8px top + 8px bottom = 48px). Justification: separates the "static configuration" area from the scrollable param list with a clear horizontal rule, mirrors the pattern in `Settings` h1 → anchor-pills → first section
- **Param row height: 44px (touch-target floor)** — WCAG 2.5.5 minimum is 44×44px for any tappable element. The lock toggle is 28×28px button hosted inside a 44px-tall row, satisfying the floor with the row itself acting as the hit area. Justification: 60Hz `param-frame` updates animate the slider thumb continuously, so visual stability of row height matters; 44px is comfortable for a slider that the user expects to drag
- **Slider track height: 4px** — thin native-feeling track; thumb is 16px diameter. Same numbers as the existing OS-native `<input type="range">` defaults the project does not currently override. Justification: HUD is diagnostic, not decorative — the thumb is the affordance, the track is just the line behind it
- **Lock toggle button: 28×28px** — same size as the existing `.icon-btn` minus 4px (we use 28 here so the row stays 44px tall with 8px vertical padding). Border radius: `rounded-xl` (12px) per button-radii rule. The 28px button inside a 44px-tall row gives 8px top/bottom breathing room and stays within the WCAG 44px hit area via the row's overall hit-tolerance
- **Footer status strip: 24px height** — single line of muted-foreground 12px copy (e.g. `"22 params · 3 locked · 15 Hz"`). Sits at the bottom of the HUD window above the OS chrome. Justification: gives the user a constant readout of what the HUD is doing without consuming vertical scroll space

## Typography

**Inherited verbatim from `01-UI-SPEC.md`.** Phase 9 adds no new sizes or weights.

| Role | Size | Weight | Line Height | Phase 9 usage |
|------|------|--------|-------------|---------------|
| Body | 14px | 400 regular | 1.5 | Param row's display name (`ParamAngleX` or `Angle X` if cdi3 present), filter chip label, empty-state body copy |
| Small / Meta | 12px | 400 regular | 1.4 | Param row's value display (e.g. `0.42`), param row's secondary param-id when display name comes from cdi3 (subtitle treatment), footer status strip, empty-state heading subtitle |
| Heading | 18px | 600 semibold | 1.3 | HUD header title `"Slider HUD"` (only one heading in this phase) |
| Mono | 12px | 400 regular | 1.5 | Param row value display when in mono context (`mono` class — same convention as the existing `[INTENT]` log-line render and `01-UI-SPEC` mono rule) |

**Param-name display rule (locked decision):**

When `RigCapabilities.cdi3_display_names[param_id]` returns a value, the row renders **two-line stacked**:
- **Line 1 — Body 14px regular foreground:** the cdi3 display name (e.g. `"Angle X"`, `"Eye Smile L"`)
- **Line 2 — Small 12px regular muted-foreground mono:** the raw param ID (e.g. `ParamAngleX`)

When `cdi3_display_names[param_id]` is absent (param is in the `RigCapabilities.writable_param_ids` set but the rig has no `.cdi3.json` mapping for it), the row renders **single-line**:
- **Single line — Body 14px regular foreground mono:** the raw param ID (e.g. `ParamBHandIN`)

Justification: the cdi3 display name is the human-friendly form when the rigger provided it; falling back to the param ID in mono signals "this is a raw machine identifier, not a label". The stacked treatment is identical density to the Settings `.kv-row` pattern (label on top, raw value below in muted) so it inherits muscle memory.

**Slider thumb / track typography:** none — sliders carry no text (the value is displayed separately in the row).

**Footer status copy:** 12px muted regular. Format: `"{N} params · {M} locked · 15 Hz"` where N = visible (post-filter) param count, M = currently-locked count. Live-updates as filters and locks change.

**Connection status dot (HUD header right side):**
- WS connected: 10px circle in `--success` (green dot, same hex as the existing `.dot.green` in `index.css:452`)
- WS reconnecting: 10px circle in `--warning` (amber dot, same hex as `.dot.amber`)
- WS disconnected (sidecar dropped): 10px circle in `--destructive` (red dot, same hex as `.dot.red`)

These dots are the same three states as the existing top-bar status hexagon's underlying logic — direct CSS reuse via the `.dot.green` / `.dot.amber` / `.dot.red` classes.

## Color

**Inherited verbatim from `01-UI-SPEC.md`** — all 7 theme classes, all 60/30/10 assignments, all status-color tokens, the saturation cap (chroma ≤ 0.13 for accents, ≤ 0.04 for surfaces). Phase 9 adds **zero new color tokens**.

### HUD surface assignments (60 / 30 / 10 mapping)

| Surface | Token | Phase 9 behavior |
|---------|-------|------------------|
| HUD window background (60%) | `--background` | The full HUD window paints the dominant background — same surface as the chat scroll area in the main shell. Justification: the HUD is a focused diagnostic surface; secondary tokens are reserved for the rows themselves so they "lift" off the background |
| Param row surface (30%) | `--card` | Each row is a `card`-tier surface against the dominant background. Border `1px` of `color-mix(in oklab, var(--border), transparent 40%)` per the existing card convention in `index.css:273-278` |
| Filter chip background (30%, inactive) | `--card` | Inactive filter chip uses the same `.anchor-pill` styling as Settings (already in `index.css:537-553`) — direct CSS class reuse |
| Filter chip background (10% accent, ACTIVE) | `--primary` at 15% mix | Active filter chip uses the same `.anchor-pill.active` styling as Settings — `background: color-mix(in oklab, var(--primary), transparent 85%)`, `border-color: color-mix(in oklab, var(--primary), transparent 60%)`. **This is one of the 6 accent-reserved-for affordances (Settings-anchor-pill rule extends to HUD filter chips since they are the same primitive)** |
| Slider track (unlocked state) | `color-mix(in oklab, var(--foreground), transparent 80%)` | Mid-grey neutral; same alpha as the existing `.switch` background-off state |
| Slider fill (left of thumb, unlocked state) | `color-mix(in oklab, var(--foreground), transparent 60%)` | Slightly darker neutral fill to show progress |
| Slider thumb (unlocked state) | `--foreground` | Solid foreground color; high contrast against track |
| Slider track (LOCKED state) | `color-mix(in oklab, var(--primary), transparent 80%)` | Tinted with active accent — communicates "this row is in special mode" without using accent at full saturation |
| Slider fill (left of thumb, LOCKED state) | `--primary` | **This is one of the 6 accent-reserved-for affordances** (extension of "active toggle / lock state communication") |
| Slider thumb (LOCKED state) | `--primary` | Same color as the fill — visually unifies the locked slider into a single accent stripe |
| Lock toggle background (UNLOCKED state) | `transparent` | Plain icon-button, hover background `color-mix(in oklab, var(--foreground), transparent 92%)` per the existing `.icon-btn:hover` rule |
| Lock toggle background (LOCKED state) | `--primary` at 15% mix | `background: color-mix(in oklab, var(--primary), transparent 85%)` — subtle accent fill matching the active filter chip |
| Lock toggle icon color (UNLOCKED) | `--muted-foreground` | De-emphasized when there's no lock |
| Lock toggle icon color (LOCKED) | `--primary` | Communicates the active state |
| Footer status strip background | `--background` | Same as window background — no surface lift |
| Footer status strip text | `--muted-foreground` | 12px regular muted, identical to the existing `.tx-sm.muted` pattern |

### Accent Reserved For (Phase 9 extensions)

The Phase 1 list of 6 accent affordances is extended with **two new HUD-specific affordances**, both directly equivalent to existing rules (so they are not "new uses" but specializations of the existing affordance categories):

7. **HUD filter chip — active state** — direct equivalent of "Settings anchor pill (active section)" (rule #6). Same primitive, different host.
8. **HUD param row — locked state** — direct equivalent of "`Agent ⏵` toggle ON state" (rule #3). The "locked" mode is conceptually identical to "active toggle" — the user has switched a control into a non-default mode and the accent communicates it. Slider fill, slider thumb, and lock-toggle button background+icon all flip to accent variants when locked.

**Forbidden uses of accent in HUD:**
- Slider track / thumb in unlocked state (use neutral foreground tones instead)
- Inactive filter chip
- HUD heading text
- Param ID labels
- Footer status strip
- Connection-status dot (uses the independent status-green/amber/red palette per Phase 1's "status colors are independent of accent" rule)

This keeps the "10% accent" budget intact: at any given moment the HUD shows accent only on (a) the active filter chips (typically 3 chips since all are on by default), and (b) the locked rows (typically 0–3 rows during normal use). A HUD with no locks and no filter changes is mostly neutral chrome.

### Status colors (independent of theme accent)

Inherited verbatim from Phase 1 — `--success` / `--warning` / `--destructive`. Phase 9 uses `--destructive` for **only one** in-HUD surface: the WS-disconnected connection dot. No other red/amber/green elements; specifically, no error toast color override, no destructive button (the HUD has no destructive actions — see Copywriting Contract).

### Theme harmony

**The HUD window must paint with the same active theme class as the main window.** Implementation: at HUD-window-mount, the renderer reads the active theme from localStorage (the existing `theme-provider` already persists there via `useTheme().prefs`); if localStorage read fails (sandboxing edge case in a separate BrowserWindow), fall back to `theme-midnight-sky` — the project default. The HUD does NOT add a theme picker of its own; users change themes from the main window's Settings, and the HUD's next paint reflects the change (no live-resync on theme change is required for v1 — closing and re-opening the HUD is acceptable, given the diagnostic-tool framing).

## Component Inventory

### New components (Phase 9)

#### `<HudRoot>`
- **What:** HUD window's root component, mounted ONLY when `window.location.hash === '#/hud'` (route-detection happens at app boot in a top-level branch outside `AppShell`)
- **Layout:** vertical flexbox; header (48px fixed) → filter chip strip (auto-height ~32px) → scrollable param list (flex 1, overflow-y auto) → footer status strip (24px fixed)
- **Background:** `--background`, fills viewport
- **Mount lifecycle:** opens `/hud/ws` on mount, closes on unmount; fetches `GET /admin/rig-capabilities` on mount; sets up the `useHudStream` hook (plan-phase decides hook signature)
- **Hosts:** `<ToastStack>` (the existing component reused verbatim — toasts inside the HUD window come from the HUD window's own `useStore`)

#### `<HudHeader>`
- **What:** Top 48px strip with title + connection dot
- **Left:** `<h1>` with copy `"Slider HUD"`, 18px semibold foreground per Heading typography rule
- **Right:** 10px circle dot in green/amber/red per WS connection state (uses existing `.dot.green` / `.dot.amber` / `.dot.red` classes verbatim)
- **Border:** 1px bottom border in `color-mix(in oklab, var(--border), transparent 40%)` (same as `.top-bar` border in `index.css:129`)
- **No close button** — the OS title-bar's close button is the closer; in-app close button would be redundant chrome

#### `<HudFilterChips>`
- **What:** Three-chip filter strip below header
- **Chips (in order):**
  1. `Writable` — toggles inclusion of params marked writable in `RigCapabilities` (this is effectively all rows since the list is sourced from `writable_param_ids`; the chip is a "show all writable" toggle that can be turned OFF to compose with the other two — turning it off shows ONLY rows that pass the other two filters' inclusion criteria intersected with whatever the user wants. v1 shipping default: chip ON, behaves as identity filter; can be toggled off but rarely useful — included for symmetry and future-proofing if param categories grow)
  2. `Animating` — when ON, restricts list to params whose recent value (last ~30 frames = ~2 seconds) has non-trivially changed. Useful for spotting which params an LLM-driven `{variant}` is actively writing. Threshold: any value movement > 0.01 within the 30-frame window
  3. `Locked` — when ON, restricts list to currently-locked rows only
- **All-on by default** (no filtering at first open — show every row)
- **Combine logic:** chips combine via INTERSECT (a row must satisfy every active chip's inclusion rule to render). Active chip = filter-active = restricts the list. Inactive chip = filter-inert = does not restrict
- **Visual:** reuse existing `.anchor-pill` and `.anchor-pill.active` classes verbatim — same primitive as the Settings page's section-anchor pills
- **Layout:** horizontal flexbox, 6px gap (same as Settings), aligned to the right side of a 32px-tall strip; left side is empty (in v1) — reserves left half for future search input if param counts balloon
- **Padding:** 4px top/bottom, 16px left/right (matches `md` outer padding rule)

#### `<HudParamRow>`
- **What:** A single row in the param list — display name + value + slider + lock toggle
- **Height:** 44px (touch-target floor)
- **Padding:** 8px top/bottom, 16px left/right
- **Background:** `--card` (the 30% surface)
- **Border:** 1px bottom border (same `color-mix` as header bottom border) — visually separates rows; no borders on left/right/top (rows stack)
- **Layout (left-to-right):**
  1. **Param name column** (flex 1, min-width 120px): two-line stacked when cdi3 display name is present, single-line mono when absent (per Typography rule)
  2. **Value display column** (96px fixed): mono 12px regular muted-foreground, right-aligned within the column, format `0.00` to `1.00` with 2 decimals (or the param's actual range with proper bounds, e.g. `-30.00` to `30.00` for `ParamAngleX` — bounds come from `RigCapabilities.ranges[param_id]`)
  3. **Slider column** (flex 1, min-width 160px): `<input type="range">` styled per Slider rules below
  4. **Lock toggle column** (28px fixed): `<button>` with `<Lock>` or `<Unlock>` icon
- **Hover:** no surface change (the row is already on `--card`); cursor stays default — hover affordance is on the slider and lock-toggle interactive children, not the row itself
- **Disabled state:** none — every row in the list is interactive (the design eliminates "disabled rows" via the HUD-exclusion rule; a param the user cannot lock is not present at all)
- **Locked variant:** Slider fill + thumb + lock-toggle background+icon all flip to accent per the Color section's "LOCKED state" rules. **No text change**, no ribbon, no border-color change — the slider+toggle are the only locked-state signals (visual budget kept tight)

#### `<HudSlider>` (renders inside `<HudParamRow>`)
- **What:** A native `<input type="range">` with custom CSS targeting `::-webkit-slider-runnable-track`, `::-webkit-slider-thumb`, and `::-moz-range-track` / `::-moz-range-thumb`
- **Min / max:** `RigCapabilities.ranges[param_id]` — typically `[0, 1]` for blendshape params, can be `[-30, 30]` or other for rotation params. Plan-phase verifies that the rendered range bounds match what the sidecar will accept (no slider should be able to express a value the sidecar would reject)
- **Step:** 0.01 (fine-grained for blendshape range; for wider ranges like `[-30, 30]`, plan-phase may step at 0.1 — accept the recommendation but the slider does NOT need to feel pixel-perfect; the LLM and the user are the two consumers)
- **Visual (unlocked):** 4px-tall track, 16px-diameter circle thumb, neutral foreground tones per Color section
- **Visual (locked):** 4px-tall track tinted with accent, fill left of thumb at full accent, thumb at full accent
- **Drag interaction:** on first non-trivial pointer movement (`movementX > 1px` cumulatively, or change of >2 step units from the initial value at pointerdown), fire `set-lock(param_id, value)` over `/hud/ws`. Subsequent drag positions also fire `set-lock(param_id, current_value)` — the sidecar accepts the most recent value. **Releasing pointer (`pointerup`) does NOT fire `clear-lock`** — the lock persists until the user clicks the row's lock toggle off (per D-D2)
- **Throttle:** during drag, throttle outbound WS sends to ~30Hz (every ~33ms) to avoid flooding the WS channel; final `pointerup` always sends the last value to ensure the resting position is exact
- **Optimistic UI:** the renderer's slider thumb tracks the user's pointer immediately; the sidecar-confirmed value (next `param-frame` snapshot) is the source of truth, but during drag the optimistic local value wins so the slider feels instant. After ~200ms of no drag activity, snap-correct to the latest sidecar value if there's drift (rare — sidecar accepts every send)

#### `<HudLockToggle>` (renders inside `<HudParamRow>`)
- **What:** 28×28px button hosting a `<Lock>` or `<Unlock>` icon
- **Icon (UNLOCKED):** `<Unlock>` — open padlock SVG (see new Icons section below)
- **Icon (LOCKED):** `<Lock>` — closed padlock SVG
- **Click behavior:**
  - When unlocked: clicking does nothing visible (lock auto-engages on slider drag, not on toggle click — this prevents accidental "lock at 0.0" mishaps where user clicks toggle without first picking a value)
  - When locked: clicking fires `clear-lock(param_id)` → sidecar removes param from `lock_state` → next `param-frame` returns `locked_ids` without this param → renderer flips toggle visual back to unlocked. **This is the only way to release a lock** other than avatar re-import or app restart
- **ARIA:** `role="switch"`, `aria-checked={isLocked}`, `aria-label={`${isLocked ? 'Unlock' : 'Lock'} ${displayName}`}`. Keyboard-accessible — Tab focuses, Enter/Space activates same as click
- **Focus ring:** 2px outline in `var(--primary)`, offset 2px (same as the existing `.icon-btn:focus-visible` rule in `index.css:147`)

#### `<HudFooterStatus>`
- **What:** Bottom 24px strip with summary
- **Copy template:** `"{N} params · {M} locked · 15 Hz"` (e.g. `"22 params · 3 locked · 15 Hz"`)
- **Style:** 12px muted regular foreground, single line, padded 0 16px left/right, vertically centered in 24px strip
- **Background:** `--background` (no surface lift)
- **Border:** 1px top border (same `color-mix` as header bottom border)

- **Behavior:** if the fetch takes > 5 seconds, swap to error state with copy `"Couldn't load rig parameters. Is the sidecar running?"` in `--destructive` color, plus a `Retry` button (uses existing `.btn.btn-secondary` class, copy `"Retry"`). Retry re-fires the fetch.

#### `<HudErrorState>` — `/hud/ws` connection lost mid-session
- **What:** A persistent inline banner at the top of the scroll area (NOT a toast — the WS error is a continuing condition, not a transient event)
- **Layout:** Reuses existing `.banner` class verbatim (from `index.css:413-428`); the warn-color variant (`.banner.warn`)
- **Copy:** `"HUD lost connection to the sidecar. Reconnecting…"` (matches the format of existing `ERRORS.SIDECAR_RESTART` copy)
- **Behavior:** auto-reconnect attempts continue every 1.5s (matching the existing main-WS reconnect cadence); banner auto-dismisses on reconnect. Slider rows continue to render with the LAST received `param-frame` snapshot's values (frozen in place); slider drags during the disconnected period buffer in renderer state but do NOT fire WS sends until reconnection (the renderer holds the last-attempted lock and re-fires on reconnect)
- **Locked state behavior during disconnect:** existing locked rows keep their visual locked state; the user can drag sliders freely (lock-engage signal queues for resend); upon reconnect, the renderer reconciles by sending all currently-locked-renderer-side `set-lock` messages to the sidecar (the sidecar IS the source of truth, but the buffered intent is recoverable — at worst, the user sees a 1.5s delay before their lock takes effect)

## Interaction Patterns

### IP-1 — Opening the HUD

1. User navigates to Settings → "Body motion plugin" section
2. User clicks `Open HUD` button
3. Renderer invokes `window.api.openHud()` → IPC to `electron-main`
4. `electron-main` creates a new `BrowserWindow` at default size (420×640), no position, with `webPreferences.preload` matching the main window, loads the renderer bundle with route hash `#/hud`
5. New window's renderer mounts → detects `/hud` route → `<HudRoot>` mounts → fires `GET /admin/rig-capabilities` (HTTP, not WS) → on success, computes the visible param list (= `writable_param_ids - SYSTEM_PRIMITIVE_OVERRIDES`) → opens `/hud/ws` → starts receiving 15Hz `param-frame` snapshots → renders rows
6. **Total time from click to first visible row:** ≤ 500ms in nominal case (rig-capabilities fetch is ~50-200ms, WS handshake ~50ms, first frame arrives within 67ms after WS open)
7. During mount, the loading state shows briefly; if mount completes within 100ms (warm sidecar) the loading state may flash and that's OK — no special "skip flash if fast" logic in v1

### IP-2 — Locking a parameter (drag-engage)

1. User hovers slider thumb on row `ParamAngleX`
2. User pointerdown on thumb (or anywhere on the slider track that snaps to a value); slider thumb visual locks to pointer
3. User drags pointer — slider thumb tracks pointer in real time (optimistic UI); on first non-trivial movement (>2 step units from pointerdown value), renderer fires `set-lock(ParamAngleX, current_value)` over `/hud/ws`
4. **Renderer-side optimistic lock visual fires immediately** — slider track tints accent, thumb flips accent, lock toggle flips to `<Lock>` icon and accent background. User sees "locked" visual within one animation frame
5. Sidecar receives `set-lock`, adds entry to `compositor.lock_state`, applies on next 60Hz tick. The next 15Hz `param-frame` returns `locked_ids` containing `ParamAngleX` — this is the sidecar's confirmation
6. Continued drag: renderer fires `set-lock(ParamAngleX, new_value)` at ~30Hz throttle; sidecar updates the locked value each tick
7. User pointerup — renderer fires final `set-lock(ParamAngleX, final_value)` to ensure exact resting position. **Lock remains engaged.** User can move pointer away
8. The locked value now wins against any LLM-driven `{variant}` / `<event>` write — verified by users seeing the lock hold even when they ask the LLM to emit a variant that drives `ParamAngleX`

### IP-3 — Unlocking a parameter

1. User clicks the lock toggle button on a locked row
2. Renderer fires `clear-lock(ParamAngleX)` over `/hud/ws`; renderer-side optimistic UI flips visuals back to unlocked immediately
3. Sidecar receives `clear-lock`, removes entry from `compositor.lock_state`. Next `param-frame` returns `locked_ids` without `ParamAngleX`
4. The compositor merge no longer enforces this lock; LLM-driven writes resume control of the param

### IP-4 — Avatar re-imported mid-session (locks cleared)

1. User completes an avatar re-import in the main window (Phase 8 flow)
2. Sidecar reloads `RigCapabilities`, **clears all entries in `compositor.lock_state`**, broadcasts a new `param-frame` over `/hud/ws` with `locked_ids: []` plus a `capabilities-changed` notification message (plan-phase decides whether the notification rides on `param-frame` as a flag or is a separate WS message — not a UI design concern, but the renderer needs SOME signal to re-fetch `GET /admin/rig-capabilities` and rebuild the visible-param-id set)
3. Renderer receives the change signal → re-fetches `/admin/rig-capabilities` → recomputes visible-param-ids → diffs against currently-rendered rows → adds new rows / removes vanished rows / preserves still-present rows (preserve scroll position to the extent possible)
4. Renderer fires a toast via `useStore().pushToast({ text: 'Avatar changed — locks cleared.' })` — toast appears in the HUD window's top-right corner via `<ToastStack>` and auto-dismisses after 4 seconds
5. User sees the new param list and the toast simultaneously; no other intervention needed

### IP-5 — `/hud/ws` disconnect mid-session

1. Sidecar process restarts (e.g., user clicks "Restart sidecar" in main app, or a sidecar crash + auto-restart)
2. HUD window's `/hud/ws` connection drops → renderer detects disconnect → renders inline `<HudErrorState>` banner at top of scroll area
3. Renderer attempts reconnect every 1.5s (existing main-WS pattern)
4. While disconnected: param rows freeze at last-known values; sliders are still draggable but `set-lock` sends are queued, not transmitted
5. On successful reconnect: banner disappears; renderer re-flushes any queued `set-lock` messages from the disconnect period (worst case: 1.5s of buffered intent); first new `param-frame` arrives and rows resume animating
6. **No toast on reconnect** — banner-disappear is the success signal; toasting on reconnect would be redundant chrome

### IP-6 — User closes HUD window

1. User clicks the OS-supplied close button (or types Cmd+W / Alt+F4 / equivalent)
2. `BrowserWindow` close triggers → `<HudRoot>` unmounts → `useEffect` cleanup runs → `/hud/ws` closes → sidecar's WS handler detects disconnect → sidecar stops the 15Hz throttle worker for THIS connection (other concurrent HUD windows, if any, continue independently — but v1 only supports one HUD window at a time)
3. **Locks survive HUD close.** The user can re-open the HUD and see the same locks active. Locks are sidecar state, not renderer state — closing the HUD is closing the diagnostic window, not "exiting lock mode"
4. Sidecar's `compositor.lock_state` continues to enforce locks at 60Hz against the running compositor pipeline regardless of HUD presence

## Aesthetic Guardrails

Inherited from Phase 1 + sharpened for HUD context:

- **Diagnostic, not decorative.** The HUD is a tool; chrome is minimal. No drop shadows, no gradients, no hover-reveal animations beyond the existing `:hover` background-mix patterns. The locked-state accent treatment is the only visual flourish, and it's load-bearing UX (communicating which rows are user-controlled)
- **Compact density first.** 44px row height is the floor (touch-target compliance), not the target. Rows that could be shorter without breaking accessibility may not be — 44px is the default. No "comfortable mode" toggle in v1
- **Native sliders.** Use `<input type="range">` with custom CSS rather than building a custom slider component. The native control gets keyboard accessibility (arrow keys), screen reader support, and pointer/touch events for free
- **Mono for raw IDs, body for human names.** Param IDs (`ParamAngleX`) and value displays (`0.42`) are mono so the eye can scan them as machine identifiers; cdi3 display names (`Angle X`, `Eye Smile L`) are body so they read as labels
- **No accent on neutral state.** A row with no lock and no special filter status is fully neutral. Accent is reserved for active filter chips and locked rows — never for default chrome
- **Status colors NEVER blend with accent.** The connection-status dot uses status-green/amber/red; it never tints to match the active accent (Sky vs Pewter vs Blush vs ...). This preserves the rule that status is communicating an objective system state, not a stylistic choice
- **No auto-scroll, no auto-focus.** When new rows appear (e.g. avatar re-import), do NOT scroll to top, do NOT focus first row. Preserve scroll position; the user is reading the HUD, not being driven by it
- **Static, not animated, framing.** The HUD window itself does not fade in / slide in. It pops at OS-default placement. The user opened it intentionally; it's not a notification

## Accessibility Contract

- **Keyboard navigation:** Tab order is filter chips → first param row → (within row: slider → lock toggle) → next param row → ... → footer. Shift+Tab reverses. Enter / Space activates lock toggle. Arrow keys on the focused slider increment/decrement by step
- **Screen reader:** every row has a labelled `<input type="range">` (`<label>` linking to slider via `htmlFor`, label text = display name + " value"). Lock toggle has dynamic `aria-label` per Copywriting Contract. Connection status dot has `aria-label={"HUD ${state} sidecar"}` where state ∈ {`connected to`, `reconnecting to`, `disconnected from`}
- **Focus-visible ring:** 2px outline in `var(--primary)` offset 2px on every interactive element (slider thumb, lock toggle, filter chips, retry button) — same rule as Phase 1
- **Color-blind safety:** the locked-state visual uses BOTH a color flip AND a glyph swap (lock icon flips from open to closed). Users with red-green color-blindness still see the icon-state change. The connection-status dot is the only color-only signal — but it's redundant with the `<HudErrorState>` banner that appears on disconnect, so a color-blind user has the textual fallback
- **Reduced motion:** there are no ambient animations in the HUD (no spinners, no decorative transitions). The slider thumb tracks pointer position directly — no motion to suppress. `prefers-reduced-motion` is satisfied automatically
- **Touch targets:** 44×44px floor satisfied by row height; lock toggle 28×28px button hosted inside the 44px row gives adequate hit area via row-level pointer event capture (plan-phase decides whether to widen the button itself or use row-level click delegation — visual contract is button-at-28, hit-area-at-44)
- **Contrast:**
  - Foreground 14px on `--card` background — verified ≥ 4.5:1 in all 7 themes per Phase 1
  - Muted-foreground 12px on `--card` background — Phase 1 verified for the chat bubble timestamp; same combination here
  - Accent on `--card` (locked-row slider fill, locked-toggle icon) — Phase 1 verified for the `Agent ⏵` toggle ON state and the active anchor pill; same combination here
  - Status-red, status-green, status-amber dots — color is signalling ONLY (redundant with banner text) so AA contrast is not strictly required, but Phase 1's status palette is AA-compliant on `--background` regardless
