# 21-01 Plan: Corpus Manifest And Scoring Harness Foundation

<objective>
Create the locked Phase 21 corpus manifest, privacy-safe user audio handling rules, scorecard schema, and reusable scoring primitives for semantic correctness, key-token retention, no-translation gates, and WER/CER diagnostics.
</objective>

<execution_context>
Phase 21 depends on Phase 19 and Phase 20. Before editing, inspect the actual STT provider package and test layout produced by Phase 19/20. If `sidecar/src/sidecar/stt/eval/` already exists, extend it; otherwise create it. Do not add runtime translation detection or any Chat submission behavior in this plan.
</execution_context>

<tasks>

## 1. Discover Actual STT And Test Structure

Read current files after Phase 19/20 execution:

- `sidecar/src/sidecar/stt/`
- `sidecar/tests/stt/`
- `sidecar/src/sidecar/admin/audio.py`
- `packages/contracts/py/contracts/audio_provider.py`
- `.planning/phases/19-stt-provider-abstraction-local-cloud-providers/*-SUMMARY.md`
- `.planning/phases/20-renderer-voice-capture-ptt-vad-preview-ux/*-SUMMARY.md`

Map actual provider request/result objects, readiness names, and test conventions. Use those names when creating eval helpers.

## 2. Add Privacy-Safe Eval Audio Ignore Rule

Update `.gitignore` with a narrow rule for local Phase 21 user voice recordings, such as:

```gitignore
.planning/eval-audio/
```

Do not ignore the committed corpus manifest or scorecard docs. The goal is to prevent accidental raw microphone audio commits while preserving repeatable metadata and expected text in git.

## 3. Create Locked Corpus Manifest

Create `.planning/phases/21-code-switch-evaluation-hardening/21-CORPUS.md`.

The manifest must include 20-30 generic companion-style cases covering:

- pure Chinese;
- pure English;
- mixed Chinese/English in one utterance;
- short natural phrases;
- numbers;
- simple commands;
- common non-product nouns;
- harmless hesitation/filler cases if useful.

Each case should include:

- stable case id;
- expected transcript text;
- category: `zh`, `en`, or `mixed`;
- key tokens;
- no-translation expectation;
- notes about acceptable variants where needed.

Do not include project/provider/avatar/product terms such as Teto, VTube Studio, LM Studio, FunASR, OpenAI, Groq, or GPT-SoVITS.

## 4. Document User Recording Procedure

In `21-CORPUS.md`, add a recording section that tells execution/UAT how to capture user-recorded audio once under real microphone conditions.

Rules:

- raw audio is local-only under `.planning/eval-audio/phase-21/`;
- use stable filenames matching corpus case ids;
- record with the same mic path used for Phase 20 Chat voice input where practical;
- store audio metadata and hashes in the scorecard after recording;
- do not commit raw audio.

## 5. Add Corpus Parser And Validator

Create `sidecar/src/sidecar/stt/eval/corpus.py`.

Implement lightweight structures/functions for:

- parsing the committed manifest or a machine-readable sidecar file if the executor chooses to create one;
- validating unique case ids;
- validating required category/key-token/no-translation metadata;
- validating local audio files by path when present;
- computing stable audio file metadata and sha256 without storing audio contents.

Tests should not require real user audio. Use temporary synthetic tiny WAV files where audio metadata behavior is needed.

## 6. Add Scoring Primitives

Create `sidecar/src/sidecar/stt/eval/scoring.py`.

Implement deterministic scoring helpers:

- text normalization that ignores harmless punctuation/casing/spacing;
- Chinese variant/script tolerance where feasible without adding a heavyweight conversion dependency;
- key-token retention checks;
- no-translation/collapsed-language gate for cases marked `zh`, `en`, or `mixed`;
- semantic score field that can be manually marked or rule-assisted for the first implementation;
- WER/CER/alignment diagnostics using `jiwer` if added, or a small local fallback if dependency risk is higher than value.

If adding `jiwer`, add it as a sidecar dev/test dependency unless runtime code needs it.

## 7. Create Scorecard Template

Create `.planning/phases/21-code-switch-evaluation-hardening/21-SCORECARD.md`.

Include:

- provider columns;
- corpus run metadata;
- local audio metadata/hash section;
- per-case pass/fail fields;
- semantic pass;
- key-token retention;
- no-translation status;
- WER/CER diagnostics;
- default recommendation decision area;
- Settings badge outcome area.

Leave provider result rows empty or marked pending; 21-02 fills them.

## 8. Add Tests

Add tests for:

- corpus ids are unique and required metadata exists;
- project-specific banned terms are absent from the committed corpus;
- audio ignore rule prevents raw eval audio from being staged by convention;
- key-token retention passes/fails deterministically;
- harmless formatting/casing differences do not fail;
- Chinese script/normalization variants do not fail when meaning is preserved;
- translated/collapsed outputs fail the no-translation gate.
