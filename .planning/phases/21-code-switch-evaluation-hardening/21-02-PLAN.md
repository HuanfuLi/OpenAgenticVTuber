# 21-02 Plan: Provider Scorecard Runner And Recommendation Evidence

<objective>
Build and run a repeatable scorecard runner that evaluates enabled STT providers against the locked corpus, records no-translation/key-token/semantic evidence, and produces a local-default recommendation without ever making cloud providers default.
</objective>

<execution_context>
This plan depends on the corpus and scoring foundation from 21-01. It also depends on the actual Phase 19 STT provider registry/readiness surface. Use fake providers in automated tests and only run real providers when their Phase 19 readiness gates are already satisfied.
</execution_context>

<tasks>

## 1. Discover Actual Provider Invocation Surface

Read the Phase 19 provider modules and readiness code after execution:

- provider base/protocol;
- registry/factory;
- local model cache status;
- cloud consent/credential checks;
- admin test transcription implementation;
- provider result/diagnostic shape.

Reuse the existing provider transcribe path. Do not add a second STT registry or bypass readiness.

## 2. Add Eval Runner

Create `sidecar/src/sidecar/stt/eval/runner.py`.

The runner should:

- load the locked corpus;
- resolve local audio files by case id;
- evaluate one selected provider or all currently ready providers;
- use Phase 19 provider readiness and selected provider config;
- refuse cloud providers unless Phase 19 consent, credentials, and readiness already pass;
- call transcription tasks only, never translation endpoints/tasks;
- record provider id, model id, relevant language/task/normalization options, audio metadata, latency, and redacted failure state;
- produce deterministic per-case score records.

Automated tests should use fake providers with controlled transcripts.

## 3. Add Score Aggregation And Recommendation Logic

Create `sidecar/src/sidecar/stt/eval/report.py`.

Aggregate per-provider results into:

- total pass rate;
- category pass rates for `zh`, `en`, and `mixed`;
- semantic pass count;
- key-token retention failures;
- no-translation hard failures;
- WER/CER diagnostic summaries;
- latency summary;
- local default recommendation status.

Recommendation rules:

- cloud providers are never default;
- FunASR/SenseVoice remains expected local default unless another local provider is a clear winner;
- if no local provider passes well enough, keep the best local provider but mark code-switch support limited/experimental;
- record rationale in `21-SCORECARD.md`.

## 4. Fill Scorecard Artifact

Update `.planning/phases/21-code-switch-evaluation-hardening/21-SCORECARD.md` with:

- runner command used;
- environment/provider readiness notes;
- audio metadata and hashes if user recordings exist;
- per-provider summary table;
- local-default recommendation result;
- pending/blocked rows when real providers are unavailable.

Do not paste raw transcript logs beyond expected/hypothesis snippets necessary for debugging. Avoid secrets, raw provider exceptions, and raw audio paths beyond local-only tokens.

## 5. Add CLI Or Test Entry Point

Expose a lightweight command path for repeatability. Choose the least invasive pattern after inspecting project conventions:

- a Python module runnable with `uv run python -m sidecar.stt.eval.runner ...`; or
- a sidecar project script if existing patterns support it.

The command should support:

- corpus path;
- audio root;
- provider id or all-ready providers;
- output scorecard JSON/Markdown path;
- dry-run/fake-provider mode for tests.

## 6. Add Tests

Cover:

- fake providers produce deterministic pass/fail scorecards;
- no-translation failures fail a case even if meaning is preserved;
- missing key tokens fail key-token gate;
- cloud provider run is refused without consent/readiness;
- cloud provider is excluded from default recommendation even when it scores highest;
- another local provider must be a clear winner to replace FunASR/SenseVoice;
- weak local results produce limited/experimental recommendation copy state.

</tasks>
