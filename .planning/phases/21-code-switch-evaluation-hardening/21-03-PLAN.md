# 21-03 Plan: Recommendation Badges And Targeted Provider Hardening

<objective>
Translate scorecard evidence into local-provider recommendation metadata, concise Settings badges, and targeted STT provider hardening when evidence shows a clear fix, without changing cloud default policy or adding runtime translation policing.
</objective>

<execution_context>
This plan depends on 21-02 scorecard output. Start by reading `21-SCORECARD.md` and only harden provider behavior for gaps that are evidenced there. If no real provider results are available, implement the contract/UI path with fake scorecard fixtures and mark live hardening blocked until provider UAT.
</execution_context>

<tasks>

## 1. Determine Recommendation Metadata Shape

Inspect Phase 18/19 Settings provider catalog/status contracts.

If the existing provider catalog already has capability labels, extend it minimally with code-switch quality badge state such as:

- `code_switch_tested`;
- `limited_code_switch`;
- `recommended_local`;
- `experimental_code_switch`;
- `not_evaluated`.

If a cross-process contract is needed, update Pydantic contracts and generated TypeScript/schema files through the existing codegen path.

Do not expose detailed scorecard rows or raw transcripts to the renderer.

## 2. Apply Local Recommendation Logic

Use `sidecar/src/sidecar/stt/eval/report.py` or the actual provider catalog layer to map scorecard results to provider badge states.

Rules:

- cloud providers keep Cloud/Requires API key badges but never receive default recommendation state;
- FunASR/SenseVoice stays recommended local default unless another local provider is a clear scorecard winner;
- weak local code-switch performance becomes Limited code-switch or Experimental code-switch;
- untested providers are Not evaluated, not Code-switch tested.

## 3. Add Concise Settings Badges

Update Settings voice input UI after inspecting actual Phase 18/19 files.

Expected files may include:

- `apps/renderer/src/screens/Settings/VoiceInputSection.tsx`;
- `apps/renderer/src/screens/Settings/ProviderCapabilityBadges.tsx`;
- `apps/renderer/src/lib/copy.ts`.

Badge rules:

- badges are short, compact, and scannable;
- show Local/Cloud and Chinese/English labels from existing provider metadata;
- show Code-switch tested only when scorecard passes;
- show Limited code-switch or Experimental code-switch for weak local results;
- do not add a user-facing scorecard table;
- do not promote cloud quality comparisons.

## 4. Implement Targeted Provider Hardening If Evidence Supports It

Read scorecard failures and provider outputs. Only make narrow fixes with clear tests, such as:

- ensure Whisper-family providers use transcription task/endpoint, not translation;
- set automatic/no-manual language mode where supported;
- configure SenseVoice/FunASR `language="auto"` and appropriate ITN behavior consistent with readable final transcripts;
- strip known SenseVoice metadata tags if they leak into transcript text;
- preserve mixed-language tokens and key terms during normalization;
- keep raw provider diagnostics redacted.

Do not:

- add a new provider;
- rewrite the provider registry;
- add runtime transcript rejection;
- translate or normalize final text into one conversation language.

## 5. Add Contract, Provider, And Settings Tests

Cover:

- generated contracts include quality badge metadata if added;
- local recommendation changes only for clear local winner;
- cloud provider never receives default recommendation;
- failed code-switch eval removes/avoids Code-switch tested badge while provider remains selectable;
- Settings renders badges only, not detailed scorecard data;
- UI does not mention cloud quality comparisons;
- provider hardening fixes evidenced tag/task/language issues without changing successful transcripts.

</tasks>
