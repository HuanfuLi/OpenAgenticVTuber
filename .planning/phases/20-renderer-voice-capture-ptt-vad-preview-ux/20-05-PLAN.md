# 20-05 Plan: Voice Input Readiness Recovery And STT Enablement Persistence

<objective>
Fix the live Phase 20 readiness gaps that block push-to-talk after startup or settings changes: sidecar startup errors must clear when sidecar becomes ready, successful STT tests must persist readiness into config, and Chat must refresh voice readiness after Voice settings changes.
</objective>

<execution_context>
This is a gap-closure plan created from live UAT feedback. Preserve the Phase 20 architecture: renderer capture remains preload-mediated, sidecar runtime transcription still uses the selected Phase 19 STT provider only, and final transcripts still submit through the existing `text-input` path unchanged.

Do not weaken the STT readiness gate. The fix is to persist and refresh real readiness, not to make Chat ignore readiness failures.
</execution_context>

<tasks>

## 1. Make Sidecar-Unavailable Readiness Recoverable

Update the voice input store so `sidecar_unavailable` and transient sidecar request failures are treated as recoverable readiness states.

Rules:

- startup `Sidecar is not ready.` may be displayed while sidecar is actually unavailable;
- once a later readiness response is ready or blocked for a different current reason, stale sidecar-unavailable text must be cleared;
- `voice.error` should represent actionable current errors, not an old startup state;
- non-recoverable failures such as permission denied or missing STT readiness should still be visible.

Add focused store tests for clearing stale `sidecar_unavailable` after a later ready response.

## 2. Refresh Chat Readiness On Sidecar Ready/Reconnect

Update `VoiceInputControl` or a nearby Chat voice lifecycle hook to refresh voice readiness when Electron emits sidecar-ready or when the renderer websocket reconnects.

Rules:

- do not start microphone capture automatically;
- do not dismiss real STT disabled/readiness errors without a fresh readiness response;
- if the fresh response is ready, the mic button becomes enabled;
- if the fresh response says microphone permission is needed, PTT remains clickable enough to request permission.

Add `ChatVoiceInput` coverage for startup `Sidecar is not ready.` followed by sidecar-ready causing a readiness refresh and clearing the blocked mic state.

## 3. Persist Successful STT Test Readiness Into Settings State

In Voice settings, when `testSttProvider` returns `ok: true` with a non-null readiness result:

- merge `result.readiness` into `sttConfig.readiness`;
- preserve the provider/language/model values that were tested;
- show the test success summary;
- do not mark readiness active for failed or empty-transcript tests;
- when provider/model/language/cloud config changes later, keep the existing invalidation behavior by resetting readiness or allowing fingerprint validation to invalidate it.

Add Settings tests proving that a successful test followed by Save writes `audio.stt.readiness.active_allowed: true` and the returned fingerprint/timestamps.

## 4. Refresh Chat Readiness After Voice Settings Save

After saving Voice settings:

- notify the renderer voice input store that config changed, or refresh readiness after the sidecar restart/ready signal;
- clear stale `Voice input is disabled in Voice settings.` only after the fresh readiness response no longer reports it;
- keep a visible current blocker if the saved config is still disabled, untested, missing model, or missing cloud consent/credential.

Prefer a small existing-pattern mechanism, such as store subscription, window event, or explicit exported invalidation helper. Avoid adding a broad app-wide event bus.

## 5. Add Regression Coverage

Cover:

- startup sidecar-unavailable error clears after sidecar-ready and fresh ready response;
- mic becomes enabled after readiness recovery;
- stale `Voice input is disabled in Voice settings.` does not persist after valid saved config plus fresh readiness;
- failed STT tests do not persist active readiness;
- successful STT tests persist readiness through `saveStoredConfig`;
- PTT still cannot bypass readiness if STT remains untested or disabled.

</tasks>
