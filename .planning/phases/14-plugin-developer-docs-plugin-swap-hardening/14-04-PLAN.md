# 14-04 Plan: Reconnect Chat WebSocket After Plugin-Switch Restart

<objective>
Close the Phase 14 recheck gap where the chat input sometimes remains disabled after plugin-switch sidecar restarts.
</objective>

<execution_context>
Phase 14 gap recheck passed AvatarOverrides docs and the VTS connection-count restart fix. The final fallback/chat check passed, but revealed a new restart-adjacent issue: after sidecar restart, the chat input textarea can remain disabled until a full app restart.

The chat input disabled expression is `turnInFlight || banners.llm || !wsOpen`. The likely failure mode is `wsOpen=false`: the renderer WS singleton in `apps/renderer/src/ws/client.ts` initializes once from `window.api.getReadyUrl()` and reconnects to `state.url`, but it does not globally subscribe to later `sidecar:ready` events and replace the old socket/url after a restart. Plugin switching now restarts sidecar frequently, so this stale URL/reconnect edge is much easier to trigger.
</execution_context>

<tasks>

## 1. Make The WS Client Follow New Sidecar Ready URLs

Update `apps/renderer/src/ws/client.ts`.

Requirements:
- subscribe once to `window.api.onSidecarReady`;
- when a new ready URL arrives, update the stored URL and connect to that URL;
- if there is an open or connecting socket for a different URL, close it and replace it;
- prevent old close handlers from reconnecting to stale URLs after a newer URL has been selected;
- keep the existing reconnect-with-fixed-backoff behavior for same-URL transient disconnects.

Prefer a small generation/token mechanism so stale socket close handlers do not schedule reconnects for old URLs.

## 2. Reset Transient Streaming State On Sidecar Reconnect

If a sidecar restarts mid-turn, the old in-flight turn cannot complete. Reset transient streaming state when the WS client connects to a new sidecar URL:
- clear `inputDisabled`;
- clear `isSpeaking`;
- clear any thinking/in-flight pending turn state that would keep the input disabled;
- preserve already committed conversation history.

Use the existing `resetStreaming()` helper if it is sufficient. If it clears too much, add a narrower helper in `useStreamingMessages.ts`.

## 3. Add Regression Coverage

Add or extend renderer tests.

Cover:
- `onSidecarReady` with a new URL causes the WS client to switch from the old socket to the new URL;
- stale close/reconnect from the old socket does not reconnect to the old URL;
- chat input becomes enabled after a sidecar-ready reconnect when no LLM banner is active.

If direct WS-client unit tests are awkward because the module initializes on import, use `Chat.test.tsx` with a fake `WebSocket` and mocked `window.api.onSidecarReady`.

## 4. Verify

Run:

```powershell
npm run typecheck:renderer
npm --workspace apps/renderer run test -- --run Chat Settings StatusIcon
```

Manual UAT:
- Select plugins a few times to restart sidecar.
- After each restart settles, confirm the chat input enables without a full app restart.
- With `broken_motion_test` selected, confirm fallback/null status remains visible and chat still accepts input.
</tasks>

<success_criteria>
- Chat input does not remain disabled after plugin-switch sidecar restarts once the new sidecar is ready.
- WS client does not reconnect to stale sidecar URLs after a restart.
- Regression tests cover the restart/reconnect behavior.
- Phase 14 can be rechecked to completion.
</success_criteria>
