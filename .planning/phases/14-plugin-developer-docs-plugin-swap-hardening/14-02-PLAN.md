# 14-02 Plan: Plugin Listing, Swap, Restart, And Health Hardening

<objective>
Make plugin swapping behave like a supported product feature: invalid plugins are visible, selected plugins take effect through an automatic sidecar restart, runtime plugin health is shown in Settings and Status, and broken plugins fall back to null motion without breaking chat or losing the user's selected configuration.
</objective>

<execution_context>
Plan 14-01 creates the authoring docs, helper kit, and sample plugin. This plan hardens the app behavior that plugin authors and users will exercise while following those docs.

The current Settings plugin picker lists plugin manifests from Electron, but it does not expose invalid entries, runtime plugin lifecycle, or reliable "this selected plugin is actually running" health. The sidecar plugin selection is boot-time, so saving Settings without restart is misleading. Phase 14 must close that product gap.
</execution_context>

<current_system_facts>
- `apps/electron-main/src/sidecar.ts` owns sidecar process lifecycle and currently has `restartSidecar()` plus lightweight plugin listing.
- `apps/electron-main/preload/index.ts` exposes `bodyMotion.listPlugins()` and `bodyMotion.getSidecarStatus()`.
- `apps/renderer/src/screens/Settings/Settings.tsx` already has a Body Motion plugin section with a plugin select control.
- `apps/renderer/src/chrome/StatusIcon.tsx` currently reports LLM, VTS, and sidecar health, not plugin health.
- `sidecar/src/sidecar/plugins/supervisor.py` falls back to `NullPlugin` if plugin initialization fails, but the UI has no precise runtime state for that fallback.
</current_system_facts>

<tasks>

## 1. Define Plugin Catalog And Runtime Status Types

Extend preload/shared TypeScript types for plugin catalog entries.

Each catalog entry should include:
- `name`;
- `version`;
- `description`;
- `source`;
- `path`;
- `valid`;
- `selectable`;
- optional `statusSummary`;
- optional `developerDetails`;
- optional `manifestApiVersion`;
- optional `isSelected`.

Add a runtime plugin-health type with:
- selected plugin id/name;
- loaded plugin id/name when available;
- lifecycle state from the approved set: `active`, `restart pending`, `load failed`, `fallback/null`, `circuit open`, `invalid manifest`, `unknown/loading`;
- actionable summary;
- optional expandable developer details;
- whether fallback/null motion is active;
- whether chat should remain available.

Keep naming consistent between Electron, renderer, and sidecar. Prefer string literal unions in TypeScript so UI tests catch unsupported states.

## 2. Harden Plugin Listing And Invalid Manifest Display

Update Electron plugin discovery so invalid manifest directories still appear in Settings.

Expected behavior:
- valid manifests show normal metadata;
- missing/invalid required fields show a clear error state;
- duplicate or unsupported manifest identities show developer details;
- invalid entries are still selectable, but the UI warns that selecting them will restart into fallback/null motion until fixed;
- no plugin directory is silently hidden solely because its manifest is invalid.

Use a deterministic catalog order matching the current Settings behavior as closely as possible.

Add tests where available, or renderer tests with mocked catalog entries if Electron-main unit test coverage is not currently set up.

## 3. Add Sidecar Runtime Plugin Status

Add a sidecar admin endpoint, for example `GET /admin/plugin/status`, wired into the existing sidecar admin/server structure.

Status must report:
- selected plugin from config/env;
- loaded plugin or null plugin;
- lifecycle state;
- fallback/null active flag;
- actionable summary;
- developer details including load/initialization exception text when available;
- circuit-open state if supervisor circuit breaker is active.

Update plugin loader/supervisor only as much as needed to retain the current failure reason and expose status cleanly.

Do not let plugin load failure prevent sidecar boot. Chat, status polling, and Settings should continue to work.

Add `sidecar/tests/admin/test_plugin_status_endpoint.py` and extend plugin loader/supervisor tests for:
- valid selected plugin active;
- invalid selected plugin reports invalid/load-failed/fallback as appropriate;
- initialization failure keeps selected config and reports fallback/null;
- circuit-open state reports distinctly.

## 4. Wire Runtime Status Into App State And Status Popover

Extend app status polling so renderer state includes plugin health.

Implementation path:
- expose a preload method for plugin runtime status, or include plugin health in the existing sidecar status polling if that is cleaner;
- update `apps/renderer/src/state/status-types.ts` and store normalization;
- update `StatusIcon.tsx` to show one compact plugin-health row in the popover;
- include action summary and expandable/copyable developer details in the popover when status is not active.

Status mapping:
- `active`: healthy;
- `restart pending` and `unknown/loading`: pending/degraded;
- `invalid manifest`, `load failed`, `fallback/null`, and `circuit open`: degraded/error for plugin health only;
- chat availability must remain independent of plugin health.

## 5. Auto-Restart Sidecar On Plugin Selection Save

Update Settings save behavior so changing the selected body-motion plugin:
- persists the selected plugin value;
- marks plugin status `restart pending` immediately;
- restarts the sidecar automatically;
- refreshes plugin runtime status after restart;
- does not disable chat permanently if plugin restart fails;
- keeps the selected plugin in Settings even when the runtime falls back to NullPlugin.

UI requirements:
- Settings plugin section shows catalog validity and runtime state separately.
- Save/restart progress is visible but compact.
- Invalid manifest selection shows a warning before or during save using existing local UI patterns.
- Failure details include one short action line and expandable developer details.

## 6. Update Docs And UAT Checklist

Patch docs from 14-01 as needed after the final runtime behavior is implemented:
- plugin-system integration flow;
- how to interpret lifecycle states;
- how to test sample plugin switching;
- what happens when a plugin fails.

Add/confirm UAT steps:
- follow docs to run/select the sample plugin;
- switch from default to sample and observe automatic sidecar restart;
- select a deliberately broken plugin and observe fallback/null status;
- confirm selected plugin remains configured after failure;
- confirm chat remains available while plugin motion is degraded.

<success_criteria>

- Selecting a plugin causes an automatic sidecar restart and the selected plugin takes effect when valid.
- A broken selected plugin keeps the selected config, runs fallback/null motion, and exposes clear failure state.
- Status popover includes plugin health without crowding the existing LLM/VTS/sidecar status.

<risks_and_mitigations>
- Risk: TypeScript and Python validation diverge. Mitigation: keep Electron catalog validation focused on display and selection warnings, while sidecar runtime status remains authoritative for actual load behavior.
- Risk: Restart timing creates confusing transient UI. Mitigation: model `restart pending` explicitly and refresh runtime status after restart.
- Risk: Plugin errors pollute global app health. Mitigation: represent plugin health as its own row/state and keep chat availability independent.
</risks_and_mitigations>
