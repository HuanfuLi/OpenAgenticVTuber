# 14-03 Plan: Close Plugin Docs And Restart UAT Gaps

<objective>
Close the Phase 14 UAT gaps: make AvatarOverrides documentation useful to plugin authors, and make plugin-switch restarts terminate the existing Windows sidecar process tree so VTS connections do not accumulate.
</objective>

<execution_context>
Phase 14 UAT found two issues:

1. `AvatarOverrides` is referenced in the docs but not explained. Plugin authors need to know what it contains, when to use it, and how default-plugin action bindings relate to avatar-specific behavior.
2. Plugin switching appears to restart logs/UI state but does not visibly disconnect the existing VTS motion-control plugin. VTS connected-plugin count increases after each switch, which indicates the old sidecar process can survive restart.

Root cause for issue 2: Electron starts the sidecar through `shell: true` on Windows, yielding a wrapper chain such as `cmd.exe -> uv.exe -> python.exe`. `shutdownSidecar()` calls `handle.child.kill()`, which can kill only the shell wrapper. The Python watchdog watches Electron's PID, and Electron remains alive during intentional restart, so orphaned Python descendants can keep the VTS websocket open.
</execution_context>

<tasks>

## 1. Patch AvatarOverrides Author Documentation

Update `docs/plugins/motion-plugin-authoring.md` with a dedicated `AvatarOverrides` section.

Cover:
- what the object represents: avatar-specific saved catalog/config loaded at sidecar boot;
- fields plugin authors are likely to care about: `body_sway_strategy`, `default_plugin_action_bindings`, variants/events as dispatch catalog context, source rig path metadata;
- fields most motion plugins should not treat as motion output controls, such as voice settings;
- when to use `RigCapabilities` first versus `AvatarOverrides`;
- how `default_plugin_action_bindings` are mainly for the production default plugin, while custom plugins may ignore them or use them as optional avatar-specific hints.

Update `docs/plugins/ai-motion-plugin-playbook.md` so AI agents have a concise adaptation rule:
- inspect `AvatarOverrides` for avatar-specific hints and default-plugin bindings;
- still gate emitted params through `RigCapabilities`;
- do not assume variants/events are plugin action codes.

Patch `docs/plugins/default-and-sample-plugins.md` if needed to make the default-plugin/action-binding relationship clear.

## 2. Fix Windows Sidecar Restart Process-Tree Cleanup

Update `apps/electron-main/src/sidecar.ts`.

Implement a shutdown path that reliably terminates the full sidecar process tree on Windows:
- preserve current `shutdownSidecar()` public API;
- keep graceful waiting behavior when the process exits normally;
- on Windows, use `taskkill /pid <pid> /t /f` or an equivalent process-tree termination after initiating shutdown;
- on non-Windows, keep direct child kill behavior unless a safer local pattern already exists;
- ensure `state.handle`, `state.readyUrl`, and `state.intentionalShutdown` are cleared consistently after shutdown;
- avoid treating intentional shutdown as a crash.

Prefer a small helper such as `terminateProcessTree(child)` that is unit-testable without launching the real sidecar.

## 3. Add Regression Coverage

Add or extend tests at the most practical level available in this repo.

Minimum expected coverage:
- `shutdownSidecar()`/process-tree helper uses Windows tree termination when `process.platform === "win32"` or when injected platform is win32;
- intentional shutdown clears state and does not mark a crash/respawn path;
- existing Settings/Status tests continue to pass.

If Electron-main currently has no test runner, make the helper dependency-injectable and document the manual verification. Do not add a large new test framework solely for this gap unless the repo already supports it.
