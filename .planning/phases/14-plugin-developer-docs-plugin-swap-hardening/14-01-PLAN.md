# 14-01 Plan: Plugin Developer Contract, Helper Kit, Sample Plugin, And Agent Playbook

<objective>
Create the human and AI developer documentation needed to build motion-control plugins against the real Phase 13/14 system, and add the small supported helper kit and sample plugin that make the documented path executable.
</objective>

<execution_context>
Phase 14 was inserted before audit because the project now needs a real plugin-authoring surface, not just an internal default plugin. The current core plugin API is intentionally small, but plugin authors still need guardrails for common failure-prone work: parsing bracket action codes from LLM text, ramping values over time, writing only supported parameters, and producing safe finite ParamFrames.

This plan intentionally does not harden Settings plugin swapping or runtime status. That is plan 14-02. This plan builds the contract and authoring materials that 14-02 can reference during UAT.
</execution_context>

<current_system_facts>
- `BodyMotionPlugin`, `ParamFrame`, `RigCapabilities`, and `AvatarOverrides` already form the practical plugin interface in `sidecar/src/sidecar/plugins/api.py`.
- `plugin.yaml` is validated by `MotionPluginManifest` in `sidecar/src/sidecar/plugins/manifest.py`.
- The default plugin already demonstrates realistic action-code driven motion in `plugins/default/__init__.py`, but it is too large to serve as the only starting point.
- There is no `docs/plugins/` tree yet.
- There is no stable plugin-author helper module yet.
</current_system_facts>

<tasks>

## 1. Add A Stable Plugin Author Helper Kit

Create `sidecar/src/sidecar/plugins/sdk.py` as the documented helper surface for plugin authors.

Implement a narrow API only:
- action-code extraction for bracketed codes such as `[wave]`, with case-normalization and allow-list filtering;
- optional pending-fragment support if the existing default-plugin parser already requires it;
- ramp/easing helpers for smooth 0..1 motion weights;
- finite numeric filtering;
- writable-parameter filtering using either `RigCapabilities.writable_param_ids` or an explicit writable-id set;
- safe construction of add-parameter `ParamFrame` values.

Constraints:
- Keep helpers pure and dependency-light.
- Do not duplicate the whole default plugin architecture.
- Keep names and docstrings stable enough to publish as plugin API v1.
- Export the helper module or selected helper symbols through `sidecar/src/sidecar/plugins/__init__.py` only if the existing package style supports that.

Add `sidecar/tests/plugins/test_plugin_sdk.py` covering:
- bracket code extraction with valid, invalid, repeated, mixed-case, and partial input;
- finite filtering drops NaN/Infinity and non-numeric values;
- writable filtering drops unsupported parameter ids;
- ramp/easing outputs are bounded and monotonic for expected cases;
- safe frame construction returns a `ParamFrame` that only contains finite writable adds.

## 2. Add A Minimal Sample Motion Plugin

Create `plugins/sample_motion/` with:
- a valid `plugin.yaml`;
- a compact `BodyMotionPlugin` implementation;
- use of the new helper kit for at least action-code parsing and safe frame creation;
- one or two simple action codes that write to common VTube Studio parameters only when capabilities allow them.

The sample should prioritize clarity over expressiveness:
- short class;
- small action map;
- no hidden global state beyond simple smoothing if needed;
- comments only where the plugin-authoring pattern would otherwise be unclear.

Add `sidecar/tests/plugins/test_sample_plugin.py` covering:
- manifest discovery/validation for the sample plugin;
- initialization against a minimal `RigCapabilities`;
- generation of a finite `ParamFrame`;
- unsupported parameters are skipped instead of emitted.

## 3. Write Human Developer Documentation

Create `docs/plugins/README.md` as the entry point with links to the detailed docs.

Create `docs/plugins/motion-plugin-authoring.md` with:
- quickstart directory layout;
- `plugin.yaml` field reference using the current manifest model;
- lifecycle of `BodyMotionPlugin.on_load`, `on_token_stream`, optional `on_action_code`, and optional `on_unload`;
- `ParamFrame` semantics and safe value expectations;
- how to use `RigCapabilities` and `AvatarOverrides`;
- supported helper kit functions with examples;
- local test commands;
- common mistakes and how to diagnose them.

Create `docs/plugins/plugin-system-integration.md` with:
- how the app discovers plugins;
- how plugin selection is configured;
- how the sidecar loads a plugin and falls back to null motion;
- what authors should expect during restart/swap flows once 14-02 is implemented;
- how to package a plugin directory for local development.

Create `docs/plugins/default-and-sample-plugins.md` with:
- when to read the sample plugin first;
- when to study the default plugin;
- a concise map of default-plugin concepts, including action codes, avatar overrides, capability gating, and smoothing.

Docs must be code-verified: every public class/function/path/test command mentioned should exist after this plan completes.

## 4. Write The Tool-Neutral AI Motion Plugin Playbook

Create `docs/plugins/ai-motion-plugin-playbook.md` for coding agents.

The playbook must:
- be independent of Codex-specific tool names;
- tell agents to inspect the target repository before editing;
- identify the stable contract files and helper kit;
- describe a safe adaptation loop for taking an existing motion algorithm and wiring it into this plugin API;
- require tests for manifest validity, capability gating, finite frames, and at least one text-to-motion behavior;
- warn against bypassing `RigCapabilities`, emitting non-finite values, relying on undocumented app internals, or silently changing plugin manifests.

## 5. Add A Thin Codex Adapter Skill

Create `.codex/skills/agenticllmvtuber-motion-plugin/SKILL.md`.

Keep it thin:
- point to `docs/plugins/ai-motion-plugin-playbook.md` as canonical;
- tell Codex to follow the tool-neutral playbook and repo docs;
- mention local verification commands;
- avoid duplicating the full playbook.

Do not create Claude/Gemini-specific adapters in this plan. The canonical artifact is the shared playbook.

- Risk: docs drift from code. Mitigation: every referenced API/path/test command must be checked before completing the plan.
- Risk: sample plugin duplicates default plugin behavior. Mitigation: keep sample deliberately minimal and point advanced users to the default plugin worked example.
</risks_and_mitigations>
