# 18-01 Plan: Audio Settings Persistence, Consent, And Redaction

<objective>
Add the durable configuration and privacy boundary that Phase 18 Settings will use: audio settings in encrypted StoredConfig, separate cloud-STT credential/consent fields, typed IPC/preload methods, sidecar env passing, and shared redaction helpers for diagnostics.
</objective>

<execution_context>
Phase 18 depends on Phase 17. Before editing, inspect the actual Phase 16/17 outputs for audio config, provider catalog, voice preset, and provider status contract names. If upstream already created equivalent files, extend those files instead of creating duplicates named here.

Current repository baseline at planning time has `safe-storage.ts` schemaVersion 1 and no audio config. Existing config saves restart the sidecar. Preserve that behavior unless upstream already ships a tested turn-boundary audio hot-swap path.
</execution_context>

<tasks>

## 1. Discover And Reuse Upstream Audio Contracts

Read the current contract files under `packages/contracts/py/contracts/` and generated TS mirrors.

If Phase 16/17 created these or equivalent contracts, extend them:
- audio config
- provider catalog/capabilities
- voice presets
- provider health/runtime status
- TTS/STT test request/result

If they do not exist, add minimal Phase 18 fields only where needed for Settings and diagnostics. Do not implement provider engines in this plan.

Run contract codegen after contract edits:

```powershell
npm run codegen:contracts
npm run check:contracts
```

## 2. Add StoredConfig Audio Persistence

Extend `apps/electron-main/src/safe-storage.ts` to persist an `audio` section.

Required shape, adjusted to upstream contract names:
- `audio.tts.activeProviderId`, selected preset id, output mode/device summary if upstream supports it.
- `audio.stt.enabled`, `audio.stt.activeProviderId`, input mode defaulting to push-to-talk, and local model/cache settings if upstream exposes them.
- `audio.stt.cloud.<provider>.apiKey` fields separate from LLM provider credentials.
- `audio.stt.cloud.<provider>.consentGranted` flags defaulting to false.
- `audio.diagnostics.redactionEnabled` defaulting to true.
- schema migration from v1 to the current schema version with Piper/default local settings and cloud STT disabled.

Do not store raw transcripts in config. Do not store absolute reference-audio paths in diagnostics payloads; use upstream asset ids or redacted display names.

## 3. Add Electron IPC And Preload Methods

Extend `apps/electron-main/src/ipc.ts`, `apps/electron-main/preload/index.ts`, and `apps/electron-main/preload/index.d.ts` with typed methods for:
- loading/saving audio settings through the same encrypted config path;
- listing audio providers from the sidecar admin provider catalog;
- reading audio runtime status;
- testing TTS/STT provider configuration;
- importing or selecting reference audio only if Phase 17 exposes the asset API;
- redacting a diagnostics payload before exposing it to renderer tests, if that boundary is easier to keep main-side.

Cloud STT test calls must reject or return a blocked status when provider consent or credentials are missing.

## 4. Pass Redacted Audio Config To Sidecar

Extend `apps/electron-main/src/sidecar.ts` to pass only the sidecar-needed audio configuration to the sidecar environment, using the upstream Phase 16 config-loading path.

Rules:
- pass API keys only when a cloud provider is explicitly configured and consented;
- do not log env payloads containing secrets;
- preserve existing plugin cursor tracking env behavior;
- keep sidecar restart on config save.

## 5. Add Shared Redaction Helpers

Add or extend sidecar redaction helpers so provider diagnostics can sanitize:
- API keys and bearer tokens;
- local home-directory paths and reference audio full paths;
- transcript text;
- raw provider exception strings that may include request URLs, headers, file names, or transcript snippets.

Add matching main/renderer redaction utility only if needed by the UI tests. Prefer one canonical sidecar redaction path for admin diagnostics.

</tasks>
