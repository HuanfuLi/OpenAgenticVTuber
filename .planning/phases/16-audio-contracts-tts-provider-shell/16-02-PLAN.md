# 16-02 Plan: TTS Provider Shell and Piper Adapter

<objective>
Move Piper behind a sidecar-owned TTS provider shell while preserving the existing ordered playback, renderer audio payload, and VTS RMS/lipsync behavior.
</objective>

  <behavior>
    - A `TTSProvider` abstraction can synthesize one sentence into provider-neutral audio output.
    - Piper adapter keeps LFS pointer guard, voice load, ORT warmup, model sample-rate detection, and shutdown semantics.
    - Tests can use a fake provider without importing Piper or opening a sounddevice stream.
  </behavior>
  <action>
    Create provider types such as:
    - `TTSProvider`
    - `TTSSynthesisRequest`
    - `TTSSynthesisResult` with `pcm_int16`, `sample_rate`, optional provider diagnostics.

    Implement `PiperTTSProvider` by moving Piper-specific voice loading and synthesis behind the provider. Keep stream ownership either in `TTSGateway` or a clearly named sidecar playback gateway; do not require future network providers to know about WebSocket or compositor queues.

    Add tests for:
    - missing model and LFS pointer guard still raise clear exceptions.
    - real Piper boot still works when the model is present.
    - fake provider synthesis result can be converted to the existing audio payload shape.
  </action>

  </done>

<task type="auto" tdd="true">
  <name>Task 2: Make audio payload assembly provider-neutral</name>
  <files>
    - sidecar/src/sidecar/tts/audio_payload_helpers.py
    - sidecar/tests/test_audio_payload_helpers.py
  </files>
  <behavior>
    - Existing `AudioPayloadMessage` shape does not change.
    - RMS volume calculation still uses normalized 20ms chunks from the same PCM that is written to the output stream.
    - Silent payload behavior still returns `audio=None`, `volumes=[]`, and no stream write.

  </behavior>

    - keep `get_volume_by_chunks`;
    - add a provider-neutral helper such as `prepare_payload_from_pcm(pcm_int16, sample_rate, display_text, dispatches, sentence_id, chunk_length_ms=20)`;
    - keep a compatibility wrapper only if needed by tests during migration.

    Ensure tests prove generated WAV headers and RMS behavior are unchanged.
  </action>

    <automated>python -m pytest sidecar/tests/test_audio_payload_helpers.py -q</automated>
  </verify>

  </done>

<task type="auto" tdd="true">
  <name>Task 3: Refactor TTSTaskManager to use the provider shell</name>
  <files>
    - sidecar/src/sidecar/tts/tts_manager.py
    - sidecar/tests/test_tts_manager.py
  </files>
  <behavior>
    - `TTSTaskManager.speak()` still returns immediately and allows parallel provider work.
    - Out-of-order provider completions are still buffered until the next sequence number is ready.
    - For non-silent payloads, queue-put, WebSocket send, and stream write still occur in the existing order.
    - `stream.write` still runs through an executor.
    - Provider synthesis runs through an async off-event-loop boundary such as `asyncio.to_thread()` or provider-owned executor work.
    - Silent action-only payloads still skip provider synthesis, compositor speech envelope, and stream write while still sending the audio envelope with dispatches.
  </behavior>
  <action>
    Replace manager dependence on `voice` with a provider object. Keep stream and queue ownership in the manager. Do not rewrite the sequencing algorithm unless required by the provider interface.

    Add an event-loop responsiveness test:
    - fake provider performs blocking work in its synth method;
    - schedule a sentinel coroutine immediately after `speak()`;
    - assert the sentinel runs before provider work completes.

    Add a regression test that the manager does not call provider synthesis for whitespace/action-only silent payloads.
  </action>
  <verify>
    <automated>python -m pytest sidecar/tests/test_tts_manager.py -q</automated>
  </verify>
  <done>
    Manager behavior stays ordered and provider synthesis cannot block the event loop.
  </done>
</task>
