<task type="auto" tdd="true">
  <name>Task 2: Add explicit provider failure behavior to the TTS queue</name>
  <files>
    - sidecar/src/sidecar/tts/provider.py
    - sidecar/src/sidecar/tts/tts_manager.py
    - sidecar/tests/test_tts_manager.py
  </files>
  <behavior>
    - Provider timeout/failure is logged with provider id, sentence id, and typed health/failure state.
    - A failed sentence cannot leave `_payload_queue.join()` stuck.
    - Failure does not silently switch providers mid-turn.
    - Failure sends a deterministic audio envelope for the sentence with `audio=None`, existing display text/dispatches, and no compositor speech envelope or stream write.
    - Subsequent queued sentences keep their ordering behavior.
  </behavior>
  <action>
    Add a small provider exception/result mapping layer. Suggested approach:
    - provider raises typed `TTSProviderError(state, summary, retryable)`;
    - `TTSTaskManager._process_tts` catches it and enqueues a silent/error-marked payload for the same sequence number;
    - optionally emit a log envelope later if the existing WS/log path supports it, but do not invent a broad UI event system in this phase.

    Add tests:
    - failed provider enqueues/sends silent payload and releases queue join;
    - failed first sentence does not unblock second sentence out of order incorrectly;
    - no stream write and no speech envelope on failed synthesis.
  </action>
  <verify>
    <automated>python -m pytest sidecar/tests/test_tts_manager.py -q</automated>
  </verify>
  <done>
    Provider failures are explicit and queue-safe.
  </done>
</task>
