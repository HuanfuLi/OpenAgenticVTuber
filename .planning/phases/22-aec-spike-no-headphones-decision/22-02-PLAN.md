# 22-02 Plan: Self-Speech Guard And Active TTS Correlation

<objective>
Prevent assistant TTS from becoming final user input by adding a provider-independent self-speech guard before the existing final STT-to-Chat submission path.
</objective>

<execution_context>
This plan depends on 22-01 diagnostics/status contracts and the actual Phase 20 final voice submission path. It must not pause all microphone capture by default, must not add barge-in cancellation, and must not create a new Chat submission path.
</execution_context>

<tasks>

## 1. Locate Final Voice Submission Path

Inspect actual Phase 20 implementation for:

- final PTT transcript submit function;
- final VAD transcript submit function;
- preview-only state;
- queueing while a turn is in flight;
- STT provider result shape;
- speaking/playback state signal.

Identify the single point where a final transcript becomes Chat `text-input`. The guard must run immediately before that point.

## 2. Add TTS Playback Lifecycle Signal

Extend `apps/renderer/src/ws/audio-player.ts` or the actual playback module so code can observe:

- playback queue starts;
- each audio payload starts;
- each audio payload ends/errors;
- all queued playback drains.

Use a small subscription or store update pattern consistent with `ws/store.ts`. Avoid leaking object URLs or retaining audio bytes.

If existing `setSpeaking(true/false)` from `ws/store.ts` is precise enough after Phase 20, reuse it and add tests proving queued audio keeps active-TTS state accurate.

## 3. Track Recent Assistant Output

Expose a redacted, bounded recent assistant text window from existing streaming state or a new guard-local tracker:

- only recent assistant display text needed for overlap comparison;
- bounded by count/time so it cannot become a conversation-history copy;
- never persisted;
- reset on sidecar reconnect/session reset.

This can reuse `appendAssistantSentence` or dispatcher events if that avoids duplicating state.

## 4. Implement Self-Speech Guard

Create `apps/renderer/src/audio/self-speech-guard.ts`.

The guard input should include:

- final transcript candidate;
- source mode: PTT or VAD if available;
- active TTS/playback state;
- recent assistant text window;
- provider id if available;
- timestamp/latency metadata if available.

Behavior:

- if no active TTS and no recent risky window, accept;
- during active TTS, normalize transcript and recent assistant text;
- conservatively discard when the transcript is exact/near-exact substring, high token overlap, or otherwise clearly assistant-output overlap;
- accept user overlap speech that does not appear assistant-like;
- return redacted diagnostics with reason codes, never raw audio.

Keep the first implementation deterministic and testable. Do not call an LLM for semantic matching.

## 5. Wire Guard Before Final Submit

Update the actual Phase 20 voice submission path:

- run guard before `appendUserMessage` / `send(text-input)` or the Phase 20 equivalent;
- do not show discarded assistant-like transcript in preview or Chat;
- log redacted diagnostics for discarded candidates;
- preserve Phase 20 queueing behavior for accepted transcripts;
- preserve PTT availability when STT is ready.

Ensure cloud and local STT final results both use this same path.

## 6. Add Tests

Add tests for:

- assistant text replay during active TTS is discarded;
- near-substring/overlap cases are discarded conservatively;
- unrelated user speech during active TTS is accepted immediately;
- guard applies to both VAD and PTT/final STT paths if both exist;
- discarded transcript is not appended to Chat and not persisted to conversation history;
- diagnostic payload contains reason codes/metadata but not raw transcript text when redaction policy requires it;
- no barge-in/cancel signal is emitted.

</tasks>
