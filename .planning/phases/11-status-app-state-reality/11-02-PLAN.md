# Phase 11 Plan 02: Restore Post-Setup LLM Reconfiguration

## Goal

Close the Phase 11 UAT gap where the app truthfully shows persisted LLM status but provides no way to change provider/model settings after first-run setup.

The finished behavior: Settings > Connection / Models lets the user edit provider, endpoint URL, model name, and API key, test the connection, save through the existing Electron safeStorage config path, and see the status popover reflect the saved provider/model. No mock status mutation or hardcoded `qwen2.5` behavior may return.

## Gap Inputs

- `.planning/phases/11-status-app-state-reality/11-UAT.md`
- `.planning/debug/phase-11-llm-reconfigure-gap.md`
- `.planning/phases/11-status-app-state-reality/11-01-SUMMARY.md`

## Requirements Covered

- STAT-01: Provider/model display reflects real persisted setup state.
- STAT-04: Normal chrome does not mutate mock status.
- STAT-05: Setup-derived app state uses real persistence instead of mock safe storage.

## Non-Goals

- Do not implement Phase 12's broader Settings cleanup for Avatars, VTube Studio, Conversation, Memory, Log level, or Status bar model display outside this LLM configuration gap.
- Do not add new LLM providers beyond the existing setup provider list.
- Do not persist API keys outside the existing `window.api.saveStoredConfig` safeStorage-backed path.
- Do not make a blank model field an implicit real model. Blank still means auto-detect.

## Tasks

### 1. Make the Settings connection section editable

**Files:** `apps/renderer/src/screens/Settings/Settings.tsx`, `apps/renderer/src/lib/copy.ts`

Actions:
- Replace the disabled "Change provider" affordance with a real edit flow.
- Hydrate the editor from `window.api.getStoredConfig()` when available, falling back only for the initial loading frame.
- Include provider, endpoint URL, model name, and API key controls. Reuse `ProviderSelect` or extract a small shared editor component rather than duplicating provider options.
- Keep a read-only summary mode acceptable only if the user can enter edit mode and save changes from the same section.
- Make blank model explicit in copy and UI as "auto-detect if blank".
- Remove stale deferred copy that says provider reconfiguration lands later.

Verify:
- Manual: Settings shows a way to edit provider/model after setup completion.
- Automated: renderer test can click the edit control and see editable provider/model controls.

Done:
- The user is no longer stuck with LM Studio + auto-detect after first-run setup.

### 2. Persist updated LLM config through the existing stored-config contract

**Files:** `apps/renderer/src/screens/Settings/Settings.tsx`, `apps/renderer/src/state/setup-store.ts`, `apps/renderer/src/state/app-store.tsx`

Actions:
- Save edited config with `window.api.saveStoredConfig`.
- Preserve unrelated `StoredConfig` fields, especially `plugin.activePluginName`.
- Keep `hasCompletedSetup: true` and `schemaVersion: 1`.
- Do not leak API keys into app-store state, React logs, or test snapshots.
- Refresh app status after a successful save so the status popover and Settings summary use the updated provider/model.
- If a helper is added to `setup-store.ts`, keep it a thin wrapper over the same stored-config persistence used by first-run setup.

Verify:
- Automated: saving a custom model calls `saveStoredConfig` with the edited provider config and preserved plugin config.
- Automated: after save, `refreshStatus` or equivalent status hydration is called.

Done:
- The persisted setup config changes from Settings and the app status reflects those changes without restart when possible.

### 3. Reuse or adapt connection testing safely

Verify:

## Acceptance Criteria

1. A completed-setup user can open Settings > Connection / Models and edit provider, endpoint URL, model name, and API key.
2. Saving writes to the existing stored config path and preserves unrelated config fields.
3. The status popover and Settings summary show the saved provider/model after save or refresh.
4. Blank model is displayed as auto-detect only because the stored model is blank, not because the UI is hardcoded.
5. No production path reintroduces `mockStatus`, `mockSafeStorage`, fake latency, or hardcoded `qwen2.5`.
