# Phase 20.1 - UI Design Contract

> Visual and interaction contract for the Settings sidebar refactor and required-input-only enablement simplification. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope

Phase 20.1 replaces the current long-scroll Settings screen with a two-pane Settings shell: sidebar categories on the left and the selected settings panel on the right. The default panel is Connection. The refactor covers the whole Settings surface, not only voice input.

**In scope**
- Replace anchor pills and long-scroll rendering with a persistent category sidebar and selected panel.
- Use user-facing categories while exposing underlying systems inside panels: Connection, Avatar, Motion, VTube Studio, Voice, Conversation, Appearance, Diagnostics, and About.
- Hide deferred placeholder sections until they contain real controls. Do not display Skills, Agent, Scheduler, Form factor, or Hotkeys placeholders.
- Keep optional tests in their relevant panels: `Test connection`, `Test synthesis`, and `Test selected STT`.
- Remove test-before-enable language and behavior from Settings UI. Enable/save blockers are limited to missing required inputs, missing cloud consent/API keys, missing local model/cache where required, invalid local assets, unavailable dependencies, or explicit provider health blockers.
- Preserve redaction, explicit cloud consent, visible selected-provider failures, and no silent provider fallback.

**Out of scope**
- New STT providers, new GPT-SoVITS installer/training UI, wake word, translation, barge-in, memory/retrieval controls, agent scheduling, form factor controls, and any no-headphones/AEC success claim before Phase 22.

Sources: `20.1-CONTEXT.md` decisions D-01 through D-19, `.planning/ROADMAP.md` Phase 20.1 success criteria, `.planning/REQUIREMENTS.md` AUDIO-01, STT-03, STT-04, VIN-01, PRIV-01, PRIV-02, and existing Settings code.

## Spacing Scale

Declared values for new Phase 20.1 UI:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline label/status gaps, small helper spacing |
| sm | 8px | Row gaps, sidebar item padding, compact button groups, inline validation spacing |
| md | 16px | Panel padding, grouped section padding, form field margin, card padding |
| lg | 24px | Sidebar-to-panel gutter, major group separation, Settings shell padding on wide screens |
| xl | 32px | Panel section separation and bottom breathing room |
| 2xl | 48px | Empty panel vertical spacing only if a real empty state appears |
| 3xl | 64px | Not used in this phase |

Exceptions: none for new Phase 20.1 CSS. Existing legacy CSS values may remain only where untouched; new sidebar and panel styles must use the scale above.

Sizing constraints:
- Sidebar width: 176px to 220px on desktop, fixed during interaction to prevent layout shifts.
- Right panel max readable width: 760px, aligned left rather than centered as a decorative card.
- Standard buttons and inputs: 36px height.
- Sidebar item and touch targets: at least 36px height; icon-only controls, if any, at least 32px and keyboard focusable.

## Typography

Use only these four sizes and two weights for new Phase 20.1 UI.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 400 | 1.4 |
| Heading | 18px | 600 | 1.3 |
| Display | 20px | 600 | 1.2 |

Rules:
- Settings title may use 20px semibold only at the shell/panel header level.
- Panel section headings use 18px semibold.
- Field labels follow existing `.settings-form .label` treatment and may be 14px semibold through the existing class.
- Helper, validation, diagnostics, and optional test result summaries use 12px regular.
- Do not introduce additional type sizes or weights for badges, sidebar items, or status chips.

## Color

Use existing CSS variables so all seven theme classes remain compatible.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` | Settings shell background and panel background |
| Secondary (30%) | `var(--card)` / `var(--popover)` | Sidebar surface, grouped panels, form controls, dialogs, optional test log surfaces |
| Accent (10%) | `var(--primary)` | Active sidebar category, primary save CTA, enabled switches, selected radio state, focus-visible outline |
| Destructive | `var(--destructive)` | Clear history, reset app data, remove local model, delete/clear actions, blocking validation errors |

Accent reserved for: active sidebar category, selected provider/preset/radio dot, primary save/apply action, active switches, focus ring, and success emphasis after user-triggered tests. Accent must not be used for every clickable element, all helper links, or optional diagnostic controls.

Semantic status colors:
- `var(--success)`: optional test passed, provider reachable, required input valid.
- `var(--warning)`: optional test failed or runtime dependency warning that does not mutate saved config.
- `var(--destructive)`: missing required input blocking save/enable, failed selected-provider runtime use, blocked destructive action.

## Copywriting Contract

All user-visible strings must live in `apps/renderer/src/lib/copy.ts`, following the existing `COPY.SETTINGS` pattern.

| Element | Copy |
|---------|------|
| Primary CTA | `Save settings` |
| Panel-level missing-required heading | `Required settings missing` |
| Panel-level missing-required body | `Add {fields} before saving this configuration.` |
| Empty state heading | `No settings here yet` |
| Empty state body | `This category is hidden until it has usable controls.` |
| Error state | `Could not save settings. Review the required fields, then try again.` |
| Optional test error | `Test failed. Your saved settings were not changed.` |
| Destructive confirmation | Clear history: `Clear conversation history? This deletes saved sessions and starts a fresh New chat.` |
| Destructive confirmation | Reset app data: `Reset app data? Provider settings, avatar state, and local history will be cleared.` |
| Destructive confirmation | Remove local STT model: `Remove local STT model? Voice input using this model will fail until a model is available again.` |

Locked test labels:
- `Test connection`
- `Test synthesis`
- `Test selected STT`

Forbidden copy patterns:
- Do not use `Run a passing test before enabling`, `not tested yet`, `certified`, `validated for activation`, `ready checklist`, or similar gate-adjacent language for optional tests.
- Do not show raw API keys, full reference-audio paths, raw transcripts from diagnostics, stack traces, or unredacted provider errors in primary Settings panels.
- Do not imply automatic cloud fallback, automatic translation, no-headphones support, or AEC success.

Required-input copy:
- Cloud STT consent: `Cloud transcription can send microphone audio to the selected provider.`
- Cloud STT key: `API key is required for this cloud provider.`
- Local STT model: `A local model is required before this provider can run.`
- GPT-SoVITS reference audio: `Reference audio and transcript are required for this preset.`
- GPT-SoVITS URL: `Server URL is required for GPT-SoVITS.`

## Interaction Contract

### Settings shell

- Replace `.settings-scroll` plus `.anchor-pills` with a Settings shell that has:
  1. Sidebar category navigation.
  2. Selected panel content.
  3. Optional panel-level status area near the panel heading.
- The primary visual focal point is the selected panel header plus its required-input summary, not the sidebar.
- Default selected category is Connection on first open.
- Category switching is instant and does not discard unsaved edits silently. If an existing form already saves immediately, preserve that behavior; otherwise keep in-panel save controls clear.
- Preserve the surrounding app chrome and bottom rail behavior.
- On narrow widths, the sidebar may collapse to a top segmented/category list only if labels remain readable and target sizes remain at least 36px.

### Sidebar categories

Use this order and intent:

| Category | Panel Content |
|----------|---------------|
| Connection | LLM provider, endpoint, model, API key, `Test connection`, save/edit controls |
| Avatar | Current avatar catalog state, edit/import actions, active voice association summary if already present |
| Motion | Body motion plugin selection and cursor tracking |
| VTube Studio | VTS connection, authorization, troubleshooting actions |
| Voice | Voice output (TTS/GPT-SoVITS/presets/reference audio) and Voice input (STT/PTT/VAD/cache/cloud consent) |
| Conversation | History/session retention and clear-history action |
| Appearance | Theme mode, light accent, dark background, dark accent, preview |
| Diagnostics | Logs, redacted diagnostics, reset actions |
| About | Version, channel, links |

Do not render Memory as a visible category unless it has usable settings beyond a deferred status note. Do not render placeholder-only future categories.

### Enablement and saving

- Save/enable actions are blocked only by required input validation.
- Required input blockers appear both inline near the field and in a concise panel-level summary when more than one blocker exists.
- Optional tests never mutate enabled/saved state on failure.
- Optional tests clear their previous result when related fields change.
- Runtime selected-provider failure remains visible but does not silently switch providers or rewrite user selection.
- If a selected provider cannot run at runtime, show a visible failure and keep the user's selected provider unchanged.

### Voice panel

- Voice panel uses two system-visible groups:
  1. `Voice output (TTS)` for Piper/GPT-SoVITS, presets, reference audio, test synthesis.
  2. `Voice input (STT)` for selected STT provider, local model cache, cloud consent/API key, PTT shortcut, VAD controls, test selected STT.
- Provider names remain visible inside the Voice panel. Do not hide STT/TTS system distinctions behind a single generic voice toggle.
- Keep VAD conservative and opt-in. Do not claim no-headphones support; use the Phase 20 safety copy that AEC/no-headphones behavior is deferred to Phase 22.
- `Test selected STT` is a diagnostic action. It may show transcript/redacted diagnostics for the current screen session, but it is not a prerequisite for enabling voice input.

### Diagnostics

- Diagnostics surfaces show redacted metadata and concise summaries by default.
- Use the existing `.test-log` / logs drawer pattern for technical detail.
- Failed optional tests must show `Test failed. Your saved settings were not changed.` or a provider-specific equivalent with a next action.
- Reset and destructive actions remain grouped away from routine provider configuration.

## Accessibility Contract

- Sidebar navigation uses `role="tablist"` and `role="tab"` with `aria-selected`, or native buttons with `aria-current="page"`. The selected panel must be reachable by keyboard after category selection.
- Each panel has one visible heading. The heading text must match the selected sidebar category.
- Required-field validation text is associated with the relevant control via `aria-describedby` where practical.
- Panel-level required summaries use `role="status"` for non-blocking updates and avoid focus stealing.
- Runtime failure banners use text, not color alone.
- Optional test results use `aria-live="polite"`.
- Destructive confirmations use existing `role="alertdialog"` pattern with explicit cancel and confirm buttons.
- Focus ring remains `2px solid var(--primary)` and must not be removed from sidebar items, buttons, inputs, radio rows, or switches.
