# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1 (Plumbing & Process Lifecycle). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope of this contract:** the entire v1 chrome shell (top bar, bottom rail, status icon popover, slide-in History panel, Logs drawer, Settings page, Agent page) plus the **fully functional** LLM Setup screen and the Phase 1 echo-chat surface. Non-functional surfaces (History list, Agent runtime, Settings sections 2-14) ship as branded placeholder copy per D-22 (Path 1: chrome shell + placeholders). Phase 2-4 will reuse these tokens; if they need new tokens they extend, not replace.
>
> **Greenfield UI:** No existing design system was found. This document is the project's first design contract. Tokens declared here are the seed for every subsequent phase.

## Spacing Scale

Declared values from the standard 8-point set `{4, 8, 16, 24, 32, 48, 64}`, tuned for a 400×700 window. **12px is deliberately excluded** from the scale to prevent sub-grid misalignment; the standard set skips it. All spacing in the doc uses one of the six tokens below or one of the chrome exceptions.

| Token | Value | Tailwind class | Usage |
|-------|-------|---------------|-------|
| `xs` | 4px | `space-1`, `gap-1`, `p-1` | Icon-to-label gap, inline tag padding, badge inner padding |
| `sm` | 8px | `space-2`, `gap-2`, `p-2` | Compact element spacing (button inner padding vertical, portal-card inner row gap), **chat-bubble padding, popover inner padding, input inner padding** (compact density per project stance) |
| `md` | 16px | `space-4`, `gap-4`, `p-4` | Default vertical rhythm between message blocks, settings row vertical gap, cards |
| `lg` | 24px | `space-6`, `gap-6`, `p-6` | Section gaps in Settings page, setup-screen form-section gaps |
| `xl` | 32px | `space-8`, `gap-8`, `p-8` | Setup screen outer padding, empty-state vertical breathing room |
| `2xl` | 48px | `space-12`, `gap-12`, `p-12` | Page-level vertical spacing on Settings (between major section groups) |

**Chrome exceptions (justified structural sizes — not scale members):**
- **Top bar height: 36px** — sized to fit `Hexagon` 16px icon + 8px vertical padding + room for the `Agent ⏵` toggle pill at 28px height. Original USERFLOW.md said "~32px"; UI researcher locks 36px because 32px clips lucide icons at 16px with any padding margin.
- **Bottom rail height: 56px** — three tap targets each ≥48px tall for touch ergonomics; 4px top/bottom padding gives a 56px rail. Original USERFLOW.md said "~48px"; UI researcher locks 56px because true touch-target minimum from WCAG 2.5.5 / Material is 48px, and the rail's 8px chrome padding pushes total to 56px.
- **Window minimum width: 320px** — icon-only mode for bottom-rail tabs; below this, labels render but truncate. The USERFLOW.md "~280px soft floor" is treated as a *soft* hint; 320px is the hard `minWidth` enforced by `BrowserWindow`.
- **Window minimum height: 480px** — preserves: 36px top bar + 56px bottom rail + ~150px setup form essentials + 16px outer padding × 2. Below this, the LLM Setup screen scrolls within its outer container.
- **Logs drawer collapsed strip height: 32px** — single-line `Logs ▾`, smaller than top bar by design — drawer is secondary chrome.
- **Status popover width: 280px** — fixed; fits "qwen2.5-7b" model names without truncation, anchored top-right under `⬢`.
- **Slide-in History panel width: 80% of window width, capped at 360px** — so on a stretched-wide window the panel stays a panel, not a half-screen split.

## Typography

Three sizes, two weights — chat density tuned for a 400px-wide window. All sizes given in `rem` for Tailwind compatibility (root = 16px).

**Two weights only — 400 regular + 600 semibold.** No third weight is declared anywhere in the contract. Inline emphasis (including error-state emphasis) is achieved through *color* (`--destructive` red) and *glyph* (`⚠`), not through weight. The 600 semibold provides sufficient contrast against 400 regular for any in-line emphasis need.

| Role | Size | Weight | Line Height | Tailwind | Usage |
|------|------|--------|-------------|----------|-------|
| **Body** | 14px (`0.875rem`) | 400 (regular) | 1.5 | `text-sm leading-normal` | Chat messages, settings field labels, button text, popover content, drawer logs (when not in mono context), error banners, placeholder copy |
| **Small / Meta** | 12px (`0.75rem`) | 400 (regular) | 1.4 | `text-xs leading-snug` | Timestamps, status-popover detail lines, badges, tooltip text, bottom-rail tab labels, "Coming in milestone-N" placeholder subtitle |
| **Heading** | 18px (`1.125rem`) | 600 (semibold) | 1.3 | `text-lg font-semibold leading-snug` | Setup screen title ("Connect a language model"), Settings section headers, History panel sections ("Today" / "Yesterday"), Agent page section headers ("Sessions", "Saved goals", "Defaults") |

**Mono variant (logs / connection-test panel only):** 12px, regular, 1.5 — `font-mono text-xs leading-normal`. Used for the verbose connection-test log on the LLM Setup screen, the Logs drawer content, and any READY-line / error-trace surfacing.

**Why three sizes only:** A 400px window cannot afford a four-tier hierarchy without crowding. Headings at 18px are visibly larger than body 14px (1.29× ratio — comfortably above the 1.2 minimum for legible hierarchy); 12px meta is small but acceptable at desktop viewing distance. No display tier — the LLM Setup screen's "Connect a language model" heading is 18px, not 24px+, because 400px width clips longer-line headings.

**Error-state inline emphasis (executor reference):** Error copy such as `⚠ LLM unreachable — connection refused at {host}:{port}` renders as **400 regular body text** in `--destructive` (status-red) color, prefixed with the `⚠` glyph. The color + glyph combination — not weight — carries the emphasis. Do NOT introduce a 700 bold weight or any other third weight to the project.

### Accent Reserved For (explicit list)

The accent (`--primary` lavender-violet) is **forbidden** on every element except those listed below. Default surfaces use neutral foreground/border tokens.

1. **Primary CTAs:** `Test connection`, `Continue →`, `Approve ▶`, `Re-test connection`, `Open VTube Studio docs ↗` (when surfaced as primary), `+ New thread`, `+ New goal`, `Add avatar`
2. **Bottom-rail active tab:** filled icon + label color of the currently-selected tab (Chat / Agent / Settings)
3. **`Agent ⏵` toggle ON state:** background fills with `--primary`, foreground flips to white-on-violet
4. **Focus ring on inputs and buttons:** `outline-2 outline-offset-2 outline-primary` (WCAG 2.4.7)
5. **Sticky agent pill:** background `--primary`, foreground white
6. **Settings anchor pill (active section):** subtle `--primary/15` tint background

**Forbidden uses of accent:** any non-CTA button, any default link color (use `--foreground` underline instead), all chrome (top bar / bottom rail / Settings rows in inactive state), all non-action icons.

### Color Contrast Targets

All foreground-on-background pairs MUST meet WCAG 2.1 AA:
- Body text on background: contrast ≥ 4.5:1
- Body text on `--muted` (input fields): contrast ≥ 4.5:1
- White text on `--primary` accent (CTAs): contrast ≥ 4.5:1
- White text on Status-Red / Status-Amber: contrast ≥ 4.5:1 (amber is the tightest — verified at `oklch(0.78 0.16 75)`)
- Disabled / `--muted-foreground` on `--background`: contrast ≥ 3:1 (relaxed for non-essential text per WCAG 1.4.3 "incidental")

## Copywriting Contract

**Voice and tone:** *Warm companion who happens to be technical.* The app is a Live2D-avatar conversation partner; copy should feel like the avatar's natural speech (in chat states) and like considered product copy (in chrome / errors). Avoid:
- Corporate stiffness ("Please ensure your connection is configured correctly")
- Faux-quirky ("Whoops! Something broke 😅")
- Anthropomorphic over-reach in error states (the avatar speaks; the *system* doesn't have feelings)

The rule: **chrome is calm, errors are direct, empty states are brief and forward-looking.**

### Destructive confirmations

Phase 1 ships **one** destructive action: `Reset all state` (Settings §15 Diagnostics). The action wipes `safeStorage` (clearing `hasCompletedSetup` + provider config + API key), and `electron-store` (window pos, theme, log-drawer state).

| Action | Confirmation pattern | Copy |
|--------|----------------------|------|
| `Reset all state` | shadcn `AlertDialog` modal, two-button (`Cancel` / `Reset everything`); `Reset everything` button is `--destructive` red. | Title: `Reset all state?` Body: `This clears your provider configuration, window settings, and any other saved preferences. The next launch will start at the LLM setup screen. This cannot be undone.` Buttons: `[ Cancel ]   [ Reset everything ]` |

**Anti-patterns that ARE forbidden in Phase 1:**
- Native browser `confirm()` dialog
- Single-step "Reset" without confirmation
- Toast-only feedback for destructive actions (must be modal)

## Interaction Patterns

### State-machine routing (USERFLOW.md §4)

The app has three routes: `setup`, `main`, and (future) `pet`. Phase 1 implements `setup` and `main`. Routing is gated by `safeStorage.hasCompletedSetup`:

- App boots → main process reads `hasCompletedSetup`
- `false` → renderer mounts `<SetupScreen>` only (no chrome)
- `true` → renderer mounts `<AppShell>` with chrome + active route from bottom rail

The state machine has **no reverse path** in skeleton. Re-test happens via the status popover button; full re-config (`Change provider →`) is grayed and disabled.

### Setup screen blocking behavior (D-09, PLUMB-04)

- `Continue →` button is **disabled** until a successful test completes (state: `idle | testing | success | error`)
- Successful test → `Continue →` enables; `Test connection again` link appears below
- Failed test → log panel shows verbose error; `Continue →` stays disabled; user must Test connection again
- Closing the window during setup → `app.quit()` (no partial config persisted; safeStorage write only on `Continue →` click)

### Bottom-rail tab switching (USERFLOW.md F)

- Tap a tab → switch route. **Preserves Chat scroll position** (use a ref-tracked scroll position keyed by route).
- Settings always opens scrolled to top.
- The `☰` hamburger in top bar is **hidden** (display: none) when not on Chat route. Reserve the 36px slot to prevent layout shift — only the icon is hidden.
- Active-tab affordance: filled icon variant + accent color on label; inactive: outline icon variant + `--muted-foreground` label.

### Status icon popover (USERFLOW.md §3.4, G)

- Click `⬢` → popover anchors top-right under the icon (offset 4px from icon bottom edge, 8px from window right edge)
- Popover contains: 3-row status list (LLM / VTS / Sidecar) + `[ Re-test connection ]` button
- `[ Re-test connection ]` runs the same 1-token test as setup
  - During: button text → `Testing...`, button disabled, spinner left of label
  - Success: row updates to green, popover stays open, "last reply 423ms" updates
  - Failure: inline error appears below row in red mono text; row turns red; banner appears in main app surface (per L.1)
- Close popover: click outside, Esc, or click `⬢` again.

### Logs drawer toggle (USERFLOW.md K)

- Off (Settings §15 toggle false): drawer is `display: none`. **No chevron strip rendered.** Chat surface butts directly against bottom rail.
- On: chevron strip renders above bottom rail at 32px height with `Logs ▾` label
  - Click strip → expand to last-saved height (default 200px) with smooth 200ms transition
  - Drag handle (top edge of expanded drawer) → resize between 80px and `windowHeight - 200px`
  - Click `▴` chevron in expanded header → collapse to 32px strip
  - Drawer state (`enabled`, `collapsed`, `height`) persists via electron-store

### History slide-in (USERFLOW.md D)

- Tap `☰` → `Sheet` slides from left with 250ms ease-out timing
- Width: `min(80vw, 360px)`
- Backdrop: 30% black overlay over chat surface (chat content **not** removed from DOM, just dimmed)
- Close: tap backdrop, Esc, or `✕` in panel header
- Skeleton content: placeholder centered vertically per "Skeleton state ⏳" in USERFLOW.md D.3

### Focus management

- Setup screen: focus moves to first field (Endpoint URL) on mount
- After successful test: focus moves to `Continue →` button
- Status popover open: focus moves to first focusable element (first row's status, or `[ Re-test ]` button)
- AlertDialog (Reset all state): focus moves to `Cancel` button (the safe default)
- Esc closes any open popover / sheet / dialog without committing destructive actions

### Keyboard shortcuts (Phase 1)

| Key | Action | Surface |
|-----|--------|---------|
| Enter | Submit chat message (single-line input) | Chat |
| Esc | Close popover / sheet / dialog | Chrome |
| Tab | Standard tab order | Everywhere |
| Ctrl/Cmd+, | Open Settings | Anywhere |

(Global hotkeys like F9 test-prop are Phase 4. Triple-Esc kill-switch is agent-runtime.)

## Accessibility

- All interactive elements have visible focus rings (2px accent outline, 2px offset)
- All icons paired with text label OR `aria-label`
- **Hamburger `☰` button:** `aria-label="Open conversation history"` (explicit, since the icon-only button has no visible text label). Disabled state on Agent / Settings views uses `aria-hidden="true"` since the icon is not rendered.
- **Status icon `⬢`:** `aria-label` reflects current state — `"Status: all connected"` / `"Status: VTube Studio degraded"` / `"Status: LLM error"` (worst-of-three component included so the label is meaningful before the popover opens).
- **Agent toggle `Agent ⏵`:** `aria-pressed="true|false"` reflecting ON / OFF state. In skeleton, `aria-disabled="true"` with the disabled tooltip copy as the announcement.
- Bottom-rail tabs have `role="tab"` and `aria-selected`; the rail itself is `role="tablist"`.
- AlertDialog uses `role="alertdialog"` with focus trap (shadcn handles)
- Chat messages have `role="log"` with `aria-live="polite"` so screen readers announce new content without interrupting
- Logs drawer content is `role="log"` `aria-live="off"` (it's chrome-secondary, not user-facing alerts)
- Color is never the sole indicator of state — the `⬢` icon has both color AND a small overlay glyph (`✓` / `!` / `✕` based on state) so colorblind users can still distinguish
- Minimum tap target 44×44px for any rail tab; 32×32px for chrome icons (status / hamburger) — within the WAI ARIA touch-target relaxation for desktop

## Phase Boundary Notes for Downstream Phases

These tokens are the **seed** for every later phase. Phase 2-5 should:

- **Phase 2** (real LLM streaming): Reuse all tokens. Add token shimmer / sentence-streaming animation — recommendation: a 12px `--muted-foreground` "thinking" dot pulse before each sentence, replaced by sentence text on arrival. The shimmer skeleton component is `skeleton` from shadcn. **No new color tokens needed.**
- **Phase 3** (TTS audio): No visual changes. Optionally add a tiny audio-waveform glyph next to currently-speaking sentence. **No new tokens.**
- **Phase 4** (compositor + VTS bridge): The VTS window is external; nothing to design *in our window* for the avatar itself. Cursor tracking sends events sidecar-direct. **No new visual tokens.** Only addition: F9 prop-toggle could surface a transient toast confirmation ("Prop toggled") — uses existing `sonner`.
- **Phase 5** (verification + codegen): No visual changes. Verification is a markdown doc, not UI.

If any later phase wants to introduce a new color, font size, or spacing value, the convention is to **extend** the scale (e.g., add `2xs: 2px` if compactness demands), not redefine an existing token. Token redefinition requires re-approval through `/gsd:ui-phase`.
