# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1 (Plumbing & Process Lifecycle). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope of this contract:** the entire v1 chrome shell (top bar, bottom rail, status icon popover, slide-in History panel, Logs drawer, Settings page, Agent page) plus the **fully functional** LLM Setup screen, the Phase 1 echo-chat surface, **and the user-selectable theme system (Settings §14 Appearance)**. Non-functional surfaces (History list, Agent runtime, Settings sections 2-13) ship as branded placeholder copy per D-22 (Path 1: chrome shell + placeholders). Phase 2-4 will reuse these tokens; if they need new tokens they extend, not replace.
>
> **Greenfield UI:** No existing design system was found. This document is the project's first design contract. Tokens declared here are the seed for every subsequent phase.
>
> **Revision 2 (2026-05-06):** Total replacement of the previous lavender-violet single-accent system per CONTEXT.md D-23. The new system is a **multi-theme, user-selectable** color contract (warm-cream Light + 3 accent options; deep-blue/near-black Dark backgrounds × 2 accent options) with a flat / Material-You-leaning aesthetic (no shadow elevation, larger radii, surface separation by background-color shift). Spacing scale, typography, component inventory, copywriting, and accessibility rules are unchanged from revision 1.

## Spacing Scale

Declared values from the standard 8-point set `{4, 8, 16, 24, 32, 48, 64}`, tuned for a 400×700 window. **12px is deliberately excluded** from the scale to prevent sub-grid misalignment; the standard set skips it. All spacing in the doc uses one of the six tokens below or one of the chrome exceptions.

| Token | Value | Tailwind class | Usage |
|-------|-------|---------------|-------|
| `xs` | 4px | `space-1`, `gap-1`, `p-1` | Icon-to-label gap, inline tag padding, badge inner padding |
| `sm` | 8px | `space-2`, `gap-2`, `p-2` | Compact element spacing (button inner padding vertical, portal-card inner row gap), **chat-bubble padding, popover inner padding, input inner padding** (compact density per project stance) |
| `md` | 16px | `space-4`, `gap-4`, `p-4` | Default vertical rhythm between message blocks, settings row vertical gap, cards |
| `lg` | 24px | `space-6`, `gap-6`, `p-6` | Section gaps in Settings page, setup-screen form-section gaps |
| `xl` | 32px | `space-8`, `gap-8`, `p-8` | Setup screen outer padding, empty-state vertical breathing room |
| `2xl` | 48px | `space-12`, `gap-12`, `p-12` | Page-level vertical spacing on Settings (between major section groups) |

**Chrome exceptions (justified structural sizes — not scale members):**
- **Top bar height: 36px** — sized to fit `Hexagon` 16px icon + 8px vertical padding + room for the `Agent ⏵` toggle pill at 28px height. Original USERFLOW.md said "~32px"; UI researcher locks 36px because 32px clips lucide icons at 16px with any padding margin.
- **Bottom rail height: 56px** — three tap targets each ≥48px tall for touch ergonomics; 4px top/bottom padding gives a 56px rail. Original USERFLOW.md said "~48px"; UI researcher locks 56px because true touch-target minimum from WCAG 2.5.5 / Material is 48px, and the rail's 8px chrome padding pushes total to 56px.
- **Window minimum width: 320px** — icon-only mode for bottom-rail tabs; below this, labels render but truncate. The USERFLOW.md "~280px soft floor" is treated as a *soft* hint; 320px is the hard `minWidth` enforced by `BrowserWindow`.
- **Window minimum height: 480px** — preserves: 36px top bar + 56px bottom rail + ~150px setup form essentials + 16px outer padding × 2. Below this, the LLM Setup screen scrolls within its outer container.
- **Logs drawer collapsed strip height: 32px** — single-line `Logs ▾`, smaller than top bar by design — drawer is secondary chrome.
- **Status popover width: 280px** — fixed; fits "qwen2.5-7b" model names without truncation, anchored top-right under `⬢`.
- **Slide-in History panel width: 80% of window width, capped at 360px** — so on a stretched-wide window the panel stays a panel, not a half-screen split.

## Typography

Three sizes, two weights — chat density tuned for a 400px-wide window. All sizes given in `rem` for Tailwind compatibility (root = 16px).

**Two weights only — 400 regular + 600 semibold.** No third weight is declared anywhere in the contract. Inline emphasis (including error-state emphasis) is achieved through *color* (`--destructive` red) and *glyph* (`⚠`), not through weight. The 600 semibold provides sufficient contrast against 400 regular for any in-line emphasis need.

| Role | Size | Weight | Line Height | Tailwind | Usage |
|------|------|--------|-------------|----------|-------|
| **Body** | 14px (`0.875rem`) | 400 (regular) | 1.5 | `text-sm leading-normal` | Chat messages, settings field labels, button text, popover content, drawer logs (when not in mono context), error banners, placeholder copy, radio-group labels |
| **Small / Meta** | 12px (`0.75rem`) | 400 (regular) | 1.4 | `text-xs leading-snug` | Timestamps, status-popover detail lines, badges, tooltip text, bottom-rail tab labels, "Coming in milestone-N" placeholder subtitle, radio-group "(default)" annotation |
| **Heading** | 18px (`1.125rem`) | 600 (semibold) | 1.3 | `text-lg font-semibold leading-snug` | Setup screen title ("Connect a language model"), Settings section headers, History panel sections ("Today" / "Yesterday"), Agent page section headers ("Sessions", "Saved goals", "Defaults") |

**Mono variant (logs / connection-test panel only):** 12px, regular, 1.5 — `font-mono text-xs leading-normal`. Used for the verbose connection-test log on the LLM Setup screen, the Logs drawer content, and any READY-line / error-trace surfacing.

**Why three sizes only:** A 400px window cannot afford a four-tier hierarchy without crowding. Headings at 18px are visibly larger than body 14px (1.29× ratio — comfortably above the 1.2 minimum for legible hierarchy); 12px meta is small but acceptable at desktop viewing distance. No display tier — the LLM Setup screen's "Connect a language model" heading is 18px, not 24px+, because 400px width clips longer-line headings.

**Error-state inline emphasis (executor reference):** Error copy such as `⚠ LLM unreachable — connection refused at {host}:{port}` renders as **400 regular body text** in `--destructive` (status-red) color, prefixed with the `⚠` glyph. The color + glyph combination — not weight — carries the emphasis. Do NOT introduce a 700 bold weight or any other third weight to the project.

## Color

**The previous revision's single lavender-violet `--primary` is GONE.** Revision 2 ships a **multi-theme system** where the user picks accent colors in Settings → §14 Appearance per D-23. At any given moment exactly **one** resolved accent is active — the picker selects which one but never displays multiple simultaneously, so the "single accent" rule still holds.

All values declared in OKLCH (shadcn 2026 default for perceptually uniform palettes). Hex equivalents given for executor reference; **OKLCH is the source of truth** in `index.css` / `tailwind.config.ts`.

**Saturation cap (locked):** chroma ≤ `0.13` for accents; chroma ≤ `0.04` for backgrounds and surfaces. Nothing above. This keeps the warm/flat aesthetic intact across all combinations — verified that every accent below sits at or below `0.13`.

### 60 / 30 / 10 ratio (unchanged framing)

| Slot | Token | Role |
|------|-------|------|
| **60% — Dominant** | `--background` | Window background, chat surface, Settings/Agent page background |
| **30% — Secondary** | `--card` / `--popover` / `--muted` | Top bar, bottom rail, popovers, History panel, Logs drawer, message bubbles, settings rows, input fields |
| **10% — Accent** | `--primary` (resolved per active theme) | Reserved for the 6 affordances listed in **Accent Reserved For** below |

The framing is identical to revision 1. Only the *colors* filling each slot have changed.

**Resolution logic (renderer-side, runs at boot and on Settings change):**

```ts
type AppearancePrefs = {
  mode: "auto" | "light" | "dark";
  lightAccent: "blush" | "sunrise" | "ember";
  darkBg: "midnight" | "onyx";
  darkAccent: "sky" | "pewter";
};

function resolveThemeClass(p: AppearancePrefs): string {
  const resolvedMode =
    p.mode === "auto"
      ? matchMedia("(prefers-color-scheme: dark)").matches ? "dark" : "light"
      : p.mode;
  if (resolvedMode === "light") return `theme-${p.lightAccent}`;
  return `theme-${p.darkBg}-${p.darkAccent}`;
}
```

Apply by setting `document.documentElement.className = resolveThemeClass(prefs)`. Listen to `matchMedia("(prefers-color-scheme: dark)")` for changes when `mode === "auto"`.

### Default at first launch

`mode: "auto"` (revision 2 default — was "dark" in revision 1). All accent/BG sub-pickers default to: `lightAccent: "blush"`, `darkBg: "midnight"`, `darkAccent: "sky"`. So a first launcher on a Dark-preferring OS sees **Midnight + Sky**; on a Light-preferring OS, **Blush**. Resolves continuously if the user toggles their OS theme while the app is open (assuming `mode: "auto"`).

### Color Contrast Targets

All foreground-on-background pairs MUST meet WCAG 2.1 AA across **every** theme variant:
- Body text on background: contrast ≥ 4.5:1 (verify all 7 variants)
- Body text on `--muted` (input fields): contrast ≥ 4.5:1
- `--primary-foreground` text on `--primary` (CTAs): contrast ≥ 4.5:1 (Light variants are tightest — Blush/Sunrise at L=0.72 with cream foreground; if axe flags, swap to near-black `--primary-foreground`)
- `--foreground` text on Status-Red / Status-Amber surfaces (banners): contrast ≥ 4.5:1
- Disabled / `--muted-foreground` on `--background`: contrast ≥ 3:1 (relaxed for non-essential text per WCAG 1.4.3 "incidental")

Executor: run `axe-core` against every theme variant at the end of plan `01-02`. Variants to test: `theme-blush`, `theme-sunrise`, `theme-ember`, `theme-midnight-sky`, `theme-midnight-pewter`, `theme-onyx-sky`, `theme-onyx-pewter`. Any failure → adjust `--primary-foreground` or status-color tier; do NOT raise accent chroma.

## Aesthetic Guardrails (locked per D-23)

These are **hard constraints** for revision 2, derived from the Material You / Claude-app reference. Auditor verifies all six.

| Guardrail | Rule | Enforcement |
|-----------|------|-------------|
| **Shadows** | **No `shadow-md` / `lg` / `xl` / `2xl` anywhere.** Only `shadow-sm` permitted, and only on popovers / sheets / drawers that need depth from background. Cards use background-color shift (`--card` lighter or warmer than `--background`) for separation, not shadow. | Grep the codebase: `\bshadow-(md\|lg\|xl\|2xl)\b` should return zero hits. |
| **Radius** | `rounded-2xl` (16px) for cards / sheets / drawers / portal cards. `rounded-xl` (12px) for buttons / inputs / radio-group containers. `rounded-full` for pills / toggles / badges / status dots. **No `rounded-sm` / `rounded` / `rounded-md` anywhere.** | Larger than shadcn's defaults — overrides are part of the `@theme` block. |
| **Surface separation** | Lift via background-color shift (`--card` ~5% lighter than `--background` in dark; ~3% darker/warmer in light). Never via shadow. | Already encoded in the OKLCH values above (Dark BG L=0.20 vs Card L=0.25; Light BG L=0.98 vs Card L=0.95). |
| **Transitions** | 200ms ease for hover / focus / state change. **No spring / bounce.** Theme-switch fade: 250ms ease (specifically for theme-class transitions on root). Slide-in History panel: 250ms ease-out. | Use Tailwind's `transition-colors duration-200 ease-out` baseline. |
| **Saturation cap** | Accents: chroma ≤ `0.13`. Backgrounds / muted / cards: chroma ≤ `0.04`. **Nothing above.** | Verified for all OKLCH values in the Color section. |
| **Overlays** | `bg-background/60` backdrop. Optional `backdrop-blur-sm` (light blur). **No `backdrop-blur-md` / `lg` / `xl`.** | History sheet backdrop, AlertDialog backdrop. |
| **Borders** | `border-border/60` opacity is the typical case; `border-border` (full opacity) only on inputs at rest. Hover/focus states use `--primary` ring instead of solid border. | Settings rows, popover dividers, separator lines: `border-border/60`. Input rest state: `border-border`. |

**Anti-patterns explicitly forbidden:**
- Any `shadow-md`+ class
- Any `rounded-sm` / `rounded-md` / `rounded` (the unsuffixed default 4px)
- Spring/bounce easings (`ease-elastic`, `ease-bounce`, custom cubic-beziers with overshoot)
- `backdrop-blur-md` or heavier
- Mixing two accents on the same surface (defeats the "one accent at a time" rule)

## Copywriting Contract

**Voice and tone:** *Warm companion who happens to be technical.* The app is a Live2D-avatar conversation partner; copy should feel like the avatar's natural speech (in chat states) and like considered product copy (in chrome / errors). Avoid:
- Corporate stiffness ("Please ensure your connection is configured correctly")
- Faux-quirky ("Whoops! Something broke 😅")
- Anthropomorphic over-reach in error states (the avatar speaks; the *system* doesn't have feelings)

The rule: **chrome is calm, errors are direct, empty states are brief and forward-looking.**

## Interaction Patterns

### State-machine routing (USERFLOW.md §4)

The app has three routes: `setup`, `main`, and (future) `pet`. Phase 1 implements `setup` and `main`. Routing is gated by `safeStorage.hasCompletedSetup`:

- App boots → main process reads `hasCompletedSetup`
- `false` → renderer mounts `<SetupScreen>` only (no chrome)
- `true` → renderer mounts `<AppShell>` with chrome + active route from bottom rail

The state machine has **no reverse path** in skeleton. Re-test happens via the status popover button; full re-config (`Change provider →`) is grayed and disabled.

`<ThemeManager>` mounts at the root **before** route resolution so the setup screen, the main shell, and any AlertDialog all share a consistent theme from the first paint. First-launch Appearance prefs default to `mode: "auto"`, so the setup screen colors itself based on the OS preference.

### Setup screen blocking behavior (D-09, PLUMB-04)

- `Continue →` button is **disabled** until a successful test completes (state: `idle | testing | success | error`)
- Successful test → `Continue →` enables; `Test connection again` link appears below
- Failed test → log panel shows verbose error; `Continue →` stays disabled; user must Test connection again
- Closing the window during setup → `app.quit()` (no partial config persisted; safeStorage write only on `Continue →` click)

### Bottom-rail tab switching (USERFLOW.md F)

- Tap a tab → switch route. **Preserves Chat scroll position** (use a ref-tracked scroll position keyed by route).
- Settings always opens scrolled to top.
- The `☰` hamburger in top bar is **hidden** (display: none) when not on Chat route. Reserve the 36px slot to prevent layout shift — only the icon is hidden.
- Active-tab affordance: filled icon variant + accent color on label; inactive: outline icon variant + `--muted-foreground` label.

### Status icon popover (USERFLOW.md §3.4, G)

- Click `⬢` → popover anchors top-right under the icon (offset 4px from icon bottom edge, 8px from window right edge)
- Popover contains: 3-row status list (LLM / VTS / Sidecar) + `[ Re-test connection ]` button
- `[ Re-test connection ]` runs the same 1-token test as setup
  - During: button text → `Testing...`, button disabled, spinner left of label
  - Success: row updates to green, popover stays open, "last reply 423ms" updates
  - Failure: inline error appears below row in red mono text; row turns red; banner appears in main app surface (per L.1)
- Close popover: click outside, Esc, or click `⬢` again.

### Logs drawer toggle (USERFLOW.md K)

- Off (Settings §15 toggle false): drawer is `display: none`. **No chevron strip rendered.** Chat surface butts directly against bottom rail.
- On: chevron strip renders above bottom rail at 32px height with `Logs ▾` label
  - Click strip → expand to last-saved height (default 200px) with smooth 200ms transition
  - Drag handle (top edge of expanded drawer) → resize between 80px and `windowHeight - 200px`
  - Click `▴` chevron in expanded header → collapse to 32px strip
  - Drawer state (`enabled`, `collapsed`, `height`) persists via electron-store

### History slide-in (USERFLOW.md D)

- Tap `☰` → `Sheet` slides from left with 250ms ease-out timing
- Width: `min(80vw, 360px)`
- Backdrop: `bg-background/60` + `backdrop-blur-sm` (chat content **not** removed from DOM, just dimmed)
- Close: tap backdrop, Esc, or `✕` in panel header
- Skeleton content: placeholder centered vertically per "Skeleton state ⏳" in USERFLOW.md D.3

## Accessibility

- All interactive elements have visible focus rings (2px accent outline, 2px offset)
- All icons paired with text label OR `aria-label`
- **Hamburger `☰` button:** `aria-label="Open conversation history"` (explicit, since the icon-only button has no visible text label). Disabled state on Agent / Settings views uses `aria-hidden="true"` since the icon is not rendered.
- **Status icon `⬢`:** `aria-label` reflects current state — `"Status: all connected"` / `"Status: VTube Studio degraded"` / `"Status: LLM error"` (worst-of-three component included so the label is meaningful before the popover opens).
- **Agent toggle `Agent ⏵`:** `aria-pressed="true|false"` reflecting ON / OFF state. In skeleton, `aria-disabled="true"` with the disabled tooltip copy as the announcement.
- Bottom-rail tabs have `role="tab"` and `aria-selected`; the rail itself is `role="tablist"`.
- AlertDialog uses `role="alertdialog"` with focus trap (shadcn handles)
- Chat messages have `role="log"` with `aria-live="polite"` so screen readers announce new content without interrupting
- Logs drawer content is `role="log"` `aria-live="off"` (it's chrome-secondary, not user-facing alerts)
- **Settings §14 Appearance radio groups:** each group is a `<fieldset>` with a `<legend>` matching the group label (`Mode`, `Light accent`, `Dark background`, `Dark accent`). Disabled groups carry `aria-disabled="true"` on the fieldset; the disabled-tooltip text is announced via `aria-describedby` pointing to the helper line. Each radio option's swatch has `aria-hidden="true"` (it's purely decorative; the label carries the meaning). Default options carry `(default)` in the visible label, which screen readers read aloud — no extra ARIA needed.
- **`<AppearancePreview>`:** `role="img"` with `aria-label="Live preview of the selected theme"`. The internal swatches and faux button are `aria-hidden="true"` so screen readers don't read decorative content; the single `aria-label` summarizes intent.
- Color is never the sole indicator of state — the `⬢` icon has both color AND a small overlay glyph (`✓` / `!` / `✕` based on state) so colorblind users can still distinguish. Theme accent swatches are also paired with text labels so a colorblind user can pick "Blush" by name without distinguishing it from "Sunrise" by hue.
- Minimum tap target 44×44px for any rail tab; 32×32px for chrome icons (status / hamburger) — within the WAI ARIA touch-target relaxation for desktop. Radio group items target 44×44px (the row including the swatch + label).

## Phase Boundary Notes for Downstream Phases

These tokens are the **seed** for every later phase. Phase 2-5 should:

- **Phase 2** (real LLM streaming): Reuse all tokens. Add token shimmer / sentence-streaming animation — recommendation: a 12px `--muted-foreground` "thinking" dot pulse before each sentence, replaced by sentence text on arrival. The shimmer skeleton component is `skeleton` from shadcn. **No new color tokens needed.** Honor the saturation cap and shadow guardrails — the pulse animation should be opacity-only, not shadow-glow.
- **Phase 3** (TTS audio): No visual changes. Optionally add a tiny audio-waveform glyph next to currently-speaking sentence (use `--primary` resolved per active theme). **No new tokens.**
- **Phase 4** (compositor + VTS bridge): The VTS window is external; nothing to design *in our window* for the avatar itself. Cursor tracking sends events sidecar-direct. **No new visual tokens.** Only addition: F9 prop-toggle could surface a transient toast confirmation ("Prop toggled") — uses existing `sonner`.
- **Phase 5** (verification + codegen): No visual changes. Verification is a markdown doc, not UI.

If any later phase wants to introduce a new color, font size, or spacing value, the convention is to **extend** the scale (e.g., add `2xs: 2px` if compactness demands), not redefine an existing token. Token redefinition requires re-approval through `/gsd:ui-phase`. Adding a new theme variant (e.g. an additional accent color) is a *contract change* and requires a fresh revision pass through `/gsd:ui-phase`.

## Visual Anchors (for the executor and auditor)

These are the must-look-right surfaces in Phase 1, ranked by user-visible importance:

1. **LLM Setup screen** — first impression, the only blocking modal in the app's lifetime. Renders in whatever theme `<ThemeManager>` resolves at first paint (typically `theme-blush` for light-OS users, `theme-midnight-sky` for dark-OS users).
2. **`⬢` status icon + popover** — the user's at-a-glance health check, looked at hundreds of times. Status colors must remain perceptually distinct from the active accent in every theme variant.
3. **Bottom rail** — always visible, must communicate active tab clearly. Active tab uses the resolved `--primary`.
4. **Echo chat surface** — proves the round-trip works; bubble styling sets the precedent for Phase 2-4. Assistant bubble = `--card`; user bubble = `--muted` (a step elevated from `--background`); both `rounded-2xl`.
5. **Settings §14 Appearance** — new in revision 2. Four radio groups + live preview must feel instant on selection; the 250ms theme-fade is the perceptual evidence the change applied.
6. **Logs drawer (when enabled)** — for power users debugging their setup; mono font and dense-but-legible.
7. **Empty states** (VTS-not-connected, fresh-thread) — first-launch perceived quality.

Designs that pass all seven visual-anchor checks pass the visual contract.
